The Observability Crisis trong Multi-Agent Systems

Tại sao 76% enterprise multi-agent incidents không phát hiện được trong 10 phút đầu — và cách giải quyết.

🎯 Sau bài này, bạn sẽ:

  • Hiểu tại sao APM truyền thống (Datadog, New Relic) không đủ cho multi-agent AI systems
  • Biết 4 tín hiệu quan sát vàng (Golden Signals) áp dụng cho agents
  • Phân biệt Logging, Metrics, và Tracing — và khi nào dùng cái nào
  • Nhận ra "Observability Gap" của LoanBot v2.0 và lộ trình vá lỗi

Vấn đề: Black Box trong Production

LoanBot v3.0 chạy 8 agent song song, xử lý 50,000 hồ sơ/tháng. Nhưng khi một hồ sơ mất 47 giây thay vì 5 giây bình thường — bạn biết ngay lỗi ở đâu không? Câu trả lời thường là: không.

⚠️ Thực tế từ Production (McKinsey 2025)

76% incidents trong multi-agent systems không được phát hiện trong 10 phút đầu. Lý do: team chỉ biết output sai nhưng không biết agent nào gây ra, lúc nào bắt đầu, tại sao. Mean Time To Detect (MTTD) trung bình: 47 phút. Chi phí mỗi phút downtime: ~15 triệu VNĐ.

Tại sao APM truyền thống không đủ?

Datadog, New Relic, Prometheus — tuyệt vời cho microservices. Nhưng agents có những đặc điểm riêng mà APM không xử lý được:

APM truyền thống bỏ sót 8 loại metric quan trọng cho multi-agent systems

Ba Trụ Cột Của Observability

Observability hiện đại dựa trên 3 loại dữ liệu bổ trợ nhau — không có cái nào thay thế được cái kia:

📌 Khái niệm: Ba Trụ Cột (Three Pillars of Observability)
1. Logs (Nhật ký) — sự kiện rời rạc, có timestamp. Ví dụ: [2024-01-15 09:23:41] CreditAgent: credit_score=720. Dùng cho: debugging sau sự cố, audit trail.
2. Metrics (Chỉ số) — số liệu tổng hợp theo thời gian. Ví dụ: agent_latency_p99=287ms. Dùng cho: alerting, capacity planning, SLO tracking.
3. Traces (Dấu vết) — chuỗi sự kiện liên kết trong một request duy nhất, qua nhiều services. Ví dụ: một hồ sơ FC-2024-001 đi qua Supervisor → CreditAgent → RiskAgent, mỗi bước là một span trong cùng một trace. Dùng cho: xác định bottleneck, hiểu request flow.

Four Golden Signals — Áp dụng vào Agents

Google SRE Book giới thiệu 4 tín hiệu vàng cho bất kỳ production system nào. Đây là cách áp dụng cho agent systems:

⏱️

Latency

Thời gian agent xử lý một request. P99 < SLA của từng agent trong LoanBot v3.0.

📊

Traffic

Số requests/giây qua hệ thống. Peak: 8–9h sáng, ~3 hồ sơ/phút cho LoanBot.

❌

Errors

Tỷ lệ agent calls thất bại. LoanBot target: < 0.1% error rate toàn pipeline.

🌡️

Saturation

Mức độ "căng" của hệ thống. Token/phút, concurrent agents, queue depth.

🧠 Kiểm tra nhanh

LoanBot v3.0 đang chạy. Bạn nhận được báo cáo "hồ sơ FC-2024-555 mất 42 giây". Bạn cần loại observability data nào để tìm root cause nhanh nhất — Logs, Metrics, hay Traces?

→ Traces trước tiên. Trace cho bạn thấy: Supervisor dispatch lúc 09:23:41, CreditAgent bắt đầu 09:23:42, kết thúc 09:24:28 (46 giây — CÁI NÀY là bottleneck), IncomeAgent chỉ 0.3 giây. Logs sẽ cho bạn biết cái gì xảy ra, nhưng Trace cho biết ở đâu và mất bao lâu. Metrics sẽ confirm sau ("CreditAgent P99 spike lúc 09:23").

OpenTelemetry cho Agent Systems

Chuẩn vendor-neutral số 1 thế giới cho distributed tracing — và cách instrument mỗi agent call thành một span.

🎯 Sau bài này, bạn sẽ:

  • Hiểu OpenTelemetry (OTel) là gì và tại sao nó trở thành chuẩn mực năm 2025
  • Biết cấu trúc Trace → Span → Attribute và cách chúng liên kết qua agents
  • Implement trace_id propagation trong LoanBot v3.0
  • Thêm custom attributes: token count, cost, model version vào mỗi span

OpenTelemetry — Tiêu Chuẩn Quốc Tế

OpenTelemetry (OTel) là project CNCF (Cloud Native Computing Foundation) mã nguồn mở, được Google, Microsoft, AWS, Anthropic, và 700+ công ty support. Thay vì mỗi vendor có SDK riêng, OTel cung cấp một API chung — bạn instrument code một lần, export đến bất kỳ backend nào (Jaeger, Grafana Tempo, Datadog, Honeycomb).

Cấu trúc OTel cho Agent Systems
Trace = Trace ID + [Span₁ → Span₂ → … → Spanₙ]
Một hồ sơ FC-2024-001 = 1 Trace · 8 agent calls = 8 Spans · Mỗi Span có parent-child relationship

Trace Context Propagation — Xuyên Suốt 8 Agents

Khi Supervisor tạo một trace_id, tất cả 8 agents phải kế thừa trace đó — không tạo trace mới. Đây là cách LoanBot v3.0 liên kết tất cả spans thành một waterfall diagram:

Trace Waterfall của LoanBot v3.0 — mỗi màu là một agent, phase 1 chạy song song, checkpoint markers màu vàng
Python — OTel Instrumentation 

from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider

# Khởi tạo OTel tracer — chạy một lần khi startup
provider = TracerProvider()
tracer = trace.get_tracer("loanbot.v3")

class ObservableCreditAgent(CreditAgent):
    async def process(self, task: A2ATask) -> A2ATask:
        # Tạo span kế thừa từ parent trace của Supervisor
        with tracer.start_as_current_span("credit_agent.process") as span:
            span.set_attribute("agent.name", "CreditAgent")
            span.set_attribute("loan.id", task.payload["loan_id"])
            span.set_attribute("llm.model", MODEL)

            result = await super().process(task)

            # Ghi kết quả vào span attributes
            span.set_attribute("credit.score", result.artifacts.get("credit_score", 0))
            span.set_attribute("llm.cost.usd", 0.002)   # từ API response
            return result
🔑 Key Insight: trace_id là "sợi chỉ đỏ"

Một trace_id duy nhất (ví dụ: 3f2a8c1b…) liên kết tất cả 8 agent spans lại. Khi bạn nhập trace_id vào Grafana Tempo hay Jaeger, bạn thấy toàn bộ waterfall — từ lúc Supervisor nhận hồ sơ đến khi AuditAgent lưu log. Không có trace_id, bạn chỉ thấy 8 điểm rời rạc — không biết chúng thuộc hồ sơ nào.

🧠 Kiểm tra nhanh

LoanBot v3.0 xử lý 50,000 hồ sơ/tháng. Mỗi hồ sơ có 8 spans. Hỏi: Tháng này có bao nhiêu spans tổng cộng? Và nếu error rate là 0.1%, có bao nhiêu spans có status ERROR?

Tổng spans: 50,000 × 8 = 400,000 spans/tháng. Spans lỗi: 400,000 × 0.1% = 400 error spans. Đây là lý do cần sampling strategy — lưu 100% error spans nhưng chỉ sample 1% success spans → tiết kiệm storage mà vẫn capture đủ incidents.

SLI / SLO / Error Budget — Cam Kết Bằng Con Số

Cách định nghĩa "LoanBot hoạt động tốt" bằng những con số cụ thể, đo lường được, và có accountability.

🎯 Sau bài này, bạn sẽ:

  • Phân biệt SLI (đo lường), SLO (mục tiêu), SLA (cam kết pháp lý)
  • Định nghĩa SLO cụ thể cho từng agent trong LoanBot v3.0
  • Tính Error Budget và biết khi nào nên "tiêu" error budget để ship nhanh hơn
  • Thiết kế alert rules dựa trên SLO (không phải dựa trên arbitrary threshold)
💡 Analogy: SLO như cam kết chất lượng dịch vụ ở nhà hàng

Nhà hàng cam kết: "Món ăn sẽ được phục vụ trong 15 phút" (SLO). Chỉ số đo: thời gian từ gọi đến khi có món (SLI). Hợp đồng với food court: nếu vi phạm > 10 lần/tháng, hoàn tiền (SLA). Error budget = 10 lần/tháng còn lại. Nếu đã vi phạm 8 lần, bếp không được thử menu mới (freeze deployments) cho đến tháng sau.

Error Budget Formula
Error Budget = 1 − SLO target
SLO 99.9% → Error Budget = 0.1% = 43.8 phút downtime/tháng được phép

Alert Rules — SLO-Based Alerting

Thay vì alert khi "CPU > 80%", alert khi "SLO đang bị consume quá nhanh". Đây là 3 alerts quan trọng nhất cho LoanBot v3.0:

🔴 CRITICAL: Error Budget Burn Rate > 14x
Nếu error rate burn rate gấp 14 lần normal → sẽ hết toàn bộ error budget trong 1 giờ. Action: page on-call engineer ngay, freeze deployments, kích hoạt circuit breaker cho CIC API.
🟡 WARNING: CreditAgent P99 > 250ms (5 phút liên tục)
CreditAgent đang chậm hơn bình thường — có thể CIC API bị congestion. Action: check circuit breaker status, xem CIC API status page, consider switch sang secondary CIC provider.
🔵 INFO: Cost per hồ sơ tăng 20% so với tuần trước
Có thể prompt dài hơn, model thay đổi, hoặc input data phức tạp hơn. Action: review prompt versions, check token counts theo từng agent, không cần action ngay.
🧠 Kiểm tra nhanh

LoanBot có SLO error rate 99.9%/tháng. Trong 3 ngày đầu tháng, đã có 120 errors trong 40,000 requests (0.3%). Hỏi: Error budget còn lại bao nhiêu phút? Có nên tiếp tục deploy tính năng mới không?

Error budget tổng: 43.8 phút/tháng. Đã tiêu: 0.3% - 0.1% = 0.2% excess × 3 ngày → ước tính ~17.5 phút đã burn. Còn lại: ~26.3 phút. Kết luận: Không nên deploy tính năng mới — đã burn 40% error budget chỉ trong 10% của tháng. Ưu tiên fix root cause trước.

AgentMetricsCollector — Đo Lường Tất Cả

Một lớp wrapper nhẹ, bọc quanh mỗi agent call để collect đủ 4 Golden Signals:

Python — AgentMetricsCollector 

import time
from collections import defaultdict

class AgentMetricsCollector:
    # Thu thập metrics theo thời gian thực — export lên Prometheus
    def __init__(self):
        self.latencies = defaultdict(list)   # agent_name → [ms, ms, ...]
        self.errors    = defaultdict(int)    # agent_name → error_count
        self.tokens    = defaultdict(int)    # agent_name → total_tokens
        self.costs     = defaultdict(float)  # agent_name → total_usd
        self.calls     = defaultdict(int)    # agent_name → total_calls

    def record(self, agent_name: str, latency_ms: float,
                  tokens: int, cost: float, error: bool = False):
        self.latencies[agent_name].append(latency_ms)
        self.tokens[agent_name]   += tokens
        self.costs[agent_name]    += cost
        self.calls[agent_name]    += 1
        if error:
            self.errors[agent_name] += 1

    def p99(self, agent_name: str) -> float:
        lats = sorted(self.latencies[agent_name])
        if not lats: return 0
        idx = int(0.99 * len(lats))
        return lats[min(idx, len(lats) - 1)]

    def error_rate(self, agent_name: str) -> float:
        total = self.calls[agent_name]
        if not total: return 0.0
        return self.errors[agent_name] / total * 100

    def report(self):
        # In dashboard summary ra console — production sẽ export Prometheus metrics
        print("\n📊 AgentOps Dashboard — LoanBot v3.0")
        print(f"{'Agent':<20} {'Calls':>5} {'P99(ms)':>8} {'Errors':>7} {'Tokens':>8} {'Cost($)':>9}")
        print("-" * 65)
        for name in self.calls:
            print(f"{name:<20} {self.calls[name]:>5} {self.p99(name):>8.0f}"
                  f" {self.errors[name]:>7} {self.tokens[name]:>8} {self.costs[name]:>9.4f}")

YAML — Prometheus Alert Rules (loanbot_slo.yml) 

groups:
  - name: loanbot_slo_alerts
    rules:
      # Alert 1: CreditAgent P99 vượt SLO
      - alert: CreditAgentLatencySLOBreach
        expr: histogram_quantile(0.99, agent_latency_ms{agent="CreditAgent"}) > 300
        for: 5m
        labels: { severity: warning, team: loanbot }
        annotations:
          summary: "CreditAgent P99 > 300ms SLO threshold"
          runbook: "https://wiki.fintechcorp/runbooks/credit-agent-slow"

      # Alert 2: Error budget burn rate quá nhanh (CRITICAL)
      - alert: ErrorBudgetBurnRateCritical
        expr: rate(agent_errors_total[1h]) / rate(agent_requests_total[1h]) > 0.014
        for: 2m
        labels: { severity: critical, team: loanbot }
        annotations:
          summary: "Error budget burning 14x faster than allowed"
          action: "FREEZE deployments, page on-call"

      # Alert 3: Cost per hồ sơ spike
      - alert: CostPerApplicationSpike
        expr: rate(agent_cost_usd_total[1h]) / rate(loan_applications_total[1h]) > 0.008
        for: 10m
        labels: { severity: info, team: loanbot }
        annotations:
          summary: "Cost per application approaching $0.01 SLO"

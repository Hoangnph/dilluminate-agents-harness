Bài 1: Testing Pyramid for AI Agents — Tại Sao AI Testing Khác?

🎯 Sau bài này, bạn sẽ:

  • Hiểu tại sao AI agents cần testing pyramid khác với software thông thường
  • Biết 4 tầng testing: Unit → Integration → E2E → Chaos
  • Phân biệt deterministic tests vs probabilistic tests
  • Thiết kế test strategy cho LoanBot: mỗi PR phải pass trước khi merge
💡 Analogy: Test như QC dây chuyền sản xuất

Nhà máy sản xuất xe hơi có 3 trạm kiểm tra: (1) Kiểm tra từng linh kiện riêng lẻ (unit test); (2) Kiểm tra engine sau khi lắp ráp (integration test); (3) Test drive toàn xe trên đường (E2E test). AI agent testing tương tự — nhưng thêm tầng thứ 4: Chaos test — cố tình phá hỏng đường để xem xe có kiểm soát được không.

Hình 1: LoanBot Testing Pyramid — 4 tầng từ dưới lên: Unit (500+ tests, on commit) → Integration (150+, per PR) → E2E (30+, per release) → Chaos (5+, monthly). AI agents cần Chaos tier mà software thông thường không có.
📌 Khái niệm: Deterministic vs Probabilistic Tests

Deterministic test: input X → output Y, luôn luôn. Ví dụ: "credit_score=400 và blacklisted=True → decision='REJECT'". Có thể verify bằng assert đơn giản.

Probabilistic test: LLM output khác nhau mỗi lần. Thay vì verify exact string, verify properties: "explanation phải chứa credit_score", "decision phải là một trong APPROVE/REJECT/REVIEW", "confidence phải 0.0–1.0". Tests này cần LLM-as-Judge hoặc regex patterns thay vì string equality.

Công thức cốt lõi
Test AI = Assert Properties, Not Strings
LLM output không bao giờ identical qua 2 lần chạy. Verify structure, format, semantic correctness — không verify exact wording.
Python (LoanBot Test Framework — Base Classes) 
import pytest
from dataclasses import dataclass
from typing import List, Callable
import json

# Deterministic test — exact assertion
def test_blacklist_always_rejects():
    # Blacklisted → REJECT is deterministic (rule-based, no LLM needed)
    result = loanbot.evaluate(credit_score=720, blacklisted=True)
    assert result["decision"] == "REJECT"
    assert result["confidence"] >= 0.95

# Probabilistic test — property assertion
def test_explanation_has_required_fields():
    result = loanbot.evaluate(credit_score=580, dti=0.58)
    explanation = result["explanation"]
    # Don't assert exact text — assert semantic properties
    assert result["decision"] in {"APPROVE", "REJECT", "REVIEW"}
    assert 0.0 <= result["confidence"] <= 1.0
    assert len(explanation) >= 50     # not empty
    assert "credit" in explanation.lower()  # mentions credit
    assert "approve" not in result["explanation"].lower() # no contradiction

# Structural test — output schema validation
REQUIRED_FIELDS = {"decision", "confidence", "explanation", "regulation_refs"}
def test_output_schema():
    result = loanbot.evaluate(credit_score=645, dti=0.38)
    assert REQUIRED_FIELDS.issubset(result.keys()), f"Missing: {REQUIRED_FIELDS - result.keys()}"

Bài 2: Golden Dataset Testing — Ground Truth & LLM-as-Judge

🎯 Sau bài này, bạn sẽ:

  • Xây dựng golden dataset: tập cases với expected outcomes đã được expert verify
  • Implement LLM-as-Judge để evaluate LoanBot explanations
  • Đo accuracy, precision, recall trên golden dataset
  • Thiết kế continuous evaluation pipeline: chạy tự động sau mỗi model/prompt change
🏢 Tình huống: LoanBot Golden Dataset v1

Tháng 1/2025, FinTech Corp nhờ 5 cán bộ tín dụng senior review 200 hồ sơ đặc biệt (50 easy approve, 50 easy reject, 50 borderline approve, 50 borderline reject). Đây là golden dataset — ground truth để evaluate LoanBot. Mỗi khi team thay đổi prompt, model, hoặc logic: chạy lại 200 cases này và so sánh kết quả với expert judgment. Accuracy drop > 2% → rollback.

Python (Golden Dataset Evaluation) 
from dataclasses import dataclass
from typing import List, Literal
import json

@dataclass
class GoldenCase:
    case_id:        str
    credit_score:   int
    dti_ratio:      float
    employment_months: int
    loan_amount:    int
    expert_decision: str   # APPROVE / REJECT / REVIEW
    expert_notes:   str   # why the expert made this decision
    difficulty:     str   # easy / borderline / edge_case

# Sample golden dataset (real dataset = 200 cases)
GOLDEN_DATASET = [
    GoldenCase("GD-001", 720, 0.32, 48, 500_000_000, "APPROVE", "Strong profile", "easy"),
    GoldenCase("GD-002", 400, 0.75, 2,  200_000_000, "REJECT",  "Multiple failures", "easy"),
    GoldenCase("GD-003", 620, 0.44, 7,  300_000_000, "APPROVE", "Just above threshold", "borderline"),
    GoldenCase("GD-004", 580, 0.46, 12, 400_000_000, "REJECT",  "DTI marginally over", "borderline"),
    GoldenCase("GD-005", 650, 0.43, 5,  250_000_000, "REVIEW",  "Employment borderline", "borderline"),
]

def evaluate_golden_dataset(loanbot_fn, dataset: List[GoldenCase]) -> dict:
    """Run LoanBot on golden dataset and compare to expert decisions."""
    total = len(dataset)
    correct = wrong = 0
    errors = []
    by_difficulty = {"easy": [], "borderline": [], "edge_case": []}

    for case in dataset:
        result = loanbot_fn(
            credit_score=case.credit_score, dti=case.dti_ratio,
            employment_months=case.employment_months, loan_amount=case.loan_amount
        )
        predicted = result.get("decision")
        is_correct = predicted == case.expert_decision
        if is_correct: correct += 1
        else:
            wrong += 1
            errors.append({"case": case.case_id, "expected": case.expert_decision,
                            "got": predicted, "difficulty": case.difficulty})
        by_difficulty.get(case.difficulty, []).append(is_correct)

    accuracy = correct / total
    return {
        "total":    total, "correct": correct, "wrong": wrong,
        "accuracy": accuracy,
        "pass":     accuracy >= 0.95,    # 95% minimum to ship
        "errors":   errors,
        "by_difficulty": {
            k: sum(v) / len(v) if v else 0
            for k, v in by_difficulty.items()
        }
    }
🧠 Kiểm tra nhanh

Accuracy 95% trên golden dataset nghe có vẻ tốt. Nhưng nếu golden dataset chỉ có 5% borderline cases và LoanBot sai 100% borderline nhưng đúng 100% easy → accuracy vẫn 95%. Vấn đề gì và cách fix?

Gợi ý: Đây là vấn đề imbalanced evaluation — overall accuracy che giấu poor performance trên important subset. Fix: (1) Stratified accuracy — report accuracy riêng cho từng difficulty level; (2) Đặt minimum threshold riêng: easy ≥ 99%, borderline ≥ 90%, edge_case ≥ 80%; (3) Weight borderline cases cao hơn trong overall metric (chúng represent business risk nhiều hơn); (4) F1-score thay vì accuracy để handle class imbalance. Golden dataset GD-001 đến GD-005 ở trên đã có 3 borderline cases trong 5 — đó là tỷ lệ tốt.

Bài 3: Property-Based Testing — Invariants Không Bao Giờ Được Vi Phạm

🎯 Sau bài này, bạn sẽ:

  • Định nghĩa 6 invariants cốt lõi của LoanBot không được vi phạm
  • Implement property tests với Hypothesis (Python library)
  • Detect hallucinations: LLM tự bịa số không có trong input
  • Test monotonicity: credit_score cao hơn → không thể có decision tệ hơn
⚠️ Property Vi Phạm Nguy Hiểm Nhất: Score Inversion

Scenario thực tế: LoanBot với prompt mới approve FC-A (score 580, DTI 58%) nhưng reject FC-B (score 680, DTI 35%). Đây là monotonicity violation — người có profile tốt hơn bị từ chối, người tệ hơn được duyệt. Về mặt business và pháp lý, đây là disaster: EU AI Act §10 yêu cầu consistent, non-discriminatory decisions; nếu phát hiện sau khi deploy → phải cancel tất cả decisions trong period đó và re-process thủ công.

Python (Property-Based Tests — Core Invariants) 
from hypothesis import given, strategies as st
import re

# INVARIANT 1: Output schema luôn đúng
@given(
    credit_score=st.integers(300, 850),
    dti=st.floats(0.0, 1.0),
    employment=st.integers(0, 120)
)
def test_output_schema_always_valid(credit_score, dti, employment):
    result = loanbot.evaluate(credit_score=credit_score, dti=dti, employment_months=employment)
    assert result["decision"] in {"APPROVE", "REJECT", "REVIEW"}
    assert 0.0 <= result["confidence"] <= 1.0
    assert isinstance(result["explanation"], str)
    assert len(result["regulation_refs"]) > 0

# INVARIANT 2: Blacklisted luôn REJECT
@given(credit_score=st.integers(300, 850))
def test_blacklist_always_rejects(credit_score):
    # Không quan trọng score bao nhiêu — blacklisted → REJECT
    result = loanbot.evaluate(credit_score=credit_score, blacklisted=True)
    assert result["decision"] == "REJECT"

# INVARIANT 3: Monotonicity — better profile ≥ better or equal decision
DECISION_RANK = {"APPROVE": 2, "REVIEW": 1, "REJECT": 0}
def test_monotonicity():
    # Higher score, lower DTI should never get worse decision
    base    = loanbot.evaluate(credit_score=600, dti=0.40)
    better  = loanbot.evaluate(credit_score=700, dti=0.30)
    assert DECISION_RANK[better["decision"]] >= DECISION_RANK[base["decision"]], \
        f"Monotonicity violation: {base['decision']} → {better['decision']}"

# INVARIANT 4: Không hallucinate numbers
def test_no_hallucinated_numbers():
    # LoanBot không được tự bịa ra loan amounts không có trong input
    result = loanbot.evaluate(credit_score=620, loan_amount=300_000_000)
    explanation = result["explanation"]
    # Extract any numbers in explanation
    numbers = [int(n.replace(",", "")) for n in re.findall(r'\d[\d,]+', explanation)]
    # 300,000,000 should appear but not invented numbers like 400,000,000
    for num in numbers:
        if num > 1_000_000:   # significant amounts
            assert abs(num - 300_000_000) < 1_000_000, f"Hallucinated amount: {num}"

Bài 4: Regression & Contract Testing

🎯 Sau bài này, bạn sẽ:

  • Implement regression tests: phát hiện khi thay đổi mới phá vỡ existing behavior
  • Contract testing for tools: đảm bảo CIC API interface không thay đổi unexpectedly
  • Snapshot testing: lock in known-good outputs và alert khi drift
  • Build CI/CD gate: PR không được merge nếu regression tests fail
Hình 2: LoanBot CI/CD Quality Gate Pipeline — 5 test stages trước khi deploy. Mỗi stage có explicit fail criteria. Regression drift >2% và bias metric >1.20 tự động block merge.
📌 Khái niệm: Contract Testing cho Tools

LoanBot dùng query_cic() để lấy credit score. Nếu CIC API thay đổi response format (thêm field, đổi key name) → LoanBot parse sai → quyết định sai. Contract test verify interface giữa LoanBot và external tools: (1) Request format đúng; (2) Response schema được validate; (3) Error cases được handle. Chạy contract tests riêng với mock CIC server → phát hiện breaking changes trước khi integrate.

🧠 Kiểm tra nhanh

Team thay đổi system prompt để LoanBot giải thích rõ hơn. Regression test baseline chạy trước khi change. Sau khi change, 3/200 golden cases đổi từ APPROVE → REVIEW. Quyết định của bạn là gì?

Gợi ý: 3/200 = 1.5% change — dưới threshold 2%. Nhưng cần investigate: (1) 3 cases này là borderline hay easy? Nếu easy cases thay đổi → red flag (stable cases không nên thay đổi do prompt explanation change); (2) Direction có hợp lý không? APPROVE → REVIEW là conservative (không reject loans that should be approved) — có thể acceptable; (3) Query những loan officers về 3 cases này — nếu expert đồng ý REVIEW là hợp lý hơn APPROVE → có thể accept change. (4) Nếu có bất kỳ APPROVE → REJECT changes → hard block, cần investigation sâu hơn. Decision: investigate trước khi merge, không auto-approve dù dưới threshold.

Bài 5: Chaos Engineering for AI Agents

🎯 Sau bài này, bạn sẽ:

  • Thiết kế chaos experiments cho LoanBot: cố tình gây lỗi để test resilience
  • Fault injection: timeout, partial response, malformed JSON từ LLM
  • Test graceful degradation: LoanBot phải hoạt động khi API down 30 giây
  • Gameday exercise: full simulated outage drill với toàn team
🏢 Tình huống: LoanBot Gameday — Tháng 2/2026

FinTech Corp tổ chức Gameday — buổi tập disaster recovery với LoanBot:

  1. 09:00 — Inject API timeout 30% requests trong 10 phút: LoanBot có switch sang backup pool không?
  2. 09:30 — Simulate CIC API down: LoanBot có queue requests hay crash không?
  3. 10:00 — Inject malformed JSON response 10% cases: LoanBot có parse error gracefully không?
  4. 10:30 — Full primary API outage: degraded mode kích hoạt trong <60 giây không?
  5. 11:00 — Recovery: drain queue, verify all queued decisions processed correctly

Kết quả: team phát hiện 2 bugs trong recovery code và fix trước khi xảy ra real outage.

Python (Chaos Test Suite) 
import random, time
from unittest.mock import patch, MagicMock
from contextlib import contextmanager

@contextmanager
def inject_api_timeout(failure_rate: float = 0.3):
    """Inject random timeouts into Claude API calls."""
    original_call = anthropic_client.messages.create

    def flaky_call(*args, **kwargs):
        if random.random() < failure_rate:
            raise TimeoutError("Injected timeout for chaos testing")
        return original_call(*args, **kwargs)

    with patch.object(anthropic_client.messages, "create", side_effect=flaky_call):
        yield

def test_resilience_under_timeout():
    """LoanBot phải xử lý được 30% API timeouts với retry logic."""
    test_apps = [(f"CHAOS-{i:03d}", 620 + i, 0.35) for i in range(20)]
    success_count = 0

    with inject_api_timeout(failure_rate=0.3):
        for app_id, score, dti in test_apps:
            try:
                result = loanbot.evaluate(credit_score=score, dti=dti)
                if result["decision"] in {"APPROVE", "REJECT", "REVIEW"}:
                    success_count += 1
            except Exception as e:
                print(f"  Failed: {app_id} — {e}")

    # With 30% timeout rate and retry, expect ≥90% success
    success_rate = success_count / len(test_apps)
    assert success_rate >= 0.90, f"Resilience failure: {success_rate:.1%} (expected ≥90%)"
    print(f"✅ Chaos test passed: {success_rate:.1%} success rate under 30% timeout")

def test_malformed_response_handling():
    """LoanBot phải handle khi LLM trả về JSON không hợp lệ."""
    bad_responses = [
        "{invalid json",           # parse error
        '{"decision": "MAYBE"}',   # invalid decision value
        "",                         # empty response
        '{"confidence": 1.5}',      # out-of-range
    ]
    for bad_resp in bad_responses:
        # LoanBot must not crash — return fallback or queue for human review
        with patch("loanbot.call_llm", return_value=bad_resp):
            result = loanbot.evaluate(credit_score=620, dti=0.35)
            assert result is not None                          # no crash
            assert result.get("decision") == "REVIEW"          # fallback to human review

📚 Glossary Module 20

Thuật ngữ Định nghĩa
Testing PyramidMô hình phân tầng tests: nhiều unit tests ở đáy, ít E2E tests ở đỉnh. AI agents thêm Chaos tier.
Deterministic TestTest có output chính xác, lặp lại được — ví dụ: blacklisted=True → REJECT luôn luôn.
Probabilistic TestTest verify properties của output thay vì exact string — dùng cho LLM output thay đổi mỗi lần.
Golden DatasetTập cases có expected outcomes đã được domain expert verify — ground truth để evaluate AI system.
LLM-as-JudgeDùng LLM khác để evaluate output của LoanBot — cho phép check semantic correctness thay vì string match.
Property-Based TestingTest invariants (điều luôn phải đúng) thay vì specific inputs — tìm ra edge cases tự động.
MonotonicityInvariant: better input → not worse output. Credit score cao hơn không bao giờ cho decision tệ hơn.
Hallucination TestVerify LLM không tự bịa ra số liệu, tên, hoặc thông tin không có trong input.
Regression TestTest phát hiện khi thay đổi mới phá vỡ existing behavior — so sánh với baseline snapshot.
Contract TestingVerify interface giữa LoanBot và external tools (CIC, gateway) không thay đổi unexpectedly.
Snapshot TestingLưu known-good output làm baseline, alert khi future output drift quá ngưỡng cho phép.
Chaos EngineeringCố tình inject failures vào hệ thống để test resilience — phát hiện weaknesses trước real incidents.
Fault InjectionKỹ thuật cố tình gây lỗi: timeout, network failure, malformed response — trong controlled environment.
GamedayBuổi tập disaster recovery có kế hoạch — toàn team thực hành xử lý various failure scenarios.
Quality GateCheckpoint tự động trong CI/CD: block deployment nếu test failure, regression drift, hoặc bias violation.

Bài 1: Orchestration Taxonomy — Phân Loại Các Mô Hình Điều Phối

🎯 Sau bài này, bạn sẽ:

  • Phân biệt 3 paradigm: Sequential, Parallel, Event-Driven
  • Chọn pattern phù hợp với từng loại task trong LoanBot
  • Hiểu trade-offs: latency vs cost vs complexity
  • Nhận biết khi nào sequential đủ và khi nào cần parallel

LoanBot v3.0 dùng sequential orchestration — agents chạy lần lượt: CreditAgent → IncomeAgent → FraudAgent → RiskAgent → ComplianceAgent → ReportAgent. Tổng latency = tổng thời gian các agent. Trong Module 13, ta sẽ xem cách restructure để đạt hiệu quả cao hơn.

➡️

Sequential Pipeline

A → B → C. Output A là input B. Đơn giản, dễ debug. Latency = sum(latencies). Dùng khi: mỗi bước phụ thuộc vào bước trước.

🔀

Fan-Out / Fan-In (Parallel)

Nhiều agents chạy đồng thời. Latency = max(latencies). Dùng khi: agents độc lập nhau, cần tốc độ.

📡

Event-Driven (Reactive)

Agents react to events, không cần coordinator trung tâm. Loose coupling, asynchronous. Dùng khi: complex workflows, many producers/consumers.

📌 Khái niệm: Dependency Graph

Để chọn orchestration pattern, vẽ dependency graph — directed acyclic graph (DAG) thể hiện agent nào phụ thuộc vào output của agent nào. Nếu A và B không có cạnh nối → có thể chạy parallel. Chỉ A → B (B cần A) → phải sequential. Trong LoanBot: CreditAgent, IncomeAgent, FraudAgent đều độc lập nhau → Fan-Out. RiskAgent cần cả 3 → chạy sau Fan-In.

🧠 Kiểm tra nhanh

ComplianceAgent có thể chạy song song với FraudAgent không? Hãy trace qua dependency graph để trả lời.

Gợi ý: Không. ComplianceAgent cần RiskAgent output (recommendation) để verify policy compliance — vì compliance check phụ thuộc vào verdict của RiskAgent. Trong DAG: RiskAgent → ComplianceAgent (có cạnh). FraudAgent và ComplianceAgent không có cạnh trực tiếp, nhưng cả hai đều có path đến qua RiskAgent → không thể fan-out. Chi tiết: nếu ComplianceAgent cần biết "là APPROVE hay REJECT" để áp đúng policy → phụ thuộc RiskAgent.

2.2 Fan-Out Implementation với asyncio

Python (async) 
import asyncio
from dataclasses import dataclass
from typing import List, Dict, Any, Optional

@dataclass
class AgentResult:
    agent_name: str
    output: Dict
    latency_ms: int
    success: bool
    error: Optional[str] = None

class FanOutOrchestrator:
    """Fan-Out/Fan-In orchestrator cho LoanBot v3.1."""

    async def fan_out(self, agents: List[dict],
                          loan_data: Dict,
                          timeout_s: float = 5.0) -> List[AgentResult]:
        # Chạy tất cả agents đồng thời với timeout
        tasks = [
            asyncio.wait_for(
                self._call_agent(agent, loan_data),
                timeout=timeout_s
            )
            for agent in agents
        ]
        # gather(return_exceptions=True) → không raise, trả về exception object
        raw = await asyncio.gather(*tasks, return_exceptions=True)
        results = []
        for agent, r in zip(agents, raw):
            if isinstance(r, Exception):
                results.append(AgentResult(
                    agent_name=agent['name'], output={},
                    latency_ms=0, success=False,
                    error=str(r)
                ))
            else:
                results.append(r)
        return results

    def fan_in(self, results: List[AgentResult],
                strategy: str = "all_required") -> Dict:
        # all_required: mọi agent phải thành công
        # best_effort:  dùng available results, flag failures
        failures = [r for r in results if not r.success]
        if strategy == "all_required" and failures:
            failed_names = [r.agent_name for r in failures]
            raise RuntimeError(f"Critical agents failed: {failed_names}")
        return {r.agent_name: r.output for r in results if r.success}

    async def run_pipeline(self, loan_data: Dict) -> Dict:
        # Stage 1: Fan-Out (parallel)
        parallel_agents = [
            {'name': 'CreditAgent', 'fn': credit_agent_fn},
            {'name': 'IncomeAgent', 'fn': income_agent_fn},
            {'name': 'FraudAgent',  'fn': fraud_agent_fn},
        ]
        parallel_results = await self.fan_out(parallel_agents, loan_data)
        aggregated       = self.fan_in(parallel_results, strategy="all_required")
        # Stage 2: Sequential (RiskAgent cần aggregated)
        risk_result = await risk_agent_fn({**loan_data, **aggregated})
        return {"aggregated": aggregated, "risk": risk_result}
⚠️ Cảnh báo: Thundering Herd Problem

Khi 50.000 hồ sơ đến cùng lúc, mỗi hồ sơ fan-out 3 agents = 150.000 concurrent API calls → Anthropic rate limit. Giải pháp: Semaphore giới hạn concurrent calls (e.g., asyncio.Semaphore(50) = tối đa 50 concurrent requests), kết hợp exponential backoff khi gặp 429 (rate limit).

Bài 3: Map-Reduce cho Agent Fleets — Batch Processing

🎯 Sau bài này, bạn sẽ:

  • Áp dụng Map-Reduce pattern để xử lý 50.000 hồ sơ/tháng
  • Thiết kế partitioning strategy phù hợp với LoanBot
  • Implement reducer aggregation cho portfolio-level analytics
  • Kết hợp Anthropic Batch API (50% off) với Map-Reduce

Map-Reduce là pattern xử lý large-scale data: Map = áp dụng function lên mỗi item song song; Reduce = tổng hợp kết quả. Với LoanBot và 50.000 hồ sơ/tháng, Map-Reduce + Batch API = cách tiết kiệm nhất.

💡 Analogy: Dây chuyền nhà máy vs phòng thống kê

Map là dây chuyền nhà máy — mỗi công nhân (agent) xử lý một sản phẩm (hồ sơ) độc lập song song. Reduce là phòng thống kê — tổng hợp tất cả kết quả: có bao nhiêu APPROVE, tổng value, risk distribution. FinTech Corp dùng Map-Reduce cho monthly portfolio report: map = evaluate từng hồ sơ, reduce = aggregate statistics cho ban lãnh đạo.

Hình 2: Map-Reduce Pattern — 50.000 hồ sơ partitioned thành 500 batches × 100; reducer tổng hợp portfolio report
Python 
from dataclasses import dataclass
from typing import List, Dict
import statistics

@dataclass
class PortfolioReducer:
    """Reducer: tổng hợp 50.000 LoanBot results thành portfolio report."""

    def reduce(self, results: List[Dict]) -> Dict:
        total = len(results)
        approved = [r for r in results if r['decision'] == 'APPROVE']
        rejected = [r for r in results if r['decision'] == 'REJECT']
        hitl     = [r for r in results if r.get('hitl_required')]

        approved_value = sum(r['loan_amount_vnd'] for r in approved)
        risk_scores    = [r['risk_score'] for r in results if 'risk_score' in r]

        # Segment breakdown by credit score bucket
        buckets = {'high(720+)': 0, 'medium(560-719)': 0, 'low(<560)': 0}
        for r in results:
            s = r.get('credit_score', 600)
            if   s >= 720: buckets['high(720+)']       += 1
            elif s >= 560: buckets['medium(560-719)']  += 1
            else:           buckets['low(<560)']         += 1

        return {
            'total_applications': total,
            'approved_count':     len(approved),
            'approve_rate':       len(approved) / total,
            'total_approved_vnd': approved_value,
            'hitl_count':         len(hitl),
            'avg_risk_score':     statistics.mean(risk_scores) if risk_scores else 0,
            'risk_p95':           sorted(risk_scores)[int(len(risk_scores)*0.95)] if risk_scores else 0,
            'credit_buckets':     buckets,
        }

Bài 4: Saga Pattern — Distributed Transactions cho Multi-Agent

🎯 Sau bài này, bạn sẽ:

  • Hiểu tại sao cần Saga Pattern trong multi-agent loan disbursement
  • Thiết kế compensating transactions cho mỗi bước
  • Implement SagaOrchestrator với rollback capability
  • Phân biệt Choreography Saga vs Orchestration Saga

Khi loan được APPROVE, LoanBot cần thực hiện chuỗi actions: tạo hợp đồng → giải ngân → cập nhật CRM → gửi thông báo. Nếu bước giải ngân fail sau khi đã tạo hợp đồng → cần rollback để không có inconsistent state. Đây là vấn đề distributed transaction.

📌 Khái niệm: Saga Pattern

Saga = chuỗi local transactions, mỗi transaction cập nhật một service. Nếu một bước fail, Saga thực thi compensating transactions (undo) cho tất cả các bước trước đó. Khác với ACID transaction (all-or-nothing trong một database), Saga work với distributed services mà không cần distributed lock.

Bài 5: Dynamic Routing & Reactive Orchestration

🎯 Sau bài này, bạn sẽ:

  • Thiết kế Dynamic Router dựa trên loan characteristics
  • Implement capability-based agent discovery
  • Xây dựng event-driven orchestration với pub/sub
  • Kết hợp tất cả patterns vào LoanBot v3.1 production architecture

Dynamic Routing là khi orchestrator quyết định routing tại runtime dựa trên content của request — không phải hard-coded flow. LoanBot v3.1 có Dynamic Router: hồ sơ đơn giản → fast path (3 agents), hồ sơ phức tạp → full path (8 agents).

🔑 Key Insight: Routing là Optimization

Không phải mọi hồ sơ cần đủ 8 agents. FC-2024-001 (score 720, loan 200M, clean history) → FraudAgent cần chạy không? CreditAgent đủ tốt để approve trong 1.5 giây. Dynamic Router = right-sizing agent pipeline theo complexity, giảm cost 40% cho simple cases mà không giảm quality.

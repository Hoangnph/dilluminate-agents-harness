1.1 Từ Minimal đến Production Harness

💡 Analogy: Từ xe đạp đến xe hơi

Harness Module 1 (dispatch_tool + agent loop) giống như xe đạp: đủ để đi từ A đến B trong điều kiện lý tưởng. Nhưng để vận chuyển hàng hóa ở quy mô doanh nghiệp — cần xe tải với hệ thống lái, phanh ABS, GPS, camera lùi, bảo hiểm va chạm.

Production harness là xe hơi đầy đủ trang bị: tool orchestration (động cơ mạnh), verification (phanh ABS), memory (GPS + bản đồ), guardrails (túi khí), observability (hộp đen máy bay).

🏢 FinTech Corp: Vấn đề với LoanBot v0.1

Sau khi demo, CTO đặt câu hỏi: "Nếu hệ thống CIC bị timeout, LoanBot sẽ làm gì?" — Câu trả lời: crash. "Nếu nhân viên kiểm tra ngày mai thì biết tại sao LoanBot từ chối hồ sơ không?" — Không có log. "Nếu LoanBot tự phê duyệt khoản vay 2 tỷ đồng thì ai chịu trách nhiệm?" — Không có guardrails.

LoanBot v0.2 cần giải quyết TẤT CẢ những vấn đề này — đó là lý do cần 5-layer harness.

1.2 Kiến Trúc 5 Tầng

Hình 1.1 — Kiến trúc 5 tầng production harness (Model ở dưới cùng làm nền)
🔵 Nguyên tắc phân tầng

Tầng cao hơn bao bọc tầng thấp hơn. Mọi request từ Model đi lên qua các tầng (Tool → Verify → Memory → Guardrail → Observe). Mọi response đi xuống theo chiều ngược lại. Đây giống kiến trúc OSI trong mạng máy tính — mỗi tầng xử lý một khía cạnh cụ thể và không quan tâm đến tầng khác.

1.3 15 Component Model — Bản Đồ Chi Tiết

🗂️ Tool Registry

Đăng ký & khám phá tools, lưu schema + metadata + permissions

⚡ Tool Dispatcher

Nhận tool call từ LLM, route đến function đúng, trả kết quả

🔄 Retry Orchestrator

Tự động retry khi tool fail với exponential backoff

💰 Budget Tracker

Theo dõi số tool calls, token dùng, chi phí per session

⚠️ Error Handler

Xử lý lỗi tool call, phân loại recoverable vs fatal errors

🔀 Fallback Router

Chuyển sang tool backup khi tool chính không available

📋 Instruction Manager

Quản lý system prompt, cập nhật theo context runtime

🏗️ Context Builder

Xây context window từ memory + history + current input

🧠 Memory Manager

Đọc/ghi working memory, trigger compaction khi gần giới hạn

💾 State Persistence

Serialize/deserialize session state để resume sau gián đoạn

🔗 Session Controller

Quản lý lifecycle: create, resume, expire, terminate session

🔐 Permission Resolver

Kiểm tra quyền gọi tool theo role, action, value thresholds

✔️ Input Validator

Validate parameters trước khi dispatch, ngăn injection attacks

👤 HITL Gateway

Dừng agent, chờ human approval cho critical actions

💸 Budget Guardrail

Dừng agent khi vượt ngưỡng chi phí hoặc iteration limit

Tầng 1: Tool Orchestration — Gọi Tools Đáng Tin Cậy

2.1 Tool Registry — Thư Viện Công Cụ

💡 Analogy: Kho dụng cụ của thợ cơ khí

Thay vì mỗi nhân viên tự mang dụng cụ, một kho trung tâm đăng ký mọi dụng cụ: tên, mô tả, ai được dùng, và tình trạng (sẵn sàng/bảo trì). Khi cần, họ "đặt mượn" từ kho.

Tool Registry là kho trung tâm đó — nơi mọi tool được đăng ký với schema, permissions, và health status. Agent không gọi tool trực tiếp — nó hỏi Registry xem tool nào đang available.

Python # Tool Registry cho LoanBot v0.2 class ToolRegistry: def register(self, name: str, fn, schema: dict, permission: str = "auto", max_retries: int = 3): """Đăng ký tool vào registry với metadata đầy đủ.""" self._tools[name] = { "fn": fn, "schema": schema, "permission": permission, # "auto" | "require_hitl" | "blocked" "max_retries": max_retries, "call_count": 0, "error_count": 0, "available": True } def get_schemas(self) -> list: """Trả về danh sách schema cho LLM (chỉ tools available).""" return [t["schema"] for t in self._tools.values() if t["available"]]

2.2 Retry Orchestrator — Kiên Nhẫn Như Robot

💡 Analogy: Nhân viên gọi điện cho đối tác

Khi nhân viên gọi cho CIC mà bận máy, họ không bỏ cuộc ngay — họ đợi 1 phút rồi gọi lại. Nếu vẫn bận, đợi 2 phút. Nếu sau 3 lần vẫn không được, họ ghi nhận và báo cáo quản lý.

Đây chính xác là Exponential Backoff with Jitter — chiến lược retry chuẩn trong distributed systems.

Hình 2.1 — Retry Orchestrator với exponential backoff: tự động thử lại trước khi báo lỗi
Python import time, random def retry_call(fn, kwargs: dict, max_retries: int = 3, base_delay: float = 1.0) -> dict: """Exponential backoff + jitter retry.""" for attempt in range(max_retries): try: return fn(**kwargs) except Exception as e: if attempt == max_retries - 1: raise # Hết lần retry → Error Handler wait = base_delay * (2 ** attempt) + random.uniform(0, 0.5) time.sleep(wait) # Đợi 1s, 2s, 4s... + jitter ngẫu nhiên

2.3 Circuit Breaker — Bộ Ngắt Tự Động

💡 Analogy: Cầu dao điện

Nếu hệ thống CIC liên tục lỗi, không nên cứ gọi mãi — điều đó vừa lãng phí thời gian vừa làm nặng thêm server đang quá tải. Giống như cầu dao điện tự ngắt khi quá tải, Circuit Breaker tự "mở" sau một số lỗi nhất định, và chỉ "đóng" lại sau một thời gian cho phép recovery.

Vòng đời Circuit Breaker
CLOSED → OPEN → HALF-OPEN → CLOSED
Sau N lỗi: mở mạch (OPEN) · Sau timeout: thử lại (HALF-OPEN) · Nếu thành công: đóng mạch (CLOSED)
CLOSED
Hoạt động bình thường — gọi tool thoải mái
HALF-OPEN
Thử nghiệm — gửi 1 request test sau timeout
OPEN
Ngắt — fail fast, không gọi tool, chờ recovery
🏢 FinTech Corp: Tool Orchestration của LoanBot v0.2

Với 5 tools của LoanBot, cấu hình Tool Orchestration:

  • check_credit_score: max_retries=3, circuit breaker threshold=5 errors/minute
  • verify_income: max_retries=3, timeout=10s (hệ thống BHXH chậm hơn)
  • check_blacklist: max_retries=2 (critical — cần nhanh)
  • calculate_dti: max_retries=1 (local calculation, ít fail)
  • generate_report: max_retries=3, với idempotency key (tránh tạo 2 báo cáo)
🧠 Kiểm tra nhanh

Tool check_credit_score của LoanBot đã thất bại 5 lần liên tiếp. Circuit Breaker chuyển sang OPEN. Điều gì xảy ra với request hồ sơ thứ 6?

Gợi ý: Khi Circuit Breaker ở trạng thái OPEN, nó fail fast ngay lập tức — không gọi tool nữa, trả về lỗi ngay. Điều này giúp tránh làm nặng thêm hệ thống CIC đang quá tải. LoanBot sẽ phản hồi: "Không thể kiểm tra điểm tín dụng tạm thời, vui lòng thử lại sau 60 giây." Sau timeout recovery, Circuit Breaker tự chuyển sang HALF-OPEN và thử 1 request test.

💡 Điểm cốt lõi Bài 2

  • Tool Registry là kho trung tâm — agent không gọi tool trực tiếp, luôn qua Registry
  • Retry Orchestrator: exponential backoff (1s → 2s → 4s) + jitter ngẫu nhiên để tránh thundering herd
  • Circuit Breaker: CLOSED → OPEN (sau N errors) → HALF-OPEN (test recovery) → CLOSED
  • Mỗi tool có cấu hình retry và timeout riêng phù hợp với đặc điểm của hệ thống backend đó

Tầng 3: Context & Memory — Bộ Nhớ Bền Vững

3.1 Vấn Đề: Context Drift

⚠️ Context Drift — Kẻ thù thầm lặng

Nghiên cứu từ MemU (2026) đo được: mỗi bước trong multi-step workflow, LLM mất ~2% khả năng truy cập context ban đầu. Sau 5 bước: chỉ còn <60% context gốc. Sau 10 bước: <40%.

Với LoanBot xử lý 4 tool calls + nhiều lượt reasoning: tổng ~8-10 bước. Không có Memory Management, LoanBot có thể "quên" thông tin đầu hồ sơ khi đến bước generate_report!

Hình 3.1 — Context Drift: retention suy giảm theo từng bước nếu không có Memory Management

3.2 Ba Loại Bộ Nhớ

⚡

Working Memory

Thông tin đang xử lý hiện tại. Lưu trong context window. Bị mất khi session kết thúc.

LoanBot: Kết quả tool calls trong session hiện tại, trạng thái reasoning

📚

Long-term Memory

Kiến thức được persist vào storage. Không bị mất qua sessions. Được truy xuất khi cần.

LoanBot: Lịch sử khách hàng, chính sách tín dụng, template quyết định

🗂️

Episodic Memory

Lưu toàn bộ session transcript. Dùng để audit và học từ quá khứ.

LoanBot: Toàn bộ quá trình thẩm định cho audit và compliance

3.3 Memory Compaction — Nén Thông Tin

💡 Analogy: Ghi chú cuộc họp

Trong một cuộc họp dài 3 tiếng, bạn không ghi nhớ từng từ — bạn ghi các điểm quan trọng. Khi phòng họp gần đầy (context window gần hết), Memory Manager "nén" lịch sử thành summary ngắn gọn, giữ lại thông tin quan trọng, bỏ chi tiết dư thừa.

Hình 3.2 — Memory Compaction: nén lịch sử thành summary để giải phóng context window
Python — Memory Manager class MemoryManager: COMPACTION_THRESHOLD = 0.75 # Compaction khi context > 75% đầy def should_compact(self, messages: list, max_tokens: int) -> bool: estimated_tokens = sum(len(str(m)) // 4 for m in messages) return estimated_tokens / max_tokens > self.COMPACTION_THRESHOLD def compact(self, messages: list, client) -> list: """Tóm tắt lịch sử cũ, giữ lại system prompt và input gần nhất.""" system_msg = messages[0] # Giữ nguyên system prompt recent_msgs = messages[-4:] # Giữ 4 messages gần nhất history_msgs = messages[1:-4] # Phần cần nén summary = client.messages.create( # Gọi LLM để tóm tắt model="claude-haiku-4-5-20251001", max_tokens=500, messages=[{"role":"user", "content": f"Tóm tắt ngắn gọn (dưới 200 từ) các bước đã thực hiện:\n{history_msgs}"}] ).content[0].text return [system_msg, {"role": "assistant", "content": f"[SUMMARY] {summary}"}, *recent_msgs]

💡 Điểm cốt lõi Bài 3

  • Context Drift: mất ~2%/bước → sau 10 bước chỉ còn 40% thông tin gốc nếu không quản lý
  • Ba loại memory: Working (context), Long-term (storage), Episodic (audit trail)
  • Compaction trigger: khi context window đạt 75% — tóm tắt lịch sử, giữ lại điểm quan trọng
  • State Persistence: serialize session state → LoanBot có thể resume nếu bị gián đoạn giữa chừng

Tầng 4: Guardrails & Safety — Hệ Thống Kiểm Soát

4.1 Permission Resolver — Ai Được Làm Gì?

💡 Analogy: Hệ thống phân quyền ngân hàng

Tại FinTech Corp, một giao dịch viên có thể duyệt khoản vay đến 50 triệu, trưởng phòng đến 200 triệu, giám đốc đến 500 triệu, và HĐQT cho vay hơn. Đây là Role-Based Access Control (RBAC).

Permission Resolver implement logic tương tự: mỗi action có một permission tier, và harness kiểm tra trước khi cho phép thực thi.

Permission LevelLoanBot ActionsKiểm soát
🟢 AUTOcheck_credit_score, check_blacklist, calculate_dtiTự động, không cần duyệt
🟡 VALIDATEverify_income, generate_reportValidate dữ liệu đầu vào trước
🔴 REQUIRE_HITLapprove_loan >500 triệu, reject khi score biênDừng, chờ human approval
⛔ BLOCKEDdelete_record, modify_credit_scoreTừ chối hoàn toàn, log incident

4.2 HITL Gateway — Cổng Kiểm Soát Con Người

Hình 4.1 — HITL Gateway: LoanBot dừng tự động khi cần human approval cho khoản vay lớn

4.3 Input Validation — Bức Tường Ngăn Lỗi

💡 Analogy: Bộ phận nhận hồ sơ

Trước khi hồ sơ đến tay nhân viên thẩm định, bộ phận nhận hồ sơ đã kiểm tra: đủ giấy tờ chưa? Chữ ký đúng chỗ chưa? Mẫu đơn đúng loại chưa? Điều này tránh mất thời gian xử lý hồ sơ không hợp lệ.

Input Validator làm điều tương tự cho tool calls: kiểm tra kiểu dữ liệu, phạm vi giá trị, bắt buộc tham số, và ngăn injection attacks — trước khi thực thi bất kỳ tool nào.

Python — Input Validator def validate_tool_input(tool_name: str, inputs: dict) -> tuple[bool, str]: """Validate inputs trước khi dispatch. Returns (is_valid, error_message).""" rules = { "check_credit_score": { "customer_id": lambda v: v.startswith("FC-") }, "calculate_dti": { "monthly_income": lambda v: 1 <= v <= 500, # triệu VNĐ "requested_loan": lambda v: 10 <= v <= 5000, # triệu VNĐ "term_months": lambda v: v in [12, 24, 36, 48, 60, 84, 120], }, "generate_report": { "decision": lambda v: v in ["APPROVED", "REJECTED", "NEEDS_HUMAN_REVIEW"] } } for field, check in rules.get(tool_name, {}).items(): if field in inputs and not check(inputs[field]): return False, f"Giá trị không hợp lệ: {field}={inputs[field]}" return True, ""

💡 Điểm cốt lõi Bài 4

  • Permission Resolver: 4 tiers — AUTO, VALIDATE, REQUIRE_HITL, BLOCKED — cho mỗi action
  • HITL Gateway: agent tự dừng và chờ approval khi gặp action có rủi ro cao (>500tr, quyết định biên)
  • Input Validator: validate trước khi thực thi — kiểu dữ liệu, phạm vi, enum values
  • Budget Guardrail: giới hạn max_iterations và max_cost_per_session để tránh vòng lặp vô hạn tốn tiền

Tầng 5: Observability — Nhìn Thấy Mọi Thứ

5.1 Tại Sao Observability Là Bắt Buộc?

⚠️ Thực tế từ enterprise deployments

"Khách hàng phàn nàn LoanBot từ chối hồ sơ sai. Nhưng chúng tôi không thể biết tại sao — không có log!" — Một tình huống thực tế từ enterprise. Không có observability, bạn không thể debug, audit, hoặc prove compliance.

Với FinTech Corp, EU AI Act (Module 5) bắt buộc lưu audit trail cho mọi quyết định tín dụng. Nếu không có observability từ đầu, retrofit sẽ cực kỳ tốn kém.

5.2 Ba Trụ Cột Observability

Hình 5.1 — Ba trụ cột Observability: Logs (gì đã xảy ra) + Traces (mất bao lâu) + Metrics (xu hướng)

5.3 Structured Audit Log — Định Dạng Chuẩn

JSON — Audit Log Event { "event_id": "evt-20260626-143052-001", "trace_id": "trace-FC-2024-001-sess42", "timestamp": "2026-06-26T14:30:52.341Z", "event_type": "tool_call", "agent_id": "loanbot-v0.2", "session_id": "sess-FC-2024-001-42", "tool_name": "check_credit_score", "input": { "customer_id": "FC-2024-001", "include_history": true }, "output": { "score": 720, "risk_level": "thấp" }, "latency_ms": 245, "status": "success", "permission_level": "auto", "retry_count": 0 }
🏢 FinTech Corp: LoanBot Observability Requirements

Theo yêu cầu của Compliance và EU AI Act (tiên lượng cho Module 5), mỗi quyết định LoanBot phải có:

  • 📋 Audit log đầy đủ: mọi tool call, kết quả, thời gian, người dùng
  • 🔍 Trace: trace_id để ghép nối toàn bộ hành trình của một hồ sơ
  • 📊 Metrics: approval_rate, avg_processing_time, cost_per_application
  • 🚨 Alert: khi approval_rate <80% hoặc latency >5 phút → notify Compliance team
  • 🗄️ Retention: giữ logs ≥7 năm (theo quy định tài chính Việt Nam)

💡 Điểm cốt lõi Bài 5

  • Ba trụ cột: Logs (gì đã xảy ra), Traces (mất bao lâu, ở đâu), Metrics (xu hướng, alert)
  • Structured logging (JSON) là bắt buộc — plain text log không thể query hay phân tích tự động
  • trace_id duy nhất per request để ghép nối toàn bộ hành trình end-to-end
  • Compliance yêu cầu retention ≥7 năm với FinTech — xây observability ngay từ đầu, không phải retrofit

Thuật Ngữ Quan Trọng Module 2

Tool Registry
Kho trung tâm đăng ký tất cả tools với schema, metadata, và trạng thái. Agent luôn qua Registry để biết tool nào available.
Exponential Backoff
Chiến lược retry: tăng thời gian chờ theo cấp số nhân (1s, 2s, 4s...) để tránh làm nặng thêm server đang quá tải.
Circuit Breaker
Pattern ngắt kết nối tool sau N lỗi liên tiếp. Ba trạng thái: CLOSED (bình thường), OPEN (ngắt), HALF-OPEN (test recovery).
Memory Compaction
Kỹ thuật nén lịch sử hội thoại thành summary ngắn khi context window gần đầy (~75%), giải phóng không gian cho các bước tiếp.
HITL (Human-in-the-Loop)
Điểm trong workflow nơi agent tự dừng, gửi notification cho con người, và chờ approval trước khi tiếp tục hành động rủi ro cao.
Trace ID
Mã định danh duy nhất cho một request end-to-end. Dùng để ghép nối tất cả logs và spans của một hồ sơ vay từ đầu đến cuối.
RBAC (Role-Based Access Control)
Hệ thống phân quyền dựa trên vai trò: giao dịch viên, trưởng phòng, giám đốc có quyền khác nhau với từng loại action.
State Persistence
Serialize và lưu toàn bộ trạng thái agent (working memory, tool results, reasoning) vào storage để có thể resume nếu bị gián đoạn.

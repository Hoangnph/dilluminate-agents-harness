Bài 1: Memory Taxonomy — 4 Loại Bộ Nhớ Của Agent

  • Nhận biết memory bottleneck trong LoanBot v3.0
  • Hiểu tại sao context window KHÔNG thể thay thế persistent memory

LoanBot v3.0 mỗi lần xử lý hồ sơ đều bắt đầu "từ đầu" — không nhớ gì về customer từ lần trước. FC-2024-004 (Phạm Thị D) đã gửi 3 hồ sơ trong 6 tháng; mỗi lần RiskAgent phải re-analyze từ đầu, không tận dụng insight lần trước. Đây là stateless memory problem.

⚡ Sensory Memory

Context Window (Input Buffer)

Token stream hiện tại trong một inference call. LoanBot: 200K tokens = ~150 trang. Mất khi inference xong. Analog: những gì bạn đang đọc ngay lúc này.

💬 Short-term Memory

Conversation / Session State

Message history trong một session. LoanBot: context qua 3–5 turns khi loan officer trao đổi với AI. Mất khi session end. Analog: cuộc hội thoại đang diễn ra.

📚 Long-term Memory

Vector Store + Database

Persistent knowledge: 5 năm lịch sử quyết định, customer profiles, regulation embeddings. LoanBot cần đây để nhớ FC-2024-004 đã bị từ chối 3 lần và lý do. Không mất.

⚙️ Procedural Memory

Tool Use Patterns + Fine-tuned Weights

"Kỹ năng" — cách dùng tools, workflow patterns. LoanBot: biết khi nào gọi CreditBureau API, khi nào escalate to HITL. Encoded trong model weights hoặc system prompt.

📌 Khái niệm: The Memory Bottleneck

Context window của claude-haiku-4-5-20251001 là 200K tokens — nghe có vẻ lớn nhưng: 1 customer profile = ~500 tokens, 50.000 customers = 25M tokens — không thể fit. 500 quy định pháp lý = ~2M tokens — quá giới hạn. Giải pháp: chỉ retrieve những gì relevant vào context lúc cần — đây là RAG (Retrieval-Augmented Generation).

LoanBot cần nhớ rằng "3 tháng trước, FC-2024-004 đã bị từ chối vì income không đủ". Đây là loại memory nào? Được lưu ở đâu?

Gợi ý: Đây là Long-term Memory — thông tin tồn tại vượt qua session boundary (3 tháng). Bài 2: RAG Architecture — Retrieval-Augmented Generation

🎯 Sau bài này, bạn sẽ:

  • Hiểu cơ chế RAG: Index → Retrieve → Augment → Generate
  • Phân biệt Naive RAG vs Advanced RAG (HyDE, Reranking, Multi-Query)
  • Thiết kế RAG pipeline cho LoanBot regulation search
  • Đo lường chất lượng RAG với RAGAS metrics

RAG (Retrieval-Augmented Generation) là pattern cho phép LLM truy cập knowledge bên ngoài training data. Thay vì "nhớ" tất cả — LLM retrieve thông tin relevant, đọc nó, rồi generate response. LoanBot dùng RAG để: (1) tìm quy định liên quan đến từng loại hồ sơ, (2) tra cứu precedents từ lịch sử quyết định tương tự.

💡 Analogy: RAG như thẩm phán tra cứu luật

Một thẩm phán giỏi không nhớ hết tất cả bộ luật. Khi xét xử một case, họ tra cứu (retrieve) các điều luật liên quan, đọc (augment context), rồi ra phán quyết (generate). RiskAgent của LoanBot làm đúng như vậy: retrieve các quy định của NHNN liên quan đến hồ sơ vay 300M VNĐ, đọc chúng, rồi generate risk assessment. Không cần nhớ hết — chỉ cần retrieve đúng.

2.1 RAG Pipeline: 4 Bước Cốt Lõi

①
Index — Xây dựng Vector Index

Offline: chunk documents (regulations, loan policies, historical decisions) → embed thành vectors → lưu vào vector store. LoanBot: 500 văn bản quy định → ~15.000 chunks → 15.000 embeddings 1536-dim.

②
Retrieve — Tìm kiếm Semantic

Online: embed query (loan application) → vector similarity search → top-K most relevant chunks. LoanBot: "hồ sơ vay 300M VNĐ, tài sản thế chấp là bất động sản" → retrieve 5 regulation chunks về collateral requirements.

③
Augment — Inject vào Context

Ghép retrieved chunks vào system prompt: "Given these regulations: [chunk1] [chunk2]... assess the following loan application...". Context window giờ chứa relevant knowledge thay vì toàn bộ knowledge base.

④
Generate — LLM tạo Answer

RiskAgent generate assessment dựa trên (1) loan data + (2) retrieved regulation chunks. Output: recommendation với citations đến cụ thể điều khoản nào → auditable, explainable.

3.1 Chunking Strategy

📏 Fixed-Size Chunking

512 tokens với 50 token overlap. Đơn giản, predictable. Phù hợp: unstructured text, general docs. Nhược: cắt giữa câu, mất ngữ cảnh.

📄 Semantic Chunking

Chunk theo paragraph/section boundaries. Giữ nguyên ngữ nghĩa. Phù hợp: regulations, legal docs. LoanBot dùng cho 500 văn bản quy định.

🔵 Hierarchical Chunking

Parent + child chunks. Parent = section (2K tokens), child = sub-section (512 tokens). Retrieve child, augment với parent context. Best for complex docs.

🏢 Tình huống: LoanBot Regulation Knowledge Base

FinTech Corp có 500 văn bản quy định từ NHNN (Ngân hàng Nhà nước), 200 nội quy nội bộ, và 5 năm lịch sử quyết định (30.000 cases). Chunking strategy:

  • Regulations (NHNN): Hierarchical chunking — circular → article → clause. Metadata: circular_number, effective_date, applies_to_loan_type.
  • Internal policy: Semantic chunking theo section. Metadata: policy_version, applicable_amount_range.
  • Historical decisions: Fixed-size per case record. Metadata: loan_id, decision, credit_score, amount, date, outcome (default/not).
Python 
from dataclasses import dataclass
from typing import List, Dict, Optional
import hashlib, textwrap

@dataclass
class Chunk:
    chunk_id:    str
    content:     str
    metadata:    Dict           # filterable fields
    parent_id:   Optional[str]  # for hierarchical chunks
    token_count: int

class LoanBotChunker:
    """Semantic chunker cho LoanBot regulation documents."""

    def chunk_regulation(self, doc_text: str, metadata: Dict) -> List[Chunk]:
        # Split theo điều khoản (Điều 1, Điều 2...)
        sections = self._split_by_articles(doc_text)
        chunks   = []
        for i, section in enumerate(sections):
            content = section.strip()
            if len(content) < 50:
                continue  # bỏ qua section quá ngắn
            chunk_id = hashlib.md5(content.encode()).hexdigest()[:12]
            chunks.append(Chunk(
                chunk_id   = chunk_id,
                content    = content,
                metadata   = {**metadata, 'article_index': i + 1},
                parent_id  = metadata.get('circular_number'),
                token_count = len(content.split()) * 4 // 3  # estimate
            ))
        return chunks

    def chunk_loan_decision(self, decision: Dict) -> Chunk:
        # Mỗi loan decision = 1 chunk, structured text
        content = (
            f"Loan Decision Record\n"
            f"ID: {decision['loan_id']} | Date: {decision['date']}\n"
            f"Customer: credit_score={decision['credit_score']} "
            f"income={decision['income_vnd']}\n"
            f"Amount: {decision['amount_vnd']:,} VNĐ\n"
            f"Decision: {decision['decision']} | Risk: {decision['risk_score']}\n"
            f"Reason: {decision['reason']}\n"
            f"Outcome: {decision.get('outcome', 'pending')}"
        )
        chunk_id = hashlib.md5(content.encode()).hexdigest()[:12]
        return Chunk(
            chunk_id   = chunk_id,
            content    = content,
            metadata   = decision,  # full metadata for filtering
            parent_id  = None,
            token_count = len(content.split()) * 4 // 3
        )

Dense search (vector similarity) giỏi tìm semantic matches — "quy định về thế chấp" match với "collateral requirements". Sparse search (BM25/TF-IDF) giỏi tìm exact keyword — "Điều 15 Thông tư 39/2016/TT-NHNN". Production LoanBot dùng Hybrid Search với RRF (Reciprocal Rank Fusion) để kết hợp cả hai: dense_score × α + sparse_score × (1-α), thường α = 0.7. Qdrant hỗ trợ hybrid search native từ v1.7.

Bài 4: Memory Management — Context Window Optimization

🎯 Sau bài này, bạn sẽ:

  • Hiểu token budget management trong multi-step agent pipelines
  • Implement conversation summarization để duy trì long-term sessions
  • Thiết kế memory compression cho LoanBot review sessions kéo dài
  • Tránh "context pollution" — noise trong context giảm quality

LoanBot v3.1 có loan officer review session có thể kéo dài 2–3 giờ với hàng chục turns. Nếu stack tất cả messages vào context window → overflow. Giải pháp: progressive summarization — khi context gần đầy, summarize oldest messages và giữ summary.

⚠️ Cảnh báo: Context Pollution

Không phải mọi retrieved chunk đều hữu ích. Retrieve 20 chunks rồi nhét tất cả vào context = "context pollution" — model bị distract bởi irrelevant information, hallucination tăng, quality giảm. Rule: retrieve top-20, rerank → giữ top-5. Faithfulness score drops 15% khi dùng top-10 thay vì top-5 trong LoanBot benchmarks.

4.1 Token Budget Manager

Python 
from dataclasses import dataclass, field
from typing import List, Tuple
import anthropic

@dataclass
class TokenBudget:
    """Token budget cho LoanBot context window (200K total)."""
    total_limit:     int = 200_000
    system_reserved: int = 8_000   # system prompt + tools
    rag_budget:      int = 12_000   # 5 regulation chunks × 2400 tokens
    history_budget:  int = 20_000   # conversation history
    loan_data_budget:int = 4_000    # loan application data
    output_reserved: int = 4_096    # max output tokens

    @property
    def available(self) -> int:
        used = (self.system_reserved + self.rag_budget +
                self.history_budget + self.loan_data_budget + self.output_reserved)
        return self.total_limit - used  # safety buffer

class ConversationMemory:
    """Rolling window memory với auto-summarization."""

    def __init__(self, budget: TokenBudget, client: anthropic.Anthropic):
        self.budget   = budget
        self.client   = client
        self.messages: List[dict] = []
        self.summary: str  = ""
        self._token_est = 0

    def _estimate_tokens(self, text: str) -> int:
        return len(text) // 4  # rough estimate: 4 chars/token

    def add_message(self, role: str, content: str):
        self.messages.append({"role": role, "content": content})
        self._token_est += self._estimate_tokens(content)
        if self._token_est > self.budget.history_budget * 0.8:
            self._compress()  # compress when 80% of budget used

    def _compress(self):
        # Giữ lại 5 messages gần nhất, summarize phần còn lại
        to_summarize = self.messages[:-5]
        if not to_summarize: return
        conv_text = "\n".join(f"{m['role']}: {m['content']}" for m in to_summarize)
        response = self.client.messages.create(
            model="claude-haiku-4-5-20251001",
            max_tokens=500,
            messages=[{"role": "user", "content":
                f"Summarize this loan review conversation in ≤3 sentences, preserving key decisions:\n{conv_text}"}]
        )
        self.summary   = response.content[0].text
        self.messages  = self.messages[-5:]  # keep recent messages
        self._token_est = self._estimate_tokens(self.summary) + sum(
            self._estimate_tokens(m['content']) for m in self.messages
        )

    def build_context(self) -> List[dict]:
        # Prepend summary nếu có, rồi recent messages
        messages = []
        if self.summary:
            messages.append({"role": "user",
                             "content": f"[Previous conversation summary]: {self.summary}"})
            messages.append({"role": "assistant", "content": "Understood."})
        messages.extend(self.messages)
        return messages
🧠 Kiểm tra nhanh

Tại sao không thể dùng history_budget = total_limit - system_reserved (dùng hết context cho history)?

Gợi ý: Vì trong mỗi inference call của LoanBot, cần budget cho: (1) RAG retrieved chunks (12K tokens = 5 regulation sections), (2) loan application data (4K), (3) output tokens (4K). Nếu history chiếm hết context, không còn chỗ cho retrieved knowledge → RAG không work. TokenBudget partition rõ ràng từng thành phần: system + rag + history + data + output ≤ 200K. Ngoài ra: "Lost in the middle" problem — context quá dài làm model ignore information ở giữa.

Bài 5: Production RAG — Evaluation, Optimization, Caching

🎯 Sau bài này, bạn sẽ:

  • Evaluate RAG quality với RAGAS: Faithfulness, Relevancy, Precision, Recall
  • Implement semantic caching để giảm 60% redundant retrievals
  • Thiết kế quality gate cho RAG: Faithfulness ≥ 0.85 trước khi serve
  • Xây dựng feedback loop để improve retrieval over time

RAG trong production cần đáp ứng hai yêu cầu: quality (retrieved information phải accurate, relevant) và efficiency (latency thấp, cost thấp). LoanBot RAG target: Faithfulness ≥ 0.85, latency P95 ≤ 800ms, cache hit rate ≥ 40%.

🔑 Key Insight: Faithfulness vs Relevancy

Faithfulness: answer có được grounded trong retrieved contexts không? LoanBot: "Theo Điều 15 TT39, collateral value phải đạt 120% loan amount" — claim này có trong retrieved text không? (0 = hallucinated, 1 = fully grounded). Answer Relevancy: retrieved contexts có relevant với query không? Retrieve 5 chunks về "thế chấp" nhưng query là "credit score minimum" → low relevancy. Cả hai cần ≥ 0.85 trong LoanBot để đảm bảo EU AI Act Art.13 transparency requirement.

from dataclasses import dataclass
import statistics

@dataclass
class RAGQualityGate:
    """Quality gate: check faithfulness trước khi serve LoanBot RAG response."""
    faithfulness_min: float = 0.85
    relevancy_min:    float = 0.80
    precision_min:    float = 0.75

    def evaluate(self, query: str, answer: str,
                    contexts: List[str]) -> dict:
        # Mock RAGAS scores — thực tế gọi RAGAS evaluator
        scores = self._compute_ragas(query, answer, contexts)
        passed = (
            scores['faithfulness']       >= self.faithfulness_min and
            scores['answer_relevancy']   >= self.relevancy_min    and
            scores['context_precision']  >= self.precision_min
        )
        return {
            'passed': passed,
            'scores': scores,
            'action': 'serve' if passed else 'hitl_fallback'
        }

                          answer: str, contexts: List[str]) -> dict:

📚 Glossary Module 14

Thuật ngữ Định nghĩa
RAG (Retrieval-Augmented Generation)Pattern cho phép LLM truy cập external knowledge: retrieve relevant docs → augment context → generate grounded response.
Sensory MemoryContext window của model trong một inference call. Ephemeral — mất khi call kết thúc.
Long-term MemoryPersistent storage (vector store + DB) cho knowledge tồn tại qua nhiều sessions và users.
Vector EmbeddingDense numerical representation của text trong high-dimensional space, cho phép semantic similarity search.
Cosine SimilarityMetric đo semantic similarity giữa 2 vectors: 1.0 = identical, 0.0 = unrelated, -1.0 = opposite.
ChunkingChia document lớn thành các đoạn nhỏ hơn để embed và retrieve. Strategy: fixed-size, semantic, hierarchical.
RerankingBước sau initial retrieval: dùng cross-encoder model để re-score và re-order top-K results, giữ top-5.
Hybrid SearchKết hợp dense (vector) và sparse (BM25) retrieval với RRF score fusion để tối ưu recall.
FaithfulnessRAGAS metric: mỗi claim trong answer có được support bởi retrieved context không. 0–1 scale.
Answer RelevancyRAGAS metric: answer có thực sự trả lời câu hỏi gốc không (relevant to query).
Context PollutionKhi retrieve quá nhiều irrelevant chunks → model bị distract, hallucination tăng, quality giảm.
Semantic CacheCache lưu query embeddings và responses. Query mới với cosine sim ≥ 0.95 → serve cached response.
Progressive SummarizationKhi conversation history vượt budget → summarize oldest messages, giữ recent messages + summary.
HyDE (Hypothetical Document Embeddings)Advanced RAG: generate hypothetical answer → embed it → use for retrieval. Improves recall for complex queries.
pgvectorPostgreSQL extension cho vector operations, cho phép store và search embeddings trong existing DB.

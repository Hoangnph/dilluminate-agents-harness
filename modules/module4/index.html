Production Readiness Checklist — Chuẩn Bị Trước Khi Deploy

🎯 Sau bài này, bạn sẽ:

  • Biết 5 nhóm kiểm tra trong Production Readiness Checklist (Correctness, Security, Performance, Compliance, Reliability)
  • Hiểu tại sao có những checks là BLOCKING — không pass thì không được deploy
  • Áp dụng checklist để đánh giá mức độ "sẵn sàng" của LoanBot v1.0

1.1 Vấn Đề: Lab-to-Production Gap

⚠️ Con số đáng lo: 37% Lab-to-Production Gap

Nghiên cứu từ Anthropic (2025) đo được: AI agents đạt điểm cao trong lab thường giảm hiệu năng trung bình 37% trong môi trường production thực tế. Nguyên nhân:

  • Real data có edge cases không có trong test set
  • Network latency và backend failures cao hơn lab
  • Concurrent requests tạo race conditions không lường trước
  • Memory leaks tích lũy theo thời gian
💡 Analogy: Checklist phi công trước khi cất cánh

Phi công không cất cánh khi chưa hoàn thành pre-flight checklist — dù đã bay ngàn chuyến. Mỗi chuyến đều kiểm tra: nhiên liệu, hệ thống thủy lực, đèn hiệu, thông tin liên lạc...

Production Readiness Checklist cho AI agent hoạt động tương tự — buộc phải kiểm tra mọi khía cạnh trước khi "cất cánh" với real users.

Docker & Containerization — Đóng Gói LoanBot

2.1 Tại Sao Cần Container?

💡 Analogy: Container vận chuyển hàng hóa

Trước khi có container vận chuyển tiêu chuẩn, mỗi chuyến tàu phải bốc dỡ từng kiện hàng — chậm, tốn công, hàng dễ vỡ. Container tiêu chuẩn hoá tất cả: cùng một kích thước, cẩu thế nào cũng được.

Docker container làm điều tương tự cho phần mềm: đóng gói LoanBot cùng tất cả dependencies vào một "hộp" chuẩn. Laptop, staging server, production cloud — chạy giống hệt nhau. Hết chuyện "chạy trên máy tôi nhưng không chạy được trên server".

Hình 2.1 — LoanBot Docker Architecture: 3 containers, load balanced, với auto-scaling

2.2 Dockerfile LoanBot v1.0

Dockerfile — LoanBot v1.0 # Multi-stage build: stage 1 install deps, stage 2 copy only needed files FROM python:3.11-slim AS builder WORKDIR /build COPY requirements.txt . RUN pip install --user --no-cache-dir -r requirements.txt FROM python:3.11-slim # Security: chạy với user không phải root RUN useradd --create-home loanbot WORKDIR /app # Copy chỉ những gì cần, không copy .env hay secrets COPY --from=builder /root/.local /home/loanbot/.local COPY --chown=loanbot:loanbot ./src /app/src COPY --chown=loanbot:loanbot ./config /app/config # API key KHÔNG được bake vào image — lấy từ Secrets Manager runtime ENV PYTHONUNBUFFERED=1 ENV PYTHONPATH=/app EXPOSE 8080 USER loanbot # Health check — Kubernetes dùng để restart khi unhealthy HEALTHCHECK --interval=30s --timeout=5s --retries=3 \ CMD curl -f http://localhost:8080/health || exit 1 CMD ["python", "-m", "uvicorn", "src.main:app", "--host", "0.0.0.0", "--port", "8080"]
🔵 Best Practices: Docker cho AI Agents
  • Multi-stage build: giảm image size từ ~2GB xuống ~300MB
  • Non-root user: chạy với user riêng, không root — giảm attack surface
  • HEALTHCHECK: Kubernetes restart container tự động khi unhealthy
  • KHÔNG bake secrets vào image: API keys, DB passwords lấy từ Secrets Manager lúc runtime
  • Pin exact versions: python:3.11.9-slim thay vì python:3-slim để tránh surprise upgrades

💡 Điểm cốt lõi Bài 2

  • Container = đóng gói app + dependencies → chạy nhất quán trên mọi môi trường
  • Multi-stage Dockerfile: giảm image size, không bao gồm build tools trong production image
  • Secrets KHÔNG BAO GIỜ bake vào Docker image — dùng Secrets Manager (AWS/GCP/Azure)
  • HEALTHCHECK bắt buộc để orchestrator (Kubernetes) biết khi nào container cần restart

CI/CD Pipeline — Tự Động Hóa Deploy

3.1 CI/CD là gì?

💡 Analogy: Dây chuyền kiểm tra chất lượng nhà máy

Khi sản phẩm ra khỏi máy, ngay lập tức đi qua băng chuyền kiểm tra: đo kích thước, kiểm tra màu sắc, test độ bền... Chỉ sản phẩm qua tất cả kiểm tra mới được đóng hộp và giao hàng.

CI/CD Pipeline là băng chuyền đó cho code: mỗi lần developer push code → tự động chạy tests, build Docker image, chạy evaluation suite, rồi deploy nếu tất cả pass.

Hình 3.1 — CI/CD Pipeline: tự động từ git push đến production, với Evaluation Suite gate bắt buộc

3.2 Quality Gate — Không Qua Là Dừng

🔵 Quality Gate trong CI/CD cho AI Agent

AI agent pipeline cần có Quality Gate đặc biệt mà software thông thường không có: chạy Evaluation Suite (12 metrics) trước mỗi deploy. Nếu F1 giảm từ 95.5% xuống 93% (dù mọi unit test vẫn pass), pipeline phải FAIL và không deploy.

Đây là điểm khác biệt quan trọng giữa CI/CD cho AI và CI/CD cho software thông thường — metric regression là bug nghiêm trọng.

YAML — GitHub Actions: Evaluation Gate jobs: evaluation-gate: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - name: Run LoanBot Evaluation Suite run: | python scripts/eval_suite.py \ --model ${{ secrets.MODEL_NAME }} \ --test-cases tests/ground_truth.json \ --output eval_results.json - name: Check Quality Gates run: | python scripts/check_quality_gates.py \ --results eval_results.json \ --f1-threshold 0.95 \ --latency-p95-s 300 \ --faithfulness 0.99 # Script exits 1 nếu bất kỳ gate nào fail → pipeline stops

💡 Điểm cốt lõi Bài 3

  • CI/CD = Continuous Integration / Continuous Deployment — tự động hóa mọi bước từ commit đến production
  • Quality Gate bắt buộc: Eval Suite (F1≥95%, Faithfulness≥99%) phải pass trước khi deploy
  • Canary deployment: deploy lên 10% traffic trước, monitor 1 giờ, rồi mới rollout 100%
  • Mỗi bước CI/CD nếu fail thì DỪNG — không bao giờ deploy code chưa pass evaluation

Monitoring & Alerting — Theo Dõi Production

4.1 Monitoring vs Observability

💡 Analogy: Giám sát vs Điều tra

Monitoring = đặt camera trong cửa hàng: biết có trộm xảy ra (anomaly detected). Observability = camera + microphone + cảm biến chuyển động + log khách vào ra: biết ai, ở đâu, làm gì, bao giờ.

Module 2 đã xây Observability (structured logs + traces + metrics). Bài này tập trung vào Monitoring: xem metrics theo thời gian thực và tự động alert khi có vấn đề.

Scaling & Cost Optimization — Mở Rộng Hiệu Quả

5.1 Scaling Strategies

💡 Analogy: Quán phở mở thêm quán

Quán phở bận không có nghĩa là phải thuê bếp to hơn — mở thêm chi nhánh thứ 2 rồi thứ 3 khi cần. Khi khách vắng, đóng chi nhánh 2 và 3. Chi phí tỷ lệ thuận với nhu cầu thực tế.

Horizontal scaling (thêm containers) thường tốt hơn vertical scaling (server to hơn) cho AI agents — linh hoạt và cost-efficient hơn.

📊 Load-based Scaling

  • Scale khi CPU >70% hoặc queue depth >100
  • Min 2 replicas (HA), Max 10 replicas
  • Scale down sau 5 phút idle
  • LoanBot: business hours 8AM-10PM

🕐 Scheduled Scaling

  • Pre-scale trước 8AM (peak intake)
  • Scale down 10PM-6AM
  • Tháng cuối quý: tăng thêm capacity
  • Tiết kiệm 40% chi phí vs always-on

💰 Cost Optimization

  • Cache results cho identical requests
  • Batch processing khi non-urgent
  • Dùng Haiku cho simple cases, Sonnet cho complex
  • Spot instances cho worker containers
Hình 5.1 — Auto-scaling: cost tăng linear với volume, tiết kiệm 99.7% so với xử lý thủ công
ROI — Lợi tức đầu tư AI của FinTech Corp
ROI = (70 tỷ − 219 triệu) / 70 tỷ = 99.7%
Manual: 200K × 350K VNĐ = 70 tỷ VNĐ/tháng  |  LoanBot: ~219 triệu VNĐ/tháng
🧠 Kiểm tra nhanh

FinTech Corp muốn scale LoanBot lên 200K hồ sơ/tháng (gấp 4 lần). Chi phí manual sẽ tăng bao nhiêu? Chi phí LoanBot sẽ tăng bao nhiêu? AI có còn lợi thế không?

Manual: 200K × 350K VNĐ = 70 tỷ VNĐ/tháng (gấp 4 lần, tuyến tính)

LoanBot: API cost ≈ $8,600/tháng ≈ 220 triệu VNĐ. Infra cost tăng (thêm replicas) nhưng vẫn dưới 10K VNĐ/hồ sơ.

Kết luận: AI vẫn tiết kiệm 99.7%. Đây là superlinear advantage — khi volume tăng, lợi thế tuyệt đối của AI tăng theo (từ 17.5 tỷ/tháng lên 69.8 tỷ/tháng tiết kiệm).

💡 Điểm cốt lõi Bài 5

  • Horizontal scaling (nhiều containers nhỏ) > Vertical scaling (1 server to) cho AI agents
  • Scheduled scaling: pre-scale trước business hours, scale down ban đêm → tiết kiệm 40% infra cost
  • Model tiering: Haiku ($0.25/M tokens) cho simple cases, Sonnet cho complex — giảm API cost 60%
  • LoanBot scale tuyến tính: 50K→200K hồ sơ = $2,150→$8,600/tháng, vẫn tiết kiệm 99.7% vs thủ công

Thuật Ngữ Quan Trọng Module 4

Docker Container
Đơn vị đóng gói phần mềm cùng tất cả dependencies, chạy nhất quán trên mọi môi trường. Nhẹ hơn Virtual Machine vì dùng chung OS kernel.
CI/CD Pipeline
Chuỗi tự động: code push → build → test → eval → staging → production. Mỗi bước fail thì dừng, không deploy code xấu lên production.
Quality Gate
Điểm kiểm tra bắt buộc trong CI/CD: nếu metrics (F1, Faithfulness, Latency) không đạt ngưỡng, pipeline dừng và không deploy.
Canary Deployment
Deploy lên một phần nhỏ traffic (10%) trước, monitor, rồi mới rollout toàn bộ. Giảm rủi ro khi deploy phiên bản mới.
Horizontal Scaling
Thêm nhiều instances (containers/nodes) thay vì nâng cấp server hiện tại. Linh hoạt hơn, không có single point of failure.
Lab-to-Production Gap
Sự chênh lệch hiệu năng giữa môi trường lab (lý tưởng) và production (thực tế). Trung bình 37% với AI agents — giảm bằng Production Checklist.
Multi-stage Dockerfile
Dockerfile dùng nhiều FROM stages: stage 1 build/compile, stage 2 chỉ copy runtime artifacts. Giảm image size từ GB xuống MB.
HEALTHCHECK
Lệnh trong Dockerfile để container orchestrator (Kubernetes) biết container có healthy không. Nếu fail → restart tự động.

🗣️ Multi-Agent Debate & Consensus

LoanBot v3.8 đang sai trên 12% borderline cases (score 580–680). Thay vì fine-tune một mình, FinTech Corp tổ chức panel tranh luận: OptimistAgent, PessimistAgent, và RegulatorAgent cùng xem xét hồ sơ — rồi một Meta-Arbitrator tổng hợp. Accuracy tăng từ 78% → 91% trên borderline cases.

⏱ 3 tuần 📖 5 bài giảng 🗣️ Collective Intelligence

Bài 1: Tại Sao Debate? — Wisdom of Crowds cho LoanBot

🎯 Sau bài này, bạn sẽ:

  • Hiểu tại sao single LLM agent mắc systematic biases trên borderline cases
  • Áp dụng "Wisdom of Crowds" principle: independent diverse agents > single expert
  • Phân biệt đúng khi nào debate có ích (high uncertainty) vs gây overhead vô ích (clear cases)
  • Đo lường improvement: accuracy, calibration, và coverage của debate system
💡 Analogy: Hội Đồng Thẩm Định Vay vs Thẩm Định Viên Đơn Lẻ

Ngân hàng truyền thống: khoản vay >2 tỷ VNĐ phải qua hội đồng gồm Credit Officer, Risk Manager, và Compliance Officer — mỗi người nhìn từ góc độ khác nhau. Không phải vì một người không đủ giỏi, mà vì systematic blind spots: Credit Officer có thể quá optimistic về growth potential; Risk Manager có thể quá conservative; chỉ khi ba góc nhìn va chạm mới lộ ra những điểm mù. LoanBot panel hoạt động theo cùng nguyên tắc — không phải democratic voting mà là structured deliberation.

🏢 Tình huống: FC-2024-004 Borderline Case

Hồ sơ: Score 645, DTI 0.42, Income 28M/tháng, Employment 8 tháng, Loan 400M — Mua xe ô tô kinh doanh.

🟢 OptimistAgent

"Employment ngắn nhưng income tốt. Mục đích vay productive (xe kinh doanh). Score 645 trong vùng có thể."

APPROVE
🔴 PessimistAgent

"8 tháng employment quá ngắn — chưa qua probation. DTI 0.42 gần ceiling 0.45. Score 645 là minimum acceptable."

REVIEW
🔵 RegulatorAgent

"Loan purpose hợp lệ. Employment < 12 tháng cần require collateral per TT39/2016 Điều 9 clause 2."

REVIEW + COLLATERAL

Meta-Arbitrator: 2/3 votes là REVIEW (hoặc variant of REVIEW) → decision = REVIEW với note "Yêu cầu bổ sung: hợp đồng kinh doanh + 1 tài sản thế chấp."

Condorcet Jury Theorem
P(panel correct) = 1 − (1−p)^n với p > 0.5, n = số agents
Nếu mỗi agent có P(correct) = 0.70 (p>0.5) và 3 agents (n=3): P(majority correct) = 1 − (1−0.70)^3 = 1 − 0.027 = 0.973. Với 5 agents: P = 0.993. Điều kiện: agents phải INDEPENDENT và mỗi agent phải có p>0.5.
Python (Debate Trigger Logic — Chỉ debate khi cần) 
from dataclasses import dataclass
from typing import Tuple

@dataclass
class DebateRoutingDecision:
    use_debate: bool
    reason: str
    n_agents: int  # 0 for no debate, 3 or 5 for panels

def should_use_debate(
    credit_score: int,
    dti_ratio: float,
    loan_amount_vnd: int,
    employment_months: int
) -> DebateRoutingDecision:
    """Route to debate panel only for genuinely ambiguous cases."""

    # Clear REJECT — no debate needed
    if credit_score < 500 or dti_ratio > 0.55:
        return DebateRoutingDecision(False, "Clear reject criteria", 0)

    # Clear APPROVE — no debate needed
    if credit_score >= 720 and dti_ratio <= 0.35 and employment_months >= 24:
        return DebateRoutingDecision(False, "Clear approve criteria", 0)

    # High-value borderline → full 5-agent panel
    if loan_amount_vnd > 2_000_000_000 and 580 <= credit_score <= 700:
        return DebateRoutingDecision(True, "High-value borderline — full panel", 5)

    # Standard borderline → 3-agent panel
    if 580 <= credit_score <= 680 or (0.38 <= dti_ratio <= 0.45):
        return DebateRoutingDecision(True, "Standard borderline zone", 3)

    # Short employment flag → 3-agent with RegulatorAgent emphasis
    if employment_months < 12:
        return DebateRoutingDecision(True, "Short employment requires regulatory check", 3)

    return DebateRoutingDecision(False, "Sufficient signal for direct decision", 0)

# Test routing decisions
for case in [
    (720, 0.30, 500_000_000, 36),   # Clear approve
    (645, 0.42, 400_000_000, 8),    # Borderline → 3 agents
    (620, 0.41, 2_500_000_000, 18),  # High-value borderline → 5 agents
    (420, 0.65, 200_000_000, 24),   # Clear reject
]:
    r = should_use_debate(*case)
    print(f"Score {case[0]}, DTI {case[1]}: debate={r.use_debate}, agents={r.n_agents} ({r.reason})")

Bài 2: Consensus Mechanisms — Từ Majority Vote đến Weighted Delphi

🎯 Sau bài này, bạn sẽ:

  • Implement 4 consensus mechanisms: majority vote, weighted vote, Borda count, Delphi
  • Biết khi nào simple majority đủ vs khi cần weighted/Delphi approach
  • Handle ties và deadlocks trong agent panel
  • Đo diversity: nếu tất cả agents đồng ý ngay → diversity thấp, debate vô ích
📌 Khái niệm: 4 Consensus Mechanisms

1. Majority Vote: quyết định nào có nhiều phiếu nhất. Simple, fast, fair. Vấn đề: không phân biệt confidence. 2. Weighted Vote: agent có track record tốt hơn có weight cao hơn — RegulatorAgent weight 1.5 cho compliance queries. 3. Borda Count: mỗi agent rank tất cả options (APPROVE=2, REVIEW=1, REJECT=0), sum ranks → option có tổng cao nhất win. Tốt hơn binary vote. 4. Delphi Method: agents debate nhiều rounds — sau mỗi round, xem lại positions của nhau và có thể thay đổi → tốt nhất cho complex cases nhưng latency cao (3-5 rounds × API calls).

Hình 1: 4 consensus mechanisms áp dụng cho FC-2024-004. Majority Vote (2/3 REVIEW), Weighted Vote (2.7 vs 1.0), Borda Count (5 vs 3), Delphi Round 2 (Optimist revised → REVIEW). Tất cả 4 mechanisms đồng thuận REVIEW → confidence cao.
🧠 Kiểm tra nhanh

3 agents vote: OptimistAgent APPROVE, PessimistAgent REJECT, RegulatorAgent APPROVE. Majority vote = APPROVE (2/3). Nhưng RegulatorAgent raise vấn đề: "loan purpose chưa rõ ràng — cần verify theo TT39". Kết quả majority (APPROVE) có nên deliver không?

Gợi ý: KHÔNG deliver ngay dù majority là APPROVE. Lý do: RegulatorAgent raise "compliance flag" — trong domain tín dụng, regulatory concerns luôn có veto priority. Đây là "weighted veto": một agent với regulatory mandate có quyền block decision cho đến khi concern được address. Implementation: check nếu RegulatorAgent flag compliance issue → escalate to REVIEW regardless of majority. Deliver APPROVE chỉ khi RegulatorAgent confirm compliant. Điều này khác với simple majority vote — trong high-stakes domains, không phải tất cả votes đều equal.

Bài 3: Devil's Advocate Pattern — Tranh Luận Có Cấu Trúc

🎯 Sau bài này, bạn sẽ:

  • Implement Devil's Advocate Agent: chủ động tìm lý do để phản đối quyết định draft
  • Thiết kế structured debate protocol: claim → challenge → rebuttal → synthesis
  • Biết khi nào nên dừng debate: convergence criteria hoặc max rounds
  • Tránh "sycophancy collapse": không để agents đồng ý chỉ vì peer pressure
⚠️ Anti-Pattern: Echo Chamber Debate

Nếu tất cả agents được prompt với cùng system prompt và cùng few-shot examples → chúng sẽ tend to agree. Debate trở thành theater, không add value. Prevention: (1) Diversify agent personas với different risk tolerances. (2) Prohibit agents from seeing each other's votes before stating own position. (3) Require each agent to find at least 1 counterargument. (4) Devil's Advocate agent được assigned explicitly để disagree — bất kể position của chính nó là gì.

Python (Devil's Advocate Agent — Structured Challenge) 
import anthropic
from dataclasses import dataclass
from typing import List

@dataclass
class DebateRound:
    round_num: int
    draft_decision: str
    devil_challenge: str
    rebuttals: List[str]
    revised_decision: str
    decision_changed: bool

async def devil_advocate_round(
    client: anthropic.Anthropic,
    application: dict,
    draft_decision: str,
    draft_reasoning: str,
    round_num: int = 1
) -> DebateRound:
    """Devil's Advocate challenges the draft decision to stress-test it."""

    # Step 1: Devil's Advocate generates challenges
    da_prompt = f"""Bạn là Devil's Advocate cho quyết định tín dụng.
Draft decision: {draft_decision}
Reasoning: {draft_reasoning}
Application: Score={application['credit_score']}, DTI={application['dti_ratio']}, Emp={application['employment_months']}mo

Nhiệm vụ: Tìm 2-3 lý do MẠNH NHẤT để phản đối quyết định này.
Không đồng ý vì礼貌 — chỉ đưa ra challenges có cơ sở thực sự.
Format: CHALLENGE_1: ... | CHALLENGE_2: ... | CHALLENGE_3: ..."""

    challenge_resp = client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=400,
        messages=[{"role": "user", "content": da_prompt}]
    )
    challenge = challenge_resp.content[0].text

    # Step 2: Original agent rebuts the challenges
    rebuttal_prompt = f"""Bạn đã propose decision: {draft_decision}.
Devil's Advocate challenges: {challenge}

Rebuttal: Giải thích tại sao decision vẫn đúng (hoặc revise nếu challenge có giá trị).
Format: REBUTTAL: ... | REVISED_DECISION: {draft_decision} (hoặc APPROVE/REVIEW/REJECT nếu thay đổi)"""

    rebuttal_resp = client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=300,
        messages=[{"role": "user", "content": rebuttal_prompt}]
    )
    rebuttal_text = rebuttal_resp.content[0].text

    # Extract revised decision from rebuttal
    revised = draft_decision
    for decision in ["APPROVE", "REVIEW", "REJECT"]:
        if f"REVISED_DECISION: {decision}" in rebuttal_text:
            revised = decision
            break

    return DebateRound(
        round_num=round_num,
        draft_decision=draft_decision,
        devil_challenge=challenge,
        rebuttals=[rebuttal_text],
        revised_decision=revised,
        decision_changed=(revised != draft_decision)
    )

# Convergence check
def debate_converged(rounds: List[DebateRound]) -> bool:
    """Stop debate if: no change in last 2 rounds, or max 3 rounds reached."""
    if len(rounds) >= 3: return True
    if len(rounds) >= 2 and not rounds[-1].decision_changed and not rounds[-2].decision_changed:
        return True  # Stable for 2 consecutive rounds
    return False

Bài 4: Agent Specialization — Persona Design cho Loan Panel

🎯 Sau bài này, bạn sẽ:

  • Thiết kế 5 specialized agent personas cho FinTech Corp loan panel
  • Hiểu cách system prompt định hình risk tolerance và decision bias của agent
  • Calibrate agents để họ có diversity nhưng không quá extreme
  • Track agent track record: ai đúng nhiều nhất trên từng loại case?
📌 Khái niệm: Persona-Based Specialization

Mỗi agent trong panel được assign một persona system prompt định hình world-view của nó. OptimistAgent: "Ưu tiên phê duyệt hồ sơ có tiềm năng tăng trưởng. False negative (từ chối người đáng vay) costly hơn false positive." PessimistAgent: "Ưu tiên bảo vệ vốn của ngân hàng. False positive (approve người không trả được) costly hơn false negative." RegulatorAgent: "Chỉ approve nếu 100% compliant với TT39/2016 và internal policy. Không có exception." Diversity của personas → diversity của reasoning paths → better collective accuracy.

Hình 2: 5-agent loan panel với risk tolerance spectrum. RegulatorAgent có weight cao nhất (1.5) và compliance veto power. 3-agent panel cho standard borderline; 5-agent panel cho high-value (>2B VNĐ) hoặc complex cases.

Bài 5: Arbitration & Meta-Agent — Người Phân Xử Cuối Cùng

🎯 Sau bài này, bạn sẽ:

  • Thiết kế Meta-Arbitrator: tổng hợp debate transcript → final decision có explanation
  • Handle deadlock: khi panel split 2-2 hoặc 3-2 với strong minority view
  • Implement audit trail cho toàn bộ debate — đáp ứng regulatory audit requirements
  • Measure và improve panel quality: track per-agent accuracy, diversity, và calibration
🔑 Key Insight: Meta-Arbitrator ≠ Aggregator

Meta-Arbitrator không chỉ đếm phiếu — nó đọc reasoning của mỗi agent và đưa ra independent judgment về chất lượng của các arguments. Ví dụ: Optimist có 2 điểm mạnh nhưng ignore 1 red flag quan trọng; Pessimist có 1 điểm mạnh nhưng reasoning circle. Meta-Arbitrator nhận ra Pessimist's core argument là stronger dù Optimist có nhiều points hơn. Điều này tương tự như judge trong tòa án — không đếm số nhân chứng mà đánh giá credibility và relevance của evidence.

Python (Meta-Arbitrator — Synthesis + Audit Trail) 
from dataclasses import dataclass, field
from typing import List, Dict
import anthropic

@dataclass
class AgentVote:
    agent_name: str
    vote: str          # APPROVE / REVIEW / REJECT
    confidence: float
    reasoning: str
    weight: float = 1.0
    compliance_flag: bool = False  # regulatory veto flag

@dataclass
class PanelDecision:
    final_decision: str
    consensus_method: str
    confidence: float
    explanation: str
    conditions: List[str]           # e.g., ["Require collateral", "Re-verify income"]
    dissent_summary: str          # minority position summary for audit
    audit_trail: List[dict]       # full debate log

async def meta_arbitrate(
    client: anthropic.Anthropic,
    application: dict,
    votes: List[AgentVote]
) -> PanelDecision:
    """Meta-Arbitrator synthesizes panel votes into final decision."""

    # Check regulatory veto first
    compliance_flags = [v for v in votes if v.compliance_flag]
    if compliance_flags:
        return PanelDecision(
            final_decision="REVIEW",
            consensus_method="regulatory_veto",
            confidence=0.99,
            explanation=f"Compliance flag raised by {compliance_flags[0].agent_name}: {compliance_flags[0].reasoning}",
            conditions=["Resolve compliance flag before proceeding"],
            dissent_summary="N/A — regulatory veto overrides panel vote",
            audit_trail=[{"agent": v.agent_name, "vote": v.vote, "reasoning": v.reasoning} for v in votes]
        )

    # Weighted vote tallying
    vote_weights: Dict[str, float] = {}
    for v in votes:
        vote_weights[v.vote] = vote_weights.get(v.vote, 0) + v.weight * v.confidence

    majority = max(vote_weights, key=vote_weights.get)
    total_weight = sum(vote_weights.values())
    panel_confidence = vote_weights[majority] / total_weight

    # Build synthesis prompt for explanation
    vote_summary = "\n".join([f"{v.agent_name} ({v.weight}w): {v.vote} — {v.reasoning[:100]}" for v in votes])
    synthesis_resp = client.messages.create(
        model="claude-haiku-4-5-20251001", max_tokens=400,
        messages=[{"role": "user", "content":
            f"Panel voted {majority}. Synthesize in 1-2 sentences why, given:\n{vote_summary}\nAlso list conditions if REVIEW."}]
    )

    dissent = [v for v in votes if v.vote != majority]
    return PanelDecision(
        final_decision=majority,
        consensus_method="weighted_vote",
        confidence=round(panel_confidence, 3),
        explanation=synthesis_resp.content[0].text,
        conditions=["Verify income source", "Require 1 collateral asset"] if majority == "REVIEW" else [],
        dissent_summary=", ".join([f"{d.agent_name}: {d.vote}" for d in dissent]),
        audit_trail=[{"agent": v.agent_name, "vote": v.vote, "weight": v.weight, "conf": v.confidence} for v in votes]
    )

Sau khi deploy panel, track metrics để improve:

  • Per-agent accuracy: mỗi tuần, so sánh agent's vote với ground truth (30-day loan outcome). Agent nào consistently sai → giảm weight hoặc update persona prompt.
  • Diversity index: entropy của vote distribution. Nếu 5 agents consistently agree 5/5 → panel quá homogeneous → add adversarial agent hoặc diversify personas.
  • Debate utilization: % cases trigger debate (target: 15-20% — chỉ genuine borderlines). Nếu >30% → routing quá sensitive; <10% → missing edge cases.
  • Reversal rate: % cases where meta-arbitrator overrules majority. Nếu >20% → majority vote is poorly calibrated hoặc arbitrator bias cần investigate.

Multi-Agent DebateNhiều LLM agents với diverse personas cùng analyze một case rồi reach consensus — better accuracy trên ambiguous cases.

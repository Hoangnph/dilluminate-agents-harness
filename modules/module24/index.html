Bài 1: Knowledge Boundaries — LoanBot Biết Gì và Không Biết Gì?

🎯 Sau bài này, bạn sẽ:

  • Phân loại 4 loại knowledge: Known Known, Known Unknown, Unknown Known, Unknown Unknown
  • Xác định knowledge boundary của LoanBot trong domain tín dụng Việt Nam
  • Implement "I don't know" gracefully — không hallucinate, không silent fail
  • Thiết kế knowledge boundary tests để phát hiện khi LoanBot vượt boundary
💡 Analogy: Johari Window cho LoanBot

Johari Window (1955) phân chia knowledge thành 4 ô: Known Known (biết và biết mình biết) — TT39/2016 credit score thresholds, DTI limits; Known Unknown (biết mình không biết) — lãi suất thị trường hôm nay, credit score khách hàng cụ thể chưa query; Unknown Known (không biết rằng mình biết, bias) — implicit patterns trong training data; Unknown Unknown (không biết mình không biết) — các regulation mới sau training cutoff. Nguồn nguy hiểm nhất: Unknown Known (bias âm thầm) và Unknown Unknown (hallucinate với full confidence).

Epistemic Humility Rule
If P(correct) < threshold → "Tôi không chắc chắn về điều này"
LoanBot phải biết nói "Tôi không có đủ thông tin để xác nhận" thay vì fabricate. Threshold cho loan decisions: P(correct) ≥ 0.85 mới confirm; 0.60–0.85 → flag as uncertain; <0.60 → escalate hoặc query external source.
🏢 Tình huống: LoanBot vượt Knowledge Boundary

Analyst hỏi LoanBot: "Theo quy định hiện hành, mức lãi suất cho vay tối đa là bao nhiêu?"

❌ Hallucination (NGUY HIỂM): "Theo Thông tư 39/2016/TT-NHNN Điều 17, lãi suất cho vay tối đa là 20%/năm đối với các khoản vay ngắn hạn bằng VNĐ."
→ Điều 17 không tồn tại trong TT39. Số liệu 20%/năm là fabricated.
✅ Epistemic Humility (ĐÚNG): "Đây là câu hỏi về regulatory threshold cụ thể mà tôi không thể xác nhận chính xác từ training data — regulation thay đổi thường xuyên và tôi có thể không có bản mới nhất. Tôi khuyến nghị kiểm tra trực tiếp tại Cổng thông tin NHNN (sbv.gov.vn) hoặc liên hệ phòng pháp chế để có con số chính xác và cập nhật nhất."
Hình 3: Johari Window cho LoanBot. Known Known (xanh): trả lời trực tiếp. Known Unknown (cam): từ chối và redirect. Unknown Known (tím): bias ẩn — cần fairness audit. Unknown Unknown (đỏ): nguy hiểm nhất — hallucinate với confidence cao về regulation sau training cutoff. RAG + citation verifier là biện pháp chính.
Python (Knowledge Boundary Classifier) 
from dataclasses import dataclass
from typing import List, Tuple
from enum import Enum

class KnowledgeType(Enum):
    KNOWN_KNOWN   = "known_known"    # high confidence, in training domain
    KNOWN_UNKNOWN = "known_unknown"  # explicitly outside domain
    UNCERTAIN     = "uncertain"      # borderline — needs verification

@dataclass
class KnowledgeBoundaryResult:
    knowledge_type: KnowledgeType
    confidence: float
    response_strategy: str
    suggested_source: str = ""

# Domains LoanBot confidently knows
KNOWN_DOMAINS = {
    "credit_score_thresholds": ("TT39/2016 thresholds: min 500, good 650+", 0.95),
    "dti_limits": ("DTI max 0.45 for standard loans per FinTech Corp policy", 0.92),
    "loan_decision_process": ("5-step underwriting process", 0.98),
    "blacklist_check": ("Always first step — CIC + internal blacklist", 0.99),
}

# Topics outside LoanBot's reliable knowledge
OUT_OF_DOMAIN_SIGNALS = [
    "lãi suất hôm nay", "giá vàng", "tỷ giá",
    "thông tư mới nhất", "circular mới", "quy định 2025",
    "thị trường bất động sản", "lãi suất tối đa",
]

def classify_knowledge_boundary(query: str) -> KnowledgeBoundaryResult:
    """Classify query against LoanBot's known knowledge domains."""
    query_lower = query.lower()

    # Check out-of-domain signals
    for signal in OUT_OF_DOMAIN_SIGNALS:
        if signal in query_lower:
            return KnowledgeBoundaryResult(
                knowledge_type=KnowledgeType.KNOWN_UNKNOWN,
                confidence=0.15,
                response_strategy="decline_and_redirect",
                suggested_source="sbv.gov.vn hoặc phòng pháp chế"
            )

    # Check known domains
    for domain, (_, conf) in KNOWN_DOMAINS.items():
        if any(kw in query_lower for kw in domain.split("_")):
            return KnowledgeBoundaryResult(
                knowledge_type=KnowledgeType.KNOWN_KNOWN,
                confidence=conf,
                response_strategy="answer_directly"
            )

    # Default: uncertain — needs verification
    return KnowledgeBoundaryResult(
        knowledge_type=KnowledgeType.UNCERTAIN,
        confidence=0.55,
        response_strategy="answer_with_caveat_and_suggest_verification",
        suggested_source="Đề nghị xác nhận với phòng pháp chế"
    )

Bài 2: Hallucination Detection — Phát Hiện và Ngăn Ngừa

🎯 Sau bài này, bạn sẽ:

  • Phân biệt 3 loại hallucination: factual, citation, numeric
  • Implement citation verifier: kiểm tra regulation references tồn tại thực sự
  • Dùng self-consistency sampling để detect inconsistent outputs
  • Thiết kế hallucination test suite cho LoanBot
📌 Khái niệm: 3 Loại Hallucination trong LoanBot

1. Factual Hallucination: model claim một fact không đúng — "TT39/2016 có hiệu lực từ 2014" (thực tế: 2016). 2. Citation Hallucination: cite điều khoản không tồn tại — "Điều 17 NĐ13/2023 quy định..." (NĐ13 không có Điều 17 liên quan). 3. Numeric Hallucination: tạo ra con số — "lãi suất tối đa 2.5%/tháng" từ không khí. Numeric hallucination nguy hiểm nhất cho LoanBot vì ảnh hưởng trực tiếp đến credit decisions và compliance.

Hình 1: 3-layer hallucination detection. Layer 1 kiểm tra citation tồn tại trong regulation database. Layer 2 kiểm tra numeric values trong valid range. Layer 3: self-consistency sampling 3 lần — agreement <2/3 → flag. Tất cả pass → deliver; bất kỳ fail → quarantine + human review.
Python (Citation Verifier + Numeric Bounds Checker) 
import re
from typing import List, Tuple

# Known valid regulations and their structure
VALID_REGULATIONS = {
    "TT39/2016": {"articles": range(1, 28), "full_name": "Thông tư 39/2016/TT-NHNN"},
    "NĐ13/2023":  {"articles": range(1, 52), "full_name": "Nghị định 13/2023/NĐ-CP"},
    "TT16/2021":  {"articles": range(1, 23), "full_name": "Thông tư 16/2021/TT-NHNN"},
}

# Valid numeric ranges for LoanBot outputs
NUMERIC_BOUNDS = {
    "confidence":      (0.0,  1.0),
    "credit_score":    (300,  900),
    "dti_ratio":       (0.0,  1.0),
    "interest_rate":   (0.01, 0.36),   # 1%–36% annual
    "loan_amount_vnd": (10_000_000, 10_000_000_000),
}

def verify_citations(regulation_refs: List[str]) -> List[Tuple[str, bool, str]]:
    """Check each regulation ref exists and article number is valid."""
    results = []
    for ref in regulation_refs:
        # Parse: "TT39/2016 Điều 9" or "NĐ13/2023 Art.15"
        match = re.search(r'(TT\d+/\d{4}|NĐ\d+/\d{4}|TT\d+/\d{4})', ref)
        article_match = re.search(r'(?:Điều|Art\.?)\s*(\d+)', ref, re.IGNORECASE)

        if not match:
            results.append((ref, False, "Cannot parse regulation identifier"))
            continue

        reg_id = match.group(1)
        if reg_id not in VALID_REGULATIONS:
            results.append((ref, False, f"Unknown regulation: {reg_id}"))
            continue

        reg = VALID_REGULATIONS[reg_id]
        if article_match:
            article_num = int(article_match.group(1))
            if article_num not in reg["articles"]:
                results.append((ref, False, f"Article {article_num} out of range for {reg_id} (max: {max(reg['articles'])})"))
                continue

        results.append((ref, True, f"Valid: {reg['full_name']}"))
    return results

def check_numeric_bounds(output: dict) -> List[Tuple[str, bool, str]]:
    """Verify numeric fields are in expected ranges."""
    results = []
    for field, (low, high) in NUMERIC_BOUNDS.items():
        if field not in output: continue
        val = output[field]
        ok = low <= val <= high
        msg = f"OK: {val} in [{low}, {high}]" if ok else f"OUT OF RANGE: {val} not in [{low}, {high}]"
        results.append((field, ok, msg))
    return results

# Test with a hallucinated output
hallucinated_output = {
    "decision": "APPROVE",
    "confidence": 0.94,
    "regulation_refs": ["TT39/2016 Điều 9", "NĐ13/2023 Điều 65"],  # Điều 65 doesn't exist!
    "interest_rate": 0.025,  # monthly — multiply by 12 = 30% annual, within range
}
citation_results = verify_citations(hallucinated_output["regulation_refs"])
for ref, ok, msg in citation_results:
    print(f"{'✅' if ok else '❌'} {ref}: {msg}")
🧠 Kiểm tra nhanh

Self-consistency sampling: LoanBot được hỏi 3 lần cùng câu hỏi về borderline case (Score 638, DTI 0.41). Lần 1: APPROVE. Lần 2: REVIEW. Lần 3: APPROVE. Agreement = 2/3. Có nên flag không? Và nếu agreement = 1/3 thì sao?

Gợi ý: 2/3 agreement (APPROVE x2, REVIEW x1) = borderline. Nên cân nhắc context: cho loan decision (high stakes) → threshold tốt hơn là 3/3 hoặc ít nhất 2/3 nhưng flag với note "low confidence — 1 sample disagreed". Quyết định: deliver APPROVE (majority) nhưng add warning "confidence degraded due to sampling inconsistency." Với 1/3 agreement (APPROVE, REVIEW, REJECT mỗi loại 1): đây là major red flag — model hoàn toàn inconsistent → KHÔNG deliver → escalate to human review. Rule: agreement <2/3 → always escalate cho high-stakes decisions như loan approval.

Bài 3: Uncertainty Quantification — Đo Lường Sự Không Chắc Chắn

🎯 Sau bài này, bạn sẽ:

  • Phân biệt aleatoric uncertainty (data noise) vs epistemic uncertainty (model ignorance)
  • Implement Monte Carlo sampling để estimate confidence distribution
  • Calibrate confidence scores: ECE (Expected Calibration Error) cho LoanBot
  • Dùng verbalized uncertainty: "Tôi khoảng 80% chắc rằng..." thay vì false precision
📌 Khái niệm: Aleatoric vs Epistemic Uncertainty

Aleatoric uncertainty: uncertainty vốn có trong data, không thể reduce — KH có income biến động theo mùa vụ, không thể predict chính xác. Epistemic uncertainty: uncertainty do thiếu knowledge — LoanBot không biết regulation mới ban hành 2 tháng trước training cutoff. Quan trọng: epistemic uncertainty có thể reduce bằng cách thu thập thêm data hoặc query external source; aleatoric thì không. Với LoanBot: borderline cases có high aleatoric uncertainty (genuinely ambiguous); câu hỏi về regulation mới có high epistemic uncertainty.

Hình 4: Aleatoric vs Epistemic Uncertainty. Aleatoric (cam): borderline case genuinely ambiguous — Monte Carlo samples inconsistent dù có đủ context. Cannot reduce — accept và verbalize uncertainty. Epistemic (tím): thiếu knowledge về regulation mới — có thể fix bằng RAG update → confidence tăng về ≥0.90.
Python (Monte Carlo Confidence Estimation) 
import statistics
from typing import List, Dict

def monte_carlo_confidence(
    model_fn,           # callable: application → decision dict
    application: dict,
    n_samples: int = 10,
    temperature_variants: List[float] = None
) -> Dict:
    """Estimate true confidence via multiple samples with slight input perturbation."""
    if temperature_variants is None:
        temperature_variants = [0.3, 0.5, 0.7]  # vary model temperature

    decisions = []
    confidences = []

    for i in range(n_samples):
        # Slight perturbation to test robustness (not PII-sensitive fields)
        perturbed = dict(application)
        perturbed["_sample_idx"] = i  # force different seed

        result = model_fn(perturbed)
        decisions.append(result["decision"])
        confidences.append(result.get("confidence", 0.8))

    # Compute agreement and uncertainty
    decision_counts = {}
    for d in decisions:
        decision_counts[d] = decision_counts.get(d, 0) + 1

    majority = max(decision_counts, key=decision_counts.get)
    agreement_rate = decision_counts[majority] / n_samples
    mean_conf = statistics.mean(confidences)
    std_conf  = statistics.stdev(confidences) if len(confidences) > 1 else 0.0

    # Calibrated confidence: combination of agreement and stated confidence
    calibrated_conf = agreement_rate * 0.6 + mean_conf * 0.4

    return {
        "decision": majority,
        "calibrated_confidence": round(calibrated_conf, 3),
        "raw_confidence_mean": round(mean_conf, 3),
        "confidence_std": round(std_conf, 3),
        "agreement_rate": round(agreement_rate, 3),
        "decision_distribution": decision_counts,
        "uncertainty_type": "high_epistemic" if agreement_rate < 0.67 else "low",
        "should_escalate": agreement_rate < 0.67 or calibrated_conf < 0.75,
    }

# Verbalized uncertainty helper
def verbalize_uncertainty(calibrated_conf: float) -> str:
    if calibrated_conf >= 0.92:   return "Tôi rất chắc chắn rằng"
    elif calibrated_conf >= 0.80: return "Tôi khá tự tin rằng"
    elif calibrated_conf >= 0.65: return "Tôi cho rằng (nhưng cần xác nhận thêm)"
    else:                          return "Tôi không chắc chắn — khuyến nghị human review"

Bài 4: RAG-Augmented Reasoning — Grounding LoanBot vào Thực Tế

🎯 Sau bài này, bạn sẽ:

  • Hiểu RAG giải quyết hallucination và knowledge staleness như thế nào
  • Thiết kế regulation knowledge base với chunking và embedding cho LoanBot
  • Implement grounded generation: model chỉ được cite những gì retrieved
  • Evaluate faithfulness: generated text có trung thành với retrieved context không?
💡 Analogy: Mở Sách vs Nhớ Thuộc Lòng

LoanBot không dùng RAG = sinh viên luật trả lời từ trí nhớ — đúng 80% nhưng 20% còn lại có thể sai lệch hoặc nhầm lẫn điều khoản. LoanBot với RAG = sinh viên luật được phép mở sách — retrieve đúng đoạn văn bản → cite chính xác. Ưu điểm: faithful citations, không hallucinate điều khoản. Nhược điểm: phải có knowledge base đầy đủ và cập nhật; retrieval quality quyết định answer quality.

Hình 2: RAG pipeline cho LoanBot. Regulation knowledge base (~611 chunks từ 3 văn bản pháp lý) được embed và index. Query → retrieve top-k relevant chunks → LoanBot chỉ được cite những gì nằm trong retrieved context → faithfulness checker verify tất cả claims có source.
⚠️ Faithfulness vs Relevance — Hai Metrics Khác Nhau

Faithfulness: generated text có trung thành với retrieved context không? (0-1). Relevance: retrieved context có relevant với query không? (0-1). LoanBot có thể: (1) High faithfulness + Low relevance: faithful với sai context; (2) Low faithfulness + High relevance: có context đúng nhưng hallucinate ngoài context. Cần evaluate cả hai. RAGAS framework cung cấp cả hai metrics. Với LoanBot: faithfulness threshold ≥ 0.90; relevance threshold ≥ 0.80.

Bài 5: Knowledge Freshness & Drift — Khi Thế Giới Thay Đổi

🎯 Sau bài này, bạn sẽ:

  • Hiểu knowledge cutoff problem và temporal reasoning limitations
  • Implement knowledge staleness detector: so sánh training knowledge vs current reality
  • Thiết kế continuous knowledge update pipeline cho LoanBot
  • Tạo temporal hedging: LoanBot tự động add caveat khi cite time-sensitive info
🏢 Tình huống: Knowledge Drift tại FinTech Corp

LoanBot được training với data đến tháng 3/2024. Đến tháng 9/2024:

  • NHNN ban hành Thông tư 12/2024: sửa đổi DTI threshold từ 0.45 → 0.40 với khoản vay >500M
  • FinTech Corp cập nhật internal policy: min employment từ 6 tháng → 12 tháng
  • Lãi suất cơ bản thay đổi: từ 4.5% → 3.75%

Nguy cơ: LoanBot không biết những thay đổi này → approve cases theo rule cũ → compliance violation. Solution: temporal hedge + RAG update + staleness detection.

Hình 5: Knowledge Drift Timeline. LoanBot training cutoff T3/2024 — trong 6 tháng tiếp theo: TT12/2024 đổi DTI threshold, FinTech policy đổi employment minimum. LoanBot không biết → compliance risk. Staleness detector alert sau 45+ ngày → trigger RAG update → knowledge fresh trong vài giờ, không cần retrain model.
Python (Temporal Hedge + Knowledge Freshness Monitor) 
from datetime import datetime, date
from typing import Optional

@dataclass
class KnowledgeSource:
    name: str
    last_updated: date
    update_frequency_days: int   # how often this source changes
    staleness_threshold_days: int  # when to flag as potentially stale
    source_url: str

# Knowledge sources with their freshness profiles
KNOWLEDGE_SOURCES = [
    KnowledgeSource("TT39/2016",          date(2024, 1, 15), update_frequency_days=365,  staleness_threshold_days=90,  source_url="sbv.gov.vn"),
    KnowledgeSource("NĐ13/2023",          date(2024, 1, 15), update_frequency_days=365,  staleness_threshold_days=90,  source_url="vanban.chinhphu.vn"),
    KnowledgeSource("FinTech Internal Policy", date(2024, 3, 1),  update_frequency_days=30,   staleness_threshold_days=45,  source_url="internal"),
    KnowledgeSource("NHNN Interest Rate",    date(2024, 2, 1),  update_frequency_days=7,    staleness_threshold_days=14,  source_url="sbv.gov.vn/rate"),
]

def check_knowledge_freshness(as_of: date = None) -> List[dict]:
    """Check which knowledge sources may be stale."""
    today = as_of or date.today()
    stale = []
    for src in KNOWLEDGE_SOURCES:
        days_since_update = (today - src.last_updated).days
        is_stale = days_since_update > src.staleness_threshold_days
        stale.append({
            "source": src.name, "days_since_update": days_since_update,
            "stale": is_stale, "source_url": src.source_url,
            "severity": "HIGH" if days_since_update > src.staleness_threshold_days * 2 else "MEDIUM" if is_stale else "OK",
        })
    return [s for s in stale if s["stale"]]

def temporal_hedge(response: dict, topic: str) -> dict:
    """Add temporal caveat if topic involves time-sensitive information."""
    TIME_SENSITIVE_TOPICS = ["lãi suất", "interest rate", "policy threshold", "thông tư mới"]
    if any(t in topic.lower() for t in TIME_SENSITIVE_TOPICS):
        response["temporal_caveat"] = (
            f"⚠️ Thông tin này dựa trên knowledge cutoff của model. "
            f"Với thông tin về {topic}, khuyến nghị xác nhận tại nguồn chính thức."
        )
    return response

FinTech Corp implement continuous knowledge update pipeline:

  1. Regulatory monitor: scrape sbv.gov.vn + vanban.chinhphu.vn daily → detect new circulars/decrees → alert legal team
  2. Human validation: legal team review và approve changes → không auto-update without human approval
  3. RAG knowledge base update: chunk new regulation → embed → update vector store → invalidate stale cached results
  4. Regression test: run full test suite sau mỗi knowledge update → verify không break existing decisions
  5. Version control: mỗi knowledge base version có timestamp → audit trail đầy đủ

Không fine-tune lại LoanBot mỗi khi regulation thay đổi — RAG update nhanh hơn nhiều (giờ vs tuần) và không cần GPU compute.

Bài 1

Các Pattern Kiến Trúc Multi-Agent

🎯 Sau bài này, bạn sẽ:

  • Phân biệt được 4 patterns chính: Supervisor, Pipeline, Debate, Peer-to-Peer
  • Giải thích được tại sao LoanBot v2.0 chọn Supervisor Pattern và lợi ích của parallel execution
  • Biết trade-offs của multi-agent: throughput tăng nhưng debugging phức tạp hơn
🏢 Analogy: Phòng ban trong công ty

Một công ty nhỏ có một nhân viên làm tất cả (kế toán + sales + support). Khi lớn lên, cần chuyên môn hóa: phòng kế toán, phòng sales, phòng kỹ thuật — và một giám đốc điều phối. Multi-agent AI làm điều tương tự: thay vì một LLM biết tất cả, ta có nhiều agent chuyên biệt và một Supervisor orchestrate workflow.

4 Pattern Multi-Agent Chính

🎯 Supervisor Pattern
Một Supervisor agent nhận task, decompose thành subtasks, dispatch cho specialized agents, và aggregate results.
Best for: Complex tasks cần nhiều expertise khác nhau — LoanBot v2.0!
🔗 Pipeline Pattern
Agents xử lý tuần tự theo dây chuyền: A → B → C. Output của agent này là input của agent kế tiếp.
Best for: Workflows có thứ tự cố định (validate → process → report)
🗳️ Debate Pattern
Nhiều agents cho ý kiến độc lập về cùng một vấn đề, sau đó vote hoặc synthesize để ra quyết định cuối.
Best for: High-stakes decisions cần cross-validation (borderline loan cases)

Supervisor Pattern: Cách Hoạt Động

Hình 1: Supervisor Pattern — Supervisor nhận loan_request, dispatch parallel tasks tới CreditAgent + IncomeAgent + RiskAgent, aggregates results, trả final_decision. ReportAgent chạy last.
💡 Tại sao cần Multi-Agent thay vì một LoanBot lớn hơn?

Thời gian xử lý — Parallel vs Sequential
T_parallel = max(T_credit, T_income) << T_sequential
CreditAgent (0.3s) + IncomeAgent (0.4s) chạy song song → tổng 0.4s, thay vì 0.7s tuần tự
🧠 Kiểm tra nhanh

LoanBot v2.0 cần xử lý 3 tasks: CreditAgent (0.3s), IncomeAgent (0.4s), RiskAgent (0.2s, phụ thuộc vào 2 agent trên). Tổng thời gian với Supervisor Pattern là bao nhiêu?

Phân tích dependency:
• CreditAgent + IncomeAgent: độc lập → chạy song song (asyncio.gather)
• RiskAgent: phụ thuộc → chỉ chạy sau khi có kết quả từ cả 2

Tổng thời gian: max(0.3s, 0.4s) + 0.2s = 0.6s
Nếu tuần tự: 0.3 + 0.4 + 0.2 = 0.9s

Kết luận: Parallel execution tiết kiệm 33% thời gian. Tại LoanBot production scale (50K hồ sơ/tháng), đây là tiết kiệm đáng kể.

Điểm Chính Bài 1

  • 4 patterns: Supervisor (orchestrate), Pipeline (sequential), Debate (consensus), Peer-to-Peer (event-driven)
  • Supervisor pattern phù hợp nhất với LoanBot: nhiều domain expertise, có thể parallel
  • Multi-agent benefits: throughput, specialization, failure isolation, upgradeability
  • Multi-agent challenges: coordination overhead, debugging harder, consistency across agents
Bài 2

LoanBot v2.0: 5-Agent Architecture

🏦 Analogy: Hội đồng thẩm định tín dụng

Tại ngân hàng lớn, một khoản vay không do một người quyết định. Có nhân viên tín dụng (thu thập hồ sơ), chuyên viên phân tích thu nhập, quản lý rủi ro, và hội đồng phê duyệt. LoanBot v2.0 mirror chính xác cấu trúc này: mỗi agent = một chuyên gia trong hội đồng.

5 Agents của LoanBot v2.0

🎯
Supervisor
Orchestrator — nhận loan request, dispatch subtasks, aggregate results, final decision
Model: claude-sonnet-4-6 | No direct tools
📊
CreditAgent
Credit specialist — kiểm tra CIC score, blacklist status, credit history
Tools: check_credit_score, check_blacklist | Model: claude-haiku-4-5
💰
IncomeAgent
Income analyst — verify_income, calculate_dti, affordability check
Tools: verify_income, calculate_dti | Model: claude-haiku-4-5
⚖️
RiskAgent
Risk manager — aggregate credit + income signals, compute composite risk score
Tools: none (analysis only) | Model: claude-sonnet-4-6
📋
ReportAgent
Report writer — generate explainable decision report, customer-facing explanation
Tools: generate_report | Model: claude-haiku-4-5

LoanBot v2.0 — Data Flow & Parallelism

Hình 2: Parallel Execution Flow — CreditAgent và IncomeAgent chạy đồng thời (asyncio.gather), giảm latency từ 8s xuống 4s. RiskAgent chờ cả hai hoàn thành mới chạy.

Điểm Chính Bài 2

  • 5 agents: Supervisor + CreditAgent + IncomeAgent + RiskAgent + ReportAgent
  • CreditAgent và IncomeAgent chạy PARALLEL → tiết kiệm ~50s so với sequential
  • RiskAgent chạy SAU khi cả hai parallel agents hoàn thành
  • Model tiering: Haiku cho simple agents, Sonnet cho Supervisor và RiskAgent
Bài 3

Agent Communication & Handoff Protocol

📞 Analogy: Cuộc họp bàn giao ca làm việc

Khi ca làm việc chuyển giao, nhân viên không chỉ nói "xong rồi đó". Họ bàn giao structured handoff note: case ID, trạng thái, kết quả, vấn đề còn lại, việc cần làm tiếp theo. Agent handoff cũng vậy — không phải text tự do mà là structured message schema để đảm bảo không mất thông tin.

AgentMessage Schema — Giao Thức Giao Tiếp Chuẩn

{
  "message_id": "MSG-001-CREDIT-20260626",
  "trace_id":   "TRC-LOAN-FC2024001-XYZ",
  "from_agent": "CreditAgent",
  "to_agent":   "Supervisor",
  "message_type": "RESULT",     // TASK | RESULT | ERROR | CLARIFY
  "timestamp":  "2026-06-26T10:15:42Z",
  "payload": {
    "customer_id":    "FC-2024-001",
    "credit_score":   720,
    "credit_grade":   "A",
    "blacklisted":    false,
    "confidence":     0.98,
    "data_source":    "CIC API v3",
    "latency_ms":     1240
  },
  "status":    "SUCCESS",       // SUCCESS | PARTIAL | FAILED
  "next_step": "Proceed with IncomeAgent results"
}

Handoff Protocol: Supervisor Orchestration Sequence

Hình 3: Agent Handoff Sequence Diagram — AgentMessage schema với trace_id xuyên suốt pipeline. Mỗi handoff giữ context, không mất data giữa các agent.
⚡ Parallel Execution: CreditAgent + IncomeAgent chạy đồng thời

Supervisor dispatch cả hai agents cùng lúc (asyncio.gather), KHÔNG chờ CreditAgent xong mới dispatch IncomeAgent. Tiết kiệm thời gian gần như bằng latency của một agent (nếu cả hai mất 60s → parallel = 65s thay vì 120s).

# Python conceptual (asyncio)
credit_task  = asyncio.create_task(credit_agent.run(loan_data))
income_task  = asyncio.create_task(income_agent.run(loan_data))

# Both run in parallel
credit_result, income_result = await asyncio.gather(
    credit_task, income_task
)
# Now risk_agent gets both results
risk_result = await risk_agent.run(credit_result, income_result)

Điểm Chính Bài 3

  • AgentMessage schema: message_id, trace_id, from/to, type, payload, status, next_step
  • message_type: TASK (giao việc), RESULT (trả kết quả), ERROR (báo lỗi), CLARIFY (xin thêm info)
  • Parallel execution qua asyncio.gather — CreditAgent và IncomeAgent chạy đồng thời
  • RiskAgent chạy AFTER cả hai — dependency graph phải rõ ràng
Bài 4

Distributed Tracing trong Multi-Agent System

🔍 Analogy: Theo dõi đơn hàng online

Khi đặt hàng online, bạn có thể track: đơn đã xác nhận → kho đóng gói → đã giao shipper → đang vận chuyển → đã giao. Mỗi bước có timestamp và người phụ trách. Distributed tracing cho multi-agent AI làm điều tương tự: một trace_id duy nhất theo suốt từ user request đến final decision, qua mọi agent.

Trace ID Propagation: Từ Request đến Decision

Thành phầnSpan IDLatencyStatusKey Info
trace_id: TRC-LOAN-FC2024001-ABC123 | Total: 137s
Supervisor.receive SPN-001 2s ✅ Parsed loan request, dispatched 2 parallel tasks
CreditAgent.run SPN-002 58s ✅ score=720, grade=A, not blacklisted | parent: SPN-001
IncomeAgent.run SPN-003 63s ✅ verified=35M, dti=28%, acceptable | parent: SPN-001
RiskAgent.compute SPN-004 42s ✅ risk_score=LOW (0.12), decision=APPROVE | parent: SPN-001
ReportAgent.generate SPN-005 25s ✅ RPT-FC001-2026 generated, PDF uploaded | parent: SPN-001
Hình 4: Distributed Trace — trace_id: abc-123 propagate qua 5 agents. Flamegraph cho thấy CreditAgent (2.1s) là bottleneck chính; cần optimize CIC API call.
📏 Per-Agent Metrics cho Dashboard

Mỗi agent publish metrics với agent_id prefix để có thể filter riêng:

  • loanbot.credit_agent.latency_p95 = 63s — CIC API là bottleneck
  • loanbot.income_agent.latency_p95 = 68s — income verification chậm hơn credit
  • loanbot.risk_agent.faithfulness = 99.5% — critical, phải monitor chặt
  • loanbot.supervisor.total_latency_p95 = 137s — tổng end-to-end
  • loanbot.pipeline.parallel_efficiency = 0.87 — 87% efficiency (ideal=1.0)

Điểm Chính Bài 4

  • trace_id propagates qua mọi agent — mỗi agent thêm span_id của mình
  • Flamegraph visualize parallel execution: bottleneck rõ ràng (IncomeAgent 63s là critical path)
  • Per-agent metrics: prefix agent_id để có thể filter trong Grafana/Prometheus
  • Parallel efficiency = actual_time / sequential_time — LoanBot v2.0 đạt 87% (137s vs 157s sequential)
Bài 5

MCP Protocol: Tool Sharing Between Agents

🔌 Analogy: USB-C là chuẩn kết nối chung

Trước USB-C, mỗi thiết bị có cổng riêng: mini-USB, micro-USB, Lightning... Rất bất tiện. USB-C là chuẩn chung — một cable kết nối được tất cả. MCP (Model Context Protocol) là "USB-C của AI tools": bất kỳ AI agent nào cũng có thể connect và dùng tool từ bất kỳ MCP server nào.

MCP Protocol: Kiến Trúc Server-Client

MCP: 4 Bước Hoạt Động

1
Discovery: Agent hỏi MCP Server có những tool gì
Agent gọi list_tools() → nhận JSON list với name, description, input_schema của mọi tool. Agent dùng list này làm tool_choice trong Claude API call.
2
Authorization: Kiểm tra permission trước khi gọi
MCP Server verify: agent có permission gọi tool này không? CreditAgent có thể gọi check_credit_score nhưng không thể gọi generate_report.
3
Execution: Claude chọn tool, MCP Server execute
LLM call → Claude quyết định dùng tool X với params Y → MCP Client forward đến MCP Server → Server execute thực tế → trả kết quả về Claude.
4
Result: Kết quả trả về agent, log vào audit trail
MCP Server trả kết quả + execution metadata (latency, data_source). Mọi call đều được log với trace_id để audit.
🎯 Lợi ích MCP cho LoanBot v2.0
  • Tool reuse: check_credit_score được define một lần trong CIC MCP Server, CreditAgent và RiskAgent đều dùng được
  • Centralized auth: Permission control tại MCP Server layer — không cần duplicate trong mỗi agent
  • Version management: Upgrade tool implementation trong MCP Server mà không cần redeploy agents
  • Standardized schemas: Mọi agent nhìn thấy cùng tool schema → không có inconsistency
  • Third-party tools: Có thể plug in external MCP Servers (ví dụ: Experian credit bureau) mà không thay đổi agent code

Điểm Chính Bài 5

  • MCP = Model Context Protocol — standard cho AI tools (Anthropic open-sourced)
  • 3 thành phần: MCP Server (provide tools), MCP Client (trong agent), Protocol (JSON-RPC over stdio/HTTP)
  • 4 bước: Discovery → Authorization → Execution → Result + Audit
  • Benefits: tool reuse, centralized auth, version management, third-party integration

📚 Glossary Module 6

Supervisor Pattern
Multi-agent pattern: một Supervisor orchestrate nhiều specialized subagents, aggregate kết quả và ra quyết định cuối. Phù hợp cho complex tasks với nhiều domain expertise.
Parallel Execution
Nhiều agents chạy đồng thời (asyncio.gather) thay vì tuần tự. LoanBot v2.0: CreditAgent + IncomeAgent parallel → tiết kiệm ~53s.
AgentMessage
Structured schema cho giao tiếp agent-to-agent: message_id, trace_id, from/to, type (TASK/RESULT/ERROR), payload, status.
Distributed Tracing
Một trace_id duy nhất theo dõi request qua nhiều agents/services. Mỗi agent thêm span_id. Flamegraph visualize parallel execution.
MCP (Model Context Protocol)
Open protocol (Anthropic) để AI agents discover và sử dụng external tools theo chuẩn. MCP Server provide tools, MCP Client consume.
Span
Một unit của distributed trace — một operation trong một agent (ví dụ: CreditAgent.run). Span có span_id, parent_id, start_time, duration, status.
Parallel Efficiency
Metric đo hiệu quả parallel execution: actual_time / sequential_time. LoanBot v2.0: 137s / 157s = 87%. Ideal = 1.0 (infinite speedup).
Debate Pattern
Multi-agent pattern: nhiều agents đưa ra ý kiến độc lập, sau đó vote hoặc synthesize. Tốt cho borderline decisions cần cross-validation.
Agent Handoff
Quá trình một agent kết thúc task và pass kết quả (structured) cho agent tiếp theo. Cần clear schema để không mất thông tin.
Critical Path
Sequence of dependent operations xác định minimum total time. LoanBot v2.0: IncomeAgent (63s) là critical path vì là parallel agent chậm nhất.

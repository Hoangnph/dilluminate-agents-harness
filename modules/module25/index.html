⚡ Agent Orchestration at Scale

FinTech Corp tăng từ 50K lên 500K hồ sơ/tháng. LoanBot v3.7 đơn lẻ không thể xử lý 10× load trong giờ cao điểm — bị timeout, queue đầy, kết quả không nhất quán. Module này dạy cách thiết kế kiến trúc distributed để LoanBot scale horizontal mà không mất reliability.

⏱ 3 tuần 📖 5 bài giảng 🔬 Distributed Systems

Bài 1: Distributed Agent Architecture — Từ Monolith đến Fleet

🎯 Sau bài này, bạn sẽ:

  • Hiểu vấn đề của monolithic LoanBot khi scale từ 50K → 500K hồ sơ/tháng
  • Phân biệt stateless vs stateful agents và tại sao stateless dễ scale hơn
  • Thiết kế agent fleet với shared state store (Redis) và message broker
  • Tránh 3 anti-patterns phổ biến khi distribute agents: shared mutable state, chatty agents, God coordinator
🏢 Tình huống: FinTech Corp Scale Problem

Tháng 1/2025, FinTech Corp ký hợp đồng với 3 ngân hàng đối tác mới. Volume tăng từ 50K → 500K hồ sơ/tháng. Hệ thống hiện tại:

  • Single LoanBot instance: 1 container, xử lý tuần tự, max 2,000 hồ sơ/ngày
  • Giờ cao điểm (9–11 giờ sáng): 40% traffic dồn vào 2 giờ → timeout 30 giây
  • Bottleneck: CIC API call mất 2–3 giây/hồ sơ → sequential = 500K × 2.5s = 14.5 ngày xử lý
  • Giải pháp cần: Parallel processing, queue-based load leveling, stateless agent design
📌 Khái niệm: Stateless vs Stateful Agent

Stateful agent: giữ memory của previous requests trong RAM — dễ implement nhưng không thể scale (instance 1 không biết gì về instance 2). Stateless agent: không giữ local state — mọi state đều persist ra external store (Redis, database). Bất kỳ instance nào cũng có thể handle bất kỳ request nào. LoanBot phải stateless để scale: session state → Redis, conversation history → database, model weights → shared read-only mount.

Throughput Scaling Formula
Throughput = N_workers × (1 / avg_latency)
Với avg_latency = 2.5s/hồ sơ: 1 worker = 0.4 hồ sơ/s = 34,560/ngày. Để xử lý 500K/tháng = 16,667/ngày → cần ≥1 worker. Nhưng với peak 40% trong 2 giờ → 3,333/giờ = 0.93/giây → cần ≥3 workers để có headroom. Thực tế: deploy 5 workers với auto-scale lên 10 khi queue depth > 100.
Hình 1: Distributed LoanBot Architecture. API Gateway buffer requests vào Redis Streams. 5 stateless workers consume queue song song, call CIC/Blacklist APIs đồng thời. Session state và cache trong Redis. Auto-scale thêm 5 workers khi queue depth >100.
Python (Stateless LoanBot Worker — Redis State Pattern) 
import redis
import json
from dataclasses import dataclass

@dataclass
class LoanBotWorker:
    """Stateless worker — zero local state, all state in Redis."""
    worker_id: str
    redis_client: redis.Redis
    llm_client: object  # claude client

    def process_task(self, task: dict) -> dict:
        app_id = task["application_id"]

        # Deduplication: only one worker processes each application
        lock_key = f"lock:app:{app_id}"
        acquired = self.redis_client.set(lock_key, self.worker_id, nx=True, ex=60)
        if not acquired:
            return {"status": "duplicate_skipped", "app_id": app_id}

        try:
            # Load state from Redis (no local state)
            cached = self.redis_client.get(f"cic:cache:{app_id}")
            if cached:
                cic_result = json.loads(cached)
            else:
                cic_result = _call_cic_api(task["citizen_id"])
                # Cache for 24h to avoid redundant CIC calls
                self.redis_client.setex(f"cic:cache:{app_id}", 86400, json.dumps(cic_result))

            # Stateless LLM decision — context fully in task payload
            decision = _make_decision(task, cic_result, self.llm_client)

            # Write result (no local storage — straight to DB)
            _save_result(app_id, decision)
            return {"status": "completed", "decision": decision}

        finally:
            self.redis_client.delete(lock_key)  # Always release lock
⚠️ Anti-Pattern: Shared Mutable State

KHÔNG làm: lưu decisions_today_count trong RAM của worker. Khi scale lên 5 workers, mỗi worker count riêng → total count sai. ĐÚng: dùng Redis INCR atomic counter — redis.incr("counter:decisions:2024-11-01"). Redis INCR là atomic, đảm bảo chính xác dù 10 workers cùng increment đồng thời.

Bài 2: Message Queue Patterns — Buffering và Load Leveling

🎯 Sau bài này, bạn sẽ:

  • Hiểu 3 delivery guarantees: at-most-once, at-least-once, exactly-once
  • Implement Redis Streams consumer group cho LoanBot task distribution
  • Thiết kế dead letter queue (DLQ) để handle failed tasks
  • Tối ưu batch size và prefetch count cho peak load handling
📌 Khái niệm: 3 Delivery Guarantees

At-most-once: task được deliver tối đa 1 lần — nếu worker crash giữa chừng, task bị mất. At-least-once: task được deliver ít nhất 1 lần — nếu worker crash sau process nhưng trước ACK, task được deliver lại (duplicate risk). Exactly-once: mỗi task được xử lý đúng 1 lần — cần idempotency key + deduplication. Với loan decisions: at-least-once + idempotency là practical choice. Exactly-once quá phức tạp, at-most-once quá nguy hiểm (mất hồ sơ).

Python (Redis Streams Consumer Group + DLQ) 
import redis
import json
import time
from typing import List

class LoanBotQueue:
    STREAM_KEY  = "loan:tasks"
    GROUP_NAME  = "loanbot-workers"
    DLQ_KEY     = "loan:tasks:dlq"
    MAX_RETRIES = 3
    BLOCK_MS    = 2000   # wait up to 2s for new messages

    def __init__(self, redis_client: redis.Redis):
        self.r = redis_client
        self._ensure_group_exists()

    def _ensure_group_exists(self):
        try:
            self.r.xgroup_create(self.STREAM_KEY, self.GROUP_NAME, id="0", mkstream=True)
        except redis.ResponseError:
            pass  # Group already exists

    def enqueue(self, application: dict) -> str:
        """Add task to stream, return message ID."""
        payload = {
            "data": json.dumps(application),
            "enqueued_at": str(time.time()),
            "retry_count": "0",
        }
        return self.r.xadd(self.STREAM_KEY, payload)

    def consume(self, worker_id: str, batch_size: int = 10) -> List[tuple]:
        """Fetch batch of unacknowledged messages for this worker."""
        messages = self.r.xreadgroup(
            groupname=self.GROUP_NAME,
            consumername=worker_id,
            streams={self.STREAM_KEY: ">"},
            count=batch_size,
            block=self.BLOCK_MS
        )
        if not messages: return []
        return [(msg_id, json.loads(data["data"])) for _, msgs in messages for msg_id, data in msgs]

    def ack(self, message_id: str):
        """Acknowledge successful processing."""
        self.r.xack(self.STREAM_KEY, self.GROUP_NAME, message_id)

    def nack_with_retry(self, message_id: str, error: str):
        """On failure: increment retry count or send to DLQ."""
        pending = self.r.xpending_range(self.STREAM_KEY, self.GROUP_NAME, "-", "+", 1, message_id)
        retry_count = pending[0]["delivery_count"] if pending else 1

        if retry_count >= self.MAX_RETRIES:
            # Move to Dead Letter Queue for manual inspection
            self.r.xadd(self.DLQ_KEY, {"original_id": message_id, "error": error})
            self.r.xack(self.STREAM_KEY, self.GROUP_NAME, message_id)
        # else: leave in PEL (pending entry list) for re-delivery after visibility timeout
🧠 Kiểm tra nhanh

Worker A consume message M1 từ Redis Streams, call CIC API thành công (mất 2.5 giây), rồi crash ngay trước khi xack(). Message M1 sẽ như thế nào sau khi visibility timeout (30 giây) hết?

Gợi ý: M1 vẫn trong PEL (Pending Entry List) của consumer group, chưa được ACK. Sau visibility timeout 30s, Redis sẽ cho phép Worker B (hoặc recovery process) claim lại M1 bằng xclaim(). Worker B sẽ xử lý lại M1 từ đầu — gọi CIC API lần 2. Đây là at-least-once: M1 được xử lý 2 lần. Để safe: cần idempotency check ở đầu process_task() — kiểm tra kết quả đã tồn tại trong DB chưa trước khi gọi CIC. Nếu đã có → skip và return existing result.

Bài 3: Event-Driven Orchestration — Choreography vs Orchestration

🎯 Sau bài này, bạn sẽ:

  • Phân biệt Orchestration (central coordinator) vs Choreography (event-driven, decentralized)
  • Implement Saga Pattern cho loan approval flow — compensating transactions
  • Handle distributed failure: rollback partial success khi 1 trong 5 steps thất bại
  • Biết khi nào dùng Orchestration (cần visibility) vs Choreography (cần loose coupling)
💡 Analogy: Nhạc Trưởng (Orchestration) vs Jazz (Choreography)

Orchestration = có nhạc trưởng (LoanBot Coordinator) chỉ huy: "Bước 1: CIC check → Bước 2: Blacklist → Bước 3: Income..." Nhạc trưởng biết state toàn bộ flow, dễ debug. Nhưng nếu nhạc trưởng ngã → cả buổi dừng lại. Choreography = nhạc jazz: mỗi nhạc cụ nghe âm thanh của người kia và react — không cần nhạc trưởng. Service A emit event → Service B nghe và react → emit event → Service C react... Loose coupling, resilient, nhưng khó trace toàn bộ flow khi có lỗi. FinTech Corp dùng Orchestration cho loan flow (cần audit trail đầy đủ) + Choreography cho notifications (fire-and-forget).

Hình 2: Saga Pattern — loan approval flow có 5 forward steps. Khi S3 (LoanBot Decision) fail: execute compensating transactions ngược lại — C2 release CIC cache lock, C1 release processing slot. S4 và S5 không cần compensate vì chưa execute.

Bài 4: Horizontal Scaling — Auto-Scale và Load Balancing

🎯 Sau bài này, bạn sẽ:

  • Implement queue-depth-based auto-scaling cho LoanBot worker fleet
  • Thiết kế graceful shutdown: worker drain queue trước khi stop
  • Handle hot-spots: distribute load evenly khi 1 worker quá tải
  • Tính toán optimal worker count dựa trên SLA latency targets
📌 Khái niệm: Little's Law — Sizing Worker Fleet

Little's Law: L = λ × W. L = số tasks trong system (queue + processing). λ = arrival rate (tasks/giây). W = processing time (giây/task). Ví dụ FinTech Corp peak: λ = 3.3 tasks/giây (20K hồ sơ trong 2 giờ), W = 2.5 giây/task. L = 3.3 × 2.5 = 8.25 tasks concurrent. Cần ≥9 workers để giữ queue không grow. Với 20% buffer → deploy 11 workers ở peak. Off-peak: λ = 0.4/giây → 1 task concurrent → scale down to 2 workers.

Python (Queue-Depth Auto-Scaler) 
import time
from dataclasses import dataclass
from typing import List

@dataclass
class AutoScaleConfig:
    min_workers: int     = 2
    max_workers: int     = 20
    scale_up_threshold: int   = 50   # queue depth to add workers
    scale_down_threshold: int = 5    # queue depth to remove workers
    cooldown_seconds: int    = 60   # min time between scale events

class LoanBotAutoScaler:
    def __init__(self, queue: LoanBotQueue, config: AutoScaleConfig):
        self.queue = queue
        self.cfg = config
        self.current_workers = config.min_workers
        self.last_scale_time = 0

    def evaluate(self) -> dict:
        queue_depth = self._get_queue_depth()
        now = time.time()
        in_cooldown = (now - self.last_scale_time) < self.cfg.cooldown_seconds

        action = "none"
        if not in_cooldown:
            if queue_depth > self.cfg.scale_up_threshold and self.current_workers < self.cfg.max_workers:
                # Scale up by 2 workers at a time
                new_count = min(self.current_workers + 2, self.cfg.max_workers)
                self._spawn_workers(new_count - self.current_workers)
                self.current_workers = new_count
                self.last_scale_time = now
                action = f"scale_up → {new_count} workers"

            elif queue_depth < self.cfg.scale_down_threshold and self.current_workers > self.cfg.min_workers:
                # Scale down: send graceful stop signal, workers drain then exit
                new_count = max(self.current_workers - 1, self.cfg.min_workers)
                self._graceful_stop_one_worker()
                self.current_workers = new_count
                self.last_scale_time = now
                action = f"scale_down → {new_count} workers"

        return {"queue_depth": queue_depth, "workers": self.current_workers, "action": action}

    def _get_queue_depth(self) -> int:
        info = self.queue.r.xinfo_groups(self.queue.STREAM_KEY)
        for g in info:
            if g["name"] == self.queue.GROUP_NAME:
                return g["pending"] + g["lag"]  # pending + undelivered
        return 0

    def _graceful_stop_one_worker(self):
        # Set stop flag in Redis — worker checks this flag after each task
        self.queue.r.set("signal:stop_one_worker", "1", ex=120)

Bài 5: Observability at Scale — Distributed Tracing và Aggregated Metrics

🎯 Sau bài này, bạn sẽ:

  • Implement correlation ID: trace một hồ sơ qua 5+ services
  • Thiết kế distributed tracing với OpenTelemetry cho LoanBot fleet
  • Define SLO/SLI cho distributed system: p99 latency, queue depth, error rate
  • Build operational runbook: khi queue depth >500 thì làm gì
🔑 Key Insight: The Three Pillars of Distributed Observability

Logs: ghi lại events với correlation_id — "app-FC-2024-9999 | worker-3 | CIC timeout after 5s". Metrics: số đo theo thời gian — queue_depth, worker_throughput, error_rate, p99_latency. Traces: end-to-end journey của một request qua nhiều services — từ khi vào Gateway đến khi ra kết quả. Với distributed LoanBot: trace = chain of spans (Gateway → Queue → Worker → CIC → DB), mỗi span có duration và status. Nếu p99 > 10 giây → trace xem span nào slow.

Python (Correlation ID + Distributed Tracing Context) 
import uuid
from contextlib import contextmanager
from dataclasses import dataclass, field
from typing import List, Optional
import time

@dataclass
class TraceSpan:
    name: str
    correlation_id: str
    trace_id: str
    worker_id: str
    start_time: float = field(default_factory=time.time)
    end_time: Optional[float] = None
    status: str = "in_progress"
    metadata: dict = field(default_factory=dict)

    def finish(self, status: str = "ok", **metadata):
        self.end_time = time.time()
        self.status = status
        self.metadata.update(metadata)

    @property
    def duration_ms(self) -> float:
        if self.end_time: return (round((self.end_time - self.start_time) * 1000, 1))
        return -1

class LoanBotTracer:
    """Lightweight tracer — correlates spans across worker fleet."""
    def __init__(self, worker_id: str):
        self.worker_id = worker_id
        self.spans: List[TraceSpan] = []

    @contextmanager
    def span(self, name: str, correlation_id: str, trace_id: str = None):
        s = TraceSpan(name, correlation_id, trace_id or str(uuid.uuid4()), self.worker_id)
        self.spans.append(s)
        try:
            yield s
            s.finish(status="ok")
        except Exception as e:
            s.finish(status="error", error=str(e))
            raise

    def export(self) -> List[dict]:
        # In production: send to Jaeger/Zipkin via OTLP exporter
        return [{
            "span": s.name, "corr_id": s.correlation_id,
            "duration_ms": s.duration_ms, "status": s.status,
            "worker": s.worker_id, **s.metadata
        } for s in self.spans]

# SLO Definitions
SLO_TARGETS = {
    "p99_latency_ms":  8000,    # 99% of decisions in <8 seconds
    "error_rate_pct":  0.5,     # <0.5% error rate
    "queue_depth_max": 200,     # queue never exceeds 200 pending
    "throughput_min":  500,     # >500 decisions/hour minimum
}
  1. Alert: queue_depth > 200 (WARNING)

    Check auto-scaler: đã trigger chưa? Nếu chưa → manual trigger. Check worker health: có worker crash không (xpending count tăng)? Scale up thêm 3 workers.

  2. Alert: queue_depth > 500 (CRITICAL)

    Tạm dừng nhận requests mới (circuit breaker at Gateway). Notify team. Spawn max workers (20). Check nếu CIC API slow → switch to backup CIC endpoint. ETA to drain = queue_depth / current_throughput.

  3. Dead Letter Queue có items

    Inspect DLQ: mỗi item có original payload + error message. Nếu error = CIC timeout → retry sau 10 phút. Nếu error = malformed data → alert data team, do NOT retry automatically.

  4. Post-incident

    Tính Apdex score cho period bị ảnh hưởng. Update runbook với root cause. Nếu incident xảy ra >2 lần/tháng → cần architecture review.

📚 Glossary Module 25

Thuật ngữ Định nghĩa
Stateless AgentAgent không giữ local state — mọi state persist ra external store (Redis/DB). Prerequisite của horizontal scaling.
Consumer GroupRedis Streams: nhóm các consumers cùng consume 1 stream, mỗi message chỉ delivered cho 1 consumer trong group.
At-least-once DeliveryMessage guarantee: task được deliver ≥1 lần. Cần idempotency để tránh double-processing.
Dead Letter Queue (DLQ)Queue chứa messages fail sau MAX_RETRIES — để human inspect, không auto-retry để tránh infinite loop.
Saga PatternPattern xử lý distributed transaction: chuỗi steps forward + compensating transactions ngược khi fail.
Compensating TransactionAction undo effect của step đã success trước đó — VD: release slot nếu downstream step fail.
OrchestrationCentral coordinator chỉ huy flow: biết toàn bộ state, dễ audit. Vulnerable: single point of failure.
ChoreographyDecentralized: services react đến events của nhau, không cần coordinator. Resilient nhưng khó trace.
Little's LawL = λ × W: queue length = arrival rate × processing time. Dùng để size worker fleet.
Graceful ShutdownWorker finish current task trước khi stop — không drop task giữa chừng khi scale down.
Correlation IDUnique identifier gắn vào mọi log/trace của 1 request xuyên suốt tất cả services.
Distributed TracingTrace end-to-end journey của request qua nhiều services — chain of spans với duration và status.
SLO/SLIService Level Objective (target: p99 <8s) vs Service Level Indicator (actual measured value).
PEL (Pending Entry List)Redis Streams: danh sách messages đã delivered nhưng chưa ACK — nếu worker crash, PEL cho biết gì cần re-process.
IdempotencyXử lý cùng task nhiều lần cho ra cùng kết quả — prerequisite để safe retry trong at-least-once systems.

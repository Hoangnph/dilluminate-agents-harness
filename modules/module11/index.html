LoanBot đã xử lý 99.7% hồ sơ tự động trong Modules 1–7. Nhưng 0.3% còn lại — khoảng 150 hồ sơ/tháng — cần loan officer thực thụ ra quyết định. Đây không phải điểm yếu của AI; đây là thiết kế có chủ ý để đảm bảo trách nhiệm giải trình (accountability) theo luật định.

📌 Khái niệm: Human-in-the-Loop (HITL)

HITL là pattern thiết kế trong đó hệ thống AI tạm dừng và chuyển quyền quyết định sang con người cho những trường hợp nhất định. Khác với Human-on-the-Loop (HOTL) — human monitor nhưng không can thiệp trực tiếp vào mỗi quyết định. LoanBot dùng cả hai: HITL cho borderline cases, HOTL cho full monitoring via Module 8 dashboard.

1.1 EU AI Act Art.14 — Bắt buộc Human Oversight

EU AI Act (2024) phân loại LoanBot là High-Risk AI System (Annex III, creditworthiness assessment). Điều 14 yêu cầu:

  1. Human oversight measures

    Phải có ít nhất một người có thẩm quyền có thể hiểu, giám sát và nếu cần, overrule AI output.

  2. Meaningful control

    Loan officer không chỉ rubber-stamp — phải có đủ thông tin để thực sự hiểu tại sao AI đề xuất như vậy (→ Lesson 3: Explainability).

  3. Override capability

    Human phải có kỹ thuật và quy trình để override quyết định AI khi cần — và override phải được log đầy đủ.

⚠️ Rủi ro: "Automation Bias"

Nghiên cứu cho thấy khi AI đưa ra recommendation, con người có xu hướng follow theo dù AI sai — gọi là automation bias. Design HITL UI cần counter-act điều này: hiển thị factors, không chỉ verdict; yêu cầu loan officer gõ reasoning trước khi xem AI recommendation; thêm friction cho override bị disable.

1.2 Bốn Loại HITL Triggers

💰 Trigger 1: Loan Amount

Loan > 500M VNĐ → HITL bắt buộc (EU AI Act Art.14). Loan > 2 tỷ → HITL + Credit Committee approval.

🚨 Trigger 2: Fraud / Risk Signals

FraudAgent: fraud_probability > 0.3 OR blacklist adjacent match → HITL ngay, không chờ pipeline hoàn tất.

📊 Trigger 3: Borderline Score

Credit score 560–680 (grey zone). RiskAgent confidence < 0.75. Agent disagreement: 2+ agents có recommendation khác nhau.

📋 Trigger 4: Regulatory Exception

Khách hàng có lịch sử tranh chấp pháp lý. Loan có mục đích bất thường. Hồ sơ từ jurisdiction đặc biệt.

HITL Rate của LoanBot v3.0
HITL Rate = (T1_count + T2_count + T3_count + T4_count) / total_apps
FinTech Corp: ~150 HITL/tháng / 50.000 apps = 0.3% — mỗi loan officer xử lý ~19 HITL cases/ngày (8 giờ ÷ 25 phút/case)

from enum import Enum
from dataclasses import dataclass, field
from datetime import datetime, timedelta
import uuid

class ApprovalStatus(Enum):
    PENDING     = "PENDING"
    IN_REVIEW   = "IN_REVIEW"
    ESCALATED   = "ESCALATED"
    APPROVED    = "APPROVED"
    REJECTED    = "REJECTED"
    AI_OVERRIDE = "AI_OVERRIDE"

@dataclass
class HITLCase:
    case_id: str = field(default_factory=lambda: str(uuid.uuid4())[:8])
    loan_id: str = ""
    loan_amount_vnd: int = 0
    trigger_type: str = ""       # "AMOUNT", "FRAUD", "BORDERLINE", "REGULATORY"
    ai_recommendation: str = ""   # "APPROVE" hoặc "REJECT"
    ai_confidence: float = 0.0
    status: ApprovalStatus = ApprovalStatus.PENDING
    assigned_to: str = ""
    approval_level: int = 1
    created_at: datetime = field(default_factory=datetime.now)
    sla_deadline: datetime = None
    decision: str = ""
    decision_reason: str = ""
    decided_at: datetime = None

    def set_sla(self):
        # SLA tùy theo approval level
        sla_hours = {1: 8, 2: 4, 3: 24}
        hours = sla_hours.get(self.approval_level, 8)
        self.sla_deadline = self.created_at + timedelta(hours=hours)

    def is_sla_breached(self) -> bool:
        return datetime.now() > self.sla_deadline if self.sla_deadline else False

    def human_decide(self, decision: str, reason: str, reviewer: str):
        # Log quyết định + detect AI override
        self.decision = decision
        self.decision_reason = reason
        self.decided_at = datetime.now()
        if decision != self.ai_recommendation:
            self.status = ApprovalStatus.AI_OVERRIDE
            # Override phải được log và report cho model improvement
        else:
            self.status = (ApprovalStatus.APPROVED
                           if decision == "APPROVE"
                           else ApprovalStatus.REJECTED)

Loan officer xem xét hồ sơ FC-2024-002 (AI recommend: REJECT, confidence 0.72). Loan officer quyết định APPROVE dựa trên soft factors. Workflow ghi nhận trạng thái gì? Tại sao điều này quan trọng?

Gợi ý: Status = AI_OVERRIDE. Quan trọng vì: (1) EU AI Act yêu cầu log mọi override; (2) Accumulated overrides → training signal — nếu loan officer thường override trong tình huống X, model cần học thêm về X; (3) Override rate là KPI: nếu >15% → model quality vấn đề; (4) Override reason tạo labeled dataset cho fine-tuning sau.

💡 Analogy: Bác sĩ và chẩn đoán AI

Hình dung loan officer như bác sĩ nhận kết quả từ hệ thống chẩn đoán AI. Bác sĩ không chỉ nhìn "Positive/Negative" — họ cần: các chỉ số cụ thể, mức độ tin cậy, yếu tố nào đóng góp nhiều nhất, lịch sử bệnh nhân. Loan officer cũng vậy: cần explanation đủ để hiểu và chịu trách nhiệm về quyết định của mình.

class ExplanationGenerator:
    """Tạo human-readable explanation từ agent outputs."""

    def generate(self, loan_data: dict,
                   agent_outputs: dict) -> dict:
        credit = agent_outputs["CreditAgent"]
        income = agent_outputs["IncomeAgent"]
        risk   = agent_outputs["RiskAgent"]
        fraud  = agent_outputs["FraudAgent"]

        # Tính factor contributions (simplified SHAP-style)
        factors = self._compute_factors(credit, income, risk, fraud)
        # Tạo natural language summary
        summary = self._generate_summary(factors, loan_data)
        # Highlight điểm loan officer cần chú ý nhất
        attention = self._key_attention_points(factors)

        return {
            "verdict": risk["recommendation"],
            "confidence": risk["confidence"],
            "factors": factors,
            "summary": summary,
            "attention_points": attention,
            "hitl_reason": self._hitl_reason(credit, risk),
        }

    def _hitl_reason(self, credit: dict, risk: dict) -> str:
        # Giải thích TẠI SAO hồ sơ này cần human review
        reasons = []
        if 560 <= credit["credit_score"] <= 680:
            reasons.append("Credit score trong borderline zone (560–680)")
        if risk.get("confidence", 1.0) < 0.75:
            reasons.append(f"AI confidence thấp ({risk['confidence']:.0%})")
        return "; ".join(reasons) or "Regulatory requirement"

Research (Goddard et al., 2012; Skitka et al., 2000) cho thấy 4 kỹ thuật chống automation bias:

  1. Delay AI recommendation — Hiển thị factors và data trước, AI verdict ở cuối sau khi loan officer đã hình thành ý kiến ban đầu.
  2. Require pre-commitment — Yêu cầu loan officer chọn "Inclined to: Approve / Reject / Unsure" trước khi unlock AI recommendation.
  3. Counterfactual display — "Nếu thu nhập tăng 5M/tháng, AI sẽ APPROVE" — giúp loan officer hiểu decision boundary.
  4. Override friction — Khi agree với AI: 1 click. Khi override: phải nhập lý do ≥ 50 ký tự. Tăng accountability, giảm rubber-stamping cả hai chiều.

import httpx
import hashlib
from dataclasses import dataclass

@dataclass
class NotificationHub:
    slack_webhook_url: str
    email_api_key: str
    sent_ids: set = None

    def __post_init__(self):
        self.sent_ids = set()  # idempotency: không gửi duplicate

    def notify_hitl(self, case: dict) -> bool:
        # Deduplicate bằng case_id hash
        notif_id = hashlib.md5(
            f"{case['case_id']}-hitl".encode()
        ).hexdigest()
        if notif_id in self.sent_ids:
            return False  # đã gửi rồi

        # Format Slack Block Kit message
        blocks = [
            {"type": "header",
             "text": {"type": "plain_text",
                      "text": f"🤝 HITL Required: {case['loan_id']}"}},
            {"type": "section",
             "fields": [
                {"type": "mrkdwn",
                 "text": f"*Loan:* {case['loan_amount_vnd']:,} VNĐ"},
                {"type": "mrkdwn",
                 "text": f"*AI:* {case['ai_recommendation']} ({case['ai_confidence']:.0%})"},
                {"type": "mrkdwn",
                 "text": f"*SLA:* {case['sla_deadline']}"},
                {"type": "mrkdwn",
                 "text": f"*Trigger:* {case['trigger_type']}"},
             ]},
            {"type": "actions",
             "elements": [{
                 "type": "button",
                 "text": {"type": "plain_text", "text": "📋 Review Now"},
                 "url": f"https://hitl.fintech.vn/cases/{case['case_id']}",
                 "style": "primary"
             }]}
        ]
        # Send với retry logic (max 3 attempts)
        for attempt in range(3):
            r = httpx.post(self.slack_webhook_url, json={"blocks": blocks})
            if r.status_code == 200:
                self.sent_ids.add(notif_id)
                return True
        return False

5.1 Human Overrides → Training Data

Mỗi lần loan officer override AI là một labeled data point có giá trị. Nếu loan officer APPROVE khi AI recommend REJECT, đó là evidence rằng AI đang quá conservative với profile đó. Accumulated over time → retraining signal.

🔑 Key Insight: Virtuous Learning Loop

LoanBot v3.0 → human overrides → labeled dataset → fine-tuning → LoanBot v3.1 (ít override hơn) → lại thu thập overrides → ... Mỗi vòng lặp, model ngày càng sát với judgment của loan officers thực tế. Điều kiện: overrides phải được structured capture, không phải chỉ log text.

Hình 4: Virtuous Learning Loop — mỗi HITL override là training signal cho model tốt hơn

5.2 Drift Detection từ Override Patterns

Model drift xảy ra khi distribution của loan applications thay đổi nhưng model không được update. Dấu hiệu: override rate tăng đột ngột, overrides tập trung vào một segment (ví dụ: self-employed post-COVID).

Python 
from collections import defaultdict
from statistics import mean
from dataclasses import dataclass
from typing import List, Dict

class FeedbackCollector:
    """Thu thập và phân tích human feedback từ HITL decisions."""

    def __init__(self):
        self._overrides: List[dict] = []
        self._override_rate_history: List[float] = []  # mỗi tháng

    def record_override(self, case_id: str, loan_id: str,
                          ai_rec: str, human_dec: str,
                          reason: str, segment: str):
        if ai_rec != human_dec:  # chỉ log nếu là override
            self._overrides.append({
                "case_id": case_id, "loan_id": loan_id,
                "ai_was": ai_rec, "human_chose": human_dec,
                "reason": reason,
                "segment": segment  # "self_employed", "high_amount", etc.
            })

    def segment_drift_analysis(self) -> Dict[str, float]:
        # Override rate theo segment → phát hiện model weakness
        segment_counts = defaultdict(lambda: {"total": 0, "overrides": 0})
        for override in self._overrides:
            seg = override["segment"]
            segment_counts[seg]["overrides"] += 1

        # Segment với override rate cao = model cần retrain cho segment đó
        rates = {seg: data["overrides"]
                 for seg, data in segment_counts.items()}
        return dict(sorted(rates.items(), key=lambda x: x[1], reverse=True))

    def should_trigger_retraining(self) -> bool:
        # Trigger nếu override rate tháng này > 1.5× trung bình 3 tháng
        if len(self._override_rate_history) < 3:
            return False
        current = self._override_rate_history[-1]
        baseline = mean(self._override_rate_history[-4:-1])
        return current > baseline * 1.5

Bài 1: Chain-of-Thought — Dạy LoanBot "Suy Nghĩ Từng Bước"

🎯 Sau bài này, bạn sẽ:

  • Hiểu tại sao CoT cải thiện accuracy trên complex cases (từ 73% → 89%)
  • Phân biệt Zero-shot CoT vs Few-shot CoT vs Auto-CoT
  • Thiết kế CoT prompt cho LoanBot borderline cases
  • Biết khi nào CoT hữu ích và khi nào không cần thiết
💡 Analogy: Loan Officer vs Rubber Stamp

Không có CoT = nhân viên "rubber stamp": nhìn credit score → bấm APPROVE/REJECT ngay. Nhanh nhưng sai nhiều trên borderline cases. Với CoT = loan officer thực sự: "Score 635 — borderline. Nhưng DTI 0.38 — OK. Employment 24 tháng — stable. Loan-to-income 3.2x — trong giới hạn. Chưa có negative CIC record 3 năm. Xét tổng thể: APPROVE với điều kiện." Kết quả tốt hơn vì quá trình reasoning bắt buộc model phải xem xét từng yếu tố trước khi kết luận.

Tại Sao CoT Hiệu Quả?

Nghiên cứu của Wei et al. (2022) chứng minh: CoT cải thiện accuracy trên arithmetic, commonsense, và symbolic reasoning tasks. Cơ chế: khi model phải "write out" từng bước, nó tạo ra intermediate states trong activation space — giống như "working memory" giúp tránh shortcut reasoning. Với LoanBot: complex loan decisions đòi hỏi kết hợp 6-8 factors — CoT buộc model xem xét tất cả thay vì pattern-match trực tiếp.

Zero-shot CoT (Wei et al.)
"Hãy suy nghĩ từng bước trước khi đưa ra quyết định."
Chỉ thêm một câu — nhưng accuracy tăng 18-40% trên complex reasoning tasks. Đây là kỹ thuật đơn giản nhất và hiệu quả nhất để bắt đầu.

3 Loại CoT cho LoanBot

1. Zero-shot CoT — "Hãy suy nghĩ từng bước"

Thêm instruction vào cuối prompt. Không cần examples. Tốt cho one-off complex cases. Nhược điểm: model tự quyết định "bước" là gì — đôi khi miss critical factors.

2. Few-shot CoT — Ví dụ mẫu với reasoning chain

Cung cấp 2-3 examples với full reasoning chain. Model học structure của reasoning. Tốt nhất cho LoanBot — đảm bảo model luôn check credit score, DTI, employment, CIC theo đúng thứ tự. Tốn thêm ~500 tokens per request.

3. Auto-CoT — Tự động generate reasoning chains

Dùng model để generate CoT examples từ training data → dùng những examples đó làm few-shot. Tự động hóa quá trình tạo training examples. Tốt khi có nhiều loại cases khác nhau cần coverage.

Bài 2: ReAct Pattern — Reasoning + Acting trong Vòng Lặp

🎯 Sau bài này, bạn sẽ:

  • Hiểu ReAct = interleave Thought → Action → Observation trong loop
  • Implement ReAct loop cho LoanBot: reason → call CIC → reason lại với kết quả
  • Biết khi nào dừng loop: kết quả đủ confidence hoặc max iterations
  • Debug ReAct traces: phát hiện khi agent bị stuck trong loop vô ích
📌 Khái niệm: ReAct (Yao et al., 2022)

ReAct = Reasoning + Acting. Thay vì reasoning một lần rồi act, agent xen kẽ: Thought (lý luận về tình huống) → Action (gọi tool/API) → Observation (đọc kết quả) → Thought (cập nhật hiểu biết) → Action... cho đến khi có đủ thông tin để kết luận. LoanBot dùng ReAct để: (1) Thought: "Score 638, cần xem CIC history"; (2) Action: call CIC API; (3) Observation: "Không có nợ xấu, 2 khoản vay đang trả đúng hạn"; (4) Thought: "CIC tốt, bù đắp DTI borderline → APPROVE"; (5) Final Answer: quyết định với đầy đủ context.

Hình 1: ReAct loop cho LoanBot. Thought 1 nhận ra cần CIC data → Action gọi API → Observation nhận kết quả → Thought 2 cập nhật assessment → Final Answer. Exit khi confidence ≥ 0.85 hoặc sau 4 iterations.
Python (ReAct Loop với max iterations và exit conditions) 
import anthropic, json
from dataclasses import dataclass
from typing import List, Optional

@dataclass
class ReActStep:
    step_type: str   # thought | action | observation | answer
    content: str

class LoanBotReAct:
    def __init__(self, client: anthropic.Anthropic, max_iterations: int = 4):
        self.client = client
        self.max_iter = max_iterations
        self.tools = self._define_tools()

    def _define_tools(self) -> List[dict]:
        return [
            {"name": "query_cic", "description": "Query CIC (Credit Information Center) for customer credit history",
             "input_schema": {"type": "object", "properties": {"customer_id": {"type": "string"}}, "required": ["customer_id"]}},
            {"name": "check_blacklist", "description": "Check if customer is on FinTech Corp blacklist",
             "input_schema": {"type": "object", "properties": {"customer_id": {"type": "string"}}, "required": ["customer_id"]}},
        ]

    def _execute_tool(self, tool_name: str, tool_input: dict) -> str:
        # Mock tool execution — in production connects to real CIC API
        if tool_name == "query_cic":
            return json.dumps({"bad_debt_count": 0, "active_loans": 2, "on_time_payments_pct": 97.5})
        elif tool_name == "check_blacklist":
            return json.dumps({"blacklisted": False})
        return '{"error": "unknown tool"}'

    def run(self, application: dict) -> dict:
        """Execute ReAct loop: Thought→Action→Observation until answer or max_iter."""
        messages = [{"role": "user", "content": self._build_initial_prompt(application)}]
        trace: List[ReActStep] = []

        for iteration in range(self.max_iter):
            response = self.client.messages.create(
                model="claude-haiku-4-5-20251001", max_tokens=1024,
                tools=self.tools, messages=messages
            )

            # Extract text reasoning (Thought)
            text_parts = [b.text for b in response.content if hasattr(b, 'text')]
            if text_parts:
                trace.append(ReActStep("thought", text_parts[0]))

            # No tool use = model has enough info for final answer
            if response.stop_reason == "end_turn":
                break

            # Execute all tool calls (Actions → Observations)
            tool_results = []
            for block in response.content:
                if block.type == "tool_use":
                    trace.append(ReActStep("action", f"{block.name}({block.input})"))
                    obs = self._execute_tool(block.name, block.input)
                    trace.append(ReActStep("observation", obs))
                    tool_results.append({"type": "tool_result", "tool_use_id": block.id, "content": obs})

            messages += [{"role": "assistant", "content": response.content},
                         {"role": "user", "content": tool_results}]

        return {"trace": trace, "iterations": iteration + 1}

    def _build_initial_prompt(self, app: dict) -> str:
        return f"""Thẩm định hồ sơ vay sau, sử dụng tools nếu cần thêm thông tin:
Customer ID: {app['customer_id']} | Credit Score: {app['credit_score']}
DTI: {app['dti']} | Employment: {app['employment_months']} tháng
Khoản vay: {app['loan_amount_vnd']:,} VNĐ

Sau khi thu thập đủ thông tin, đưa ra quyết định JSON:
{{"decision": "APPROVE|REJECT|REVIEW", "confidence": 0.0-1.0, "explanation": "...", "regulation_refs": ["..."]}}"""

Bài 3: Tree-of-Thought — Khám Phá Nhiều Hướng Suy Luận

🎯 Sau bài này, bạn sẽ:

  • Hiểu ToT = explore multiple reasoning paths, evaluate, chọn tốt nhất
  • Biết khi nào ToT cần thiết (vs CoT đơn giản)
  • Implement Breadth-First ToT cho complex loan decisions
  • Design evaluation function để score từng reasoning branch
📌 Khái niệm: Tree-of-Thought (Yao et al., 2023)

CoT: một path duy nhất từ problem → answer. ToT: branching — tại mỗi bước, generate nhiều possible thoughts, evaluate chúng, tiếp tục từ thought tốt nhất. Giống như: loan officer viết 3 draft assessments theo 3 góc nhìn khác nhau (conservative/moderate/aggressive), sau đó chọn assessment cân bằng nhất. Với LoanBot: dùng ToT khi có conflicting signals — VD: credit score tốt (APPROVE signal) nhưng employment mới 3 tháng (REJECT signal) — ToT explore cả hai branches trước khi quyết định.

Hình 2: ToT cho conflicting signals. LoanBot generates 3 reasoning branches (Conservative/Moderate/Progressive), scores từng branch theo policy fit, chọn Branch B (REVIEW + request docs) với score 8.7/10.
⚠️ Khi Nào KHÔNG Dùng ToT

ToT tốn gấp 3-5x tokens và latency so với CoT đơn giản. Chỉ dùng ToT khi: (1) có conflicting signals rõ ràng; (2) decision có impact cao (khoản vay lớn, edge case chưa từng gặp); (3) team muốn audit reasoning process. 90% cases của LoanBot: dùng CoT hoặc direct decision. 8% borderline: dùng CoT. Chỉ 2% complex conflicting cases: dùng ToT.

Thay vì explore tất cả branches (tốn kém), dùng Beam Search: chỉ giữ top-k branches tốt nhất ở mỗi depth level. Với LoanBot ToT, k=2: generate 3 thoughts → evaluate → keep 2 best → expand 2 → evaluate → keep 2 best → final answer từ best leaf. Beam search giảm complexity từ O(b^d) xuống O(k×d) trong đó b=branching factor, d=depth, k=beam width.

Implementation: sau khi generate n thoughts, dùng một "evaluator prompt" riêng: "Trong 3 reasoning branches sau, branch nào phù hợp nhất với TT39/2016 và chính sách FinTech Corp? Score 1-10 và giải thích." → chọn top-k để tiếp tục.

Bài 4: Planning Agents — Lập Kế Hoạch Nhiều Bước

🎯 Sau bài này, bạn sẽ:

  • Phân biệt Reactive agents (phản ứng) vs Deliberative agents (lập kế hoạch)
  • Implement PDDL-style planning cho LoanBot approval workflow
  • Dùng HTN (Hierarchical Task Network) để decompose complex tasks
  • Handle plan failures: replan khi tool call thất bại
🏢 Tình huống: LoanBot Phải Lập Kế Hoạch

Khách hàng FC-2024-007: khoản vay 800M VNĐ (>500M threshold). Workflow phức tạp:

  1. Query CIC → kiểm tra credit history
  2. Verify employment với employer database
  3. Calculate loan-to-income với income verification
  4. Check collateral value (nếu có)
  5. Escalate to Senior Loan Officer nếu bất kỳ step nào borderline
  6. Generate formal approval letter với tất cả required disclosures

Reactive approach: làm từng bước không cần plan — dễ miss steps, không handle failures. Planning approach: tạo plan trước, execute theo plan, replan nếu cần.

Python (Hierarchical Task Planning cho Large Loan Approval) 
from dataclasses import dataclass, field
from typing import List, Optional, Callable
from enum import Enum

class TaskStatus(Enum):
    PENDING = "pending"
    IN_PROGRESS = "in_progress"
    COMPLETED = "completed"
    FAILED = "failed"
    SKIPPED = "skipped"

@dataclass
class PlanStep:
    step_id: str
    name: str
    tool: str
    params: dict
    required: bool = True
    depends_on: List[str] = field(default_factory=list)
    status: TaskStatus = TaskStatus.PENDING
    result: Optional[dict] = None
    failure_strategy: str = "abort"  # abort | skip | replan | escalate

class LoanApprovalPlanner:
    def create_plan(self, application: dict) -> List[PlanStep]:
        """Generate approval plan based on loan amount and profile."""
        loan_amount = application.get("loan_amount_vnd", 0)
        has_collateral = application.get("has_collateral", False)

        plan = [
            PlanStep("S1", "Kiểm tra blacklist", "check_blacklist",
                     {"customer_id": application["customer_id"]},
                     required=True, failure_strategy="abort"),

            PlanStep("S2", "Query CIC history", "query_cic",
                     {"customer_id": application["customer_id"]},
                     depends_on=["S1"], failure_strategy="escalate"),

            PlanStep("S3", "Verify employment", "verify_employment",
                     {"employer_id": application.get("employer_id")},
                     depends_on=["S1"], failure_strategy="skip"),
        ]

        # Large loan (>500M): add extra verification steps
        if loan_amount > 500_000_000:
            plan.append(PlanStep(
                "S4", "Income verification", "verify_income",
                {"customer_id": application["customer_id"]},
                depends_on=["S2", "S3"], required=True, failure_strategy="abort"
            ))
        if has_collateral:
            plan.append(PlanStep(
                "S5", "Collateral appraisal", "appraise_collateral",
                {"collateral_id": application.get("collateral_id")},
                depends_on=["S1"], failure_strategy="escalate"
            ))

        plan.append(PlanStep(
            "S_final", "Generate decision", "make_decision",
            {}, depends_on=[s.step_id for s in plan], failure_strategy="abort"
        ))
        return plan

    def execute_plan(self, plan: List[PlanStep], executor) -> dict:
        """Execute plan respecting dependencies. Replan on failure."""
        results = {}
        for step in plan:
            # Check all dependencies completed
            deps_ok = all(results.get(d, {}).get("status") == "ok" for d in step.depends_on)
            if not deps_ok:
                step.status = TaskStatus.SKIPPED
                continue

            step.status = TaskStatus.IN_PROGRESS
            result = executor(step.tool, step.params)
            if result.get("status") == "ok":
                step.status = TaskStatus.COMPLETED
                results[step.step_id] = result
            else:
                step.status = TaskStatus.FAILED
                if step.failure_strategy == "abort" and step.required:
                    return {"decision": "REJECT", "reason": f"Critical step {step.name} failed"}
                elif step.failure_strategy == "escalate":
                    return {"decision": "REVIEW", "reason": f"Escalate: {step.name} needs manual verification"}
        return results.get("S_final", {"decision": "REVIEW", "reason": "Plan incomplete"})
🧠 Kiểm tra nhanh

Trong plan của LoanBot, Step S3 (Verify Employment) fail với lỗi "employer database timeout". failure_strategy = "skip". Điều gì xảy ra tiếp theo trong plan execution? Step S_final có chạy không?

Gợi ý: Với failure_strategy="skip", S3 được đánh dấu FAILED nhưng execution tiếp tục (không abort). S4 (Income Verification) depends_on=[S2, S3] — vì S3 FAILED (không phải COMPLETED), S4 sẽ bị SKIPPED (deps_ok = False). S_final depends_on tất cả steps — vì S3 và S4 không COMPLETED, S_final cũng bị SKIPPED. Result: plan trả về fallback {"decision": "REVIEW", "reason": "Plan incomplete"} — đúng behavior vì thiếu employment verification cho loan lớn. Alternative tốt hơn: trong real implementation, S3 failure strategy nên là "escalate" (không phải "skip") để trigger manual review thay vì âm thầm bỏ qua.

Bài 5: Self-Reflection & Critique — LoanBot Tự Kiểm Tra

🎯 Sau bài này, bạn sẽ:

  • Implement Self-Reflection: LoanBot review quyết định của chính nó
  • Dùng Constitutional AI critique để check bias và fairness
  • Multi-agent debate: hai LoanBot instances tranh luận để reach consensus
  • Biết khi nào self-reflection improve accuracy và khi nào gây overthinking
📌 Khái niệm: Self-Reflection Loop

Self-Reflection: sau khi generate initial answer, model đọc lại answer của mình và hỏi "Quyết định này có vấn đề gì không?" → generate critique → revise nếu cần. Với LoanBot: (1) Generate initial decision → (2) Critique: "Quyết định này có discrimination không? Có consistent với TT39/2016 không? Có missing information nào không?" → (3) Nếu critique phát hiện issue → revise decision. Yếu điểm: model đôi khi không tự phát hiện được bias của mình. Cần external critique prompt hoặc Constitutional AI principles làm rubric.

Python (Self-Reflection với Constitutional AI Principles) 
CONSTITUTIONAL_PRINCIPLES = [
    "Quyết định không được dựa trên province/vùng miền của khách hàng (geographic discrimination)",
    "Quyết định phải cite ít nhất 1 regulation reference (TT39/2016 hoặc NĐ13/2023)",
    "Confidence phải phù hợp với evidence — không claim 0.99 confidence trên borderline case",
    "Explanation phải mention tất cả negative factors, không chỉ positive",
    "REVIEW decision phải specify chính xác tài liệu cần bổ sung",
]

async def self_reflect_and_critique(
    client: anthropic.AsyncAnthropic,
    application: dict,
    initial_decision: dict
) -> dict:
    """Apply Constitutional AI self-critique to initial loan decision."""
    principles_text = "\n".join(f"{i+1}. {p}" for i, p in enumerate(CONSTITUTIONAL_PRINCIPLES))

    critique_prompt = f"""Bạn là LoanBot Quality Reviewer. Kiểm tra quyết định sau:

Application: {application}
Initial Decision: {initial_decision}

Kiểm tra theo 5 nguyên tắc Constitutional AI:
{principles_text}

Với mỗi nguyên tắc, đánh giá: PASS hoặc FAIL và lý do.
Nếu có bất kỳ FAIL nào → revise quyết định.

Output JSON:
{{"critiques": [{{"principle_id": 1, "status": "PASS|FAIL", "reason": "..."}}],
  "has_issues": true|false,
  "revised_decision": {{...}} (chỉ khi has_issues=true, format giống initial_decision)
}}"""

    response = await client.messages.create(
        model="claude-haiku-4-5-20251001", max_tokens=1024,
        messages=[{"role": "user", "content": critique_prompt}]
    )
    critique_result = json.loads(response.content[0].text)

    if critique_result.get("has_issues"):
        # Return revised decision if critique found problems
        revised = critique_result.get("revised_decision", initial_decision)
        revised["_critique_applied"] = True
        revised["_critiques"] = critique_result["critiques"]
        return revised
    return initial_decision  # no issues: original decision stands

Thuật ngữ Định nghĩa
Chain-of-Thought (CoT)Kỹ thuật prompting buộc model viết ra reasoning steps trước khi đưa ra kết luận. Tăng accuracy 18-40% trên complex tasks.
Zero-shot CoTCoT chỉ bằng instruction "hãy suy nghĩ từng bước" — không cần examples. Đơn giản nhất.
Few-shot CoTCoT với 2-3 worked examples có full reasoning chain. Tốt nhất cho structured domains như loan assessment.
ReActReasoning + Acting: interleave Thought→Action→Observation trong loop cho đến khi có đủ info để kết luận.
Tree-of-Thought (ToT)Generate nhiều reasoning branches, evaluate từng branch, chọn tốt nhất. Dùng cho conflicting signals.
Beam SearchToT variant: chỉ giữ top-k branches tốt nhất ở mỗi depth → giảm computation từ O(b^d) xuống O(k×d).
Reactive AgentAgent phản ứng trực tiếp với stimuli không có plan. Nhanh nhưng dễ miss steps trong complex workflows.
Deliberative AgentAgent tạo plan trước khi act. Tốt hơn cho multi-step workflows với dependencies và failure handling.
HTN PlanningHierarchical Task Network: decompose complex task thành subtasks có thứ tự và dependencies.
Self-ReflectionModel review quyết định của chính nó, generate critique, revise nếu phát hiện vấn đề.
Constitutional AICritique model output dựa trên explicit principles/rules. Anthropic dùng để align Claude; LoanBot dùng để check bias và compliance.
OverthinkingKhi CoT/ToT làm giảm accuracy vì model bắt đầu doubt đáp án đúng. Thường xảy ra với simple tasks — dùng direct prompting tốt hơn.
ReAct TraceLog đầy đủ Thought/Action/Observation của ReAct loop — dùng để debug, audit, và improve agent.
Failure StrategyPolicy khi plan step fail: abort (stop), skip (continue), escalate (human review), replan (adjust plan).
Multi-Agent Debate2+ LLM instances argue opposing positions → reach consensus. Improves factual accuracy on contested questions.

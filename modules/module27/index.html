Bài 1: Vòng Lặp Cải Thiện — Self-Improvement Framework

🎯 Sau bài này, bạn sẽ:

  • Hiểu Self-Improvement Loop: Deploy → Monitor → Analyze → Update → Re-deploy
  • Phân biệt improvement via prompt engineering vs retraining vs threshold tuning
  • Thiết kế Ground Truth Pipeline: thu thập outcome data từ loan repayment records
  • Biết khi nào nên trigger re-improvement cycle (drift detected, accuracy drop, regulation change)
🏢 Tình huống: LoanBot v3.9 → v4.0 Improvement Cycle

Ngày 0: Deploy LoanBot v3.9 với 3-agent debate panel. Ngày 30: Thu thập outcome data — 5.200 hồ sơ từ tháng trước đã có kết quả (default hoặc repaid on time). Ngày 32: Error analysis: 18% borderline approvals đã default trong 30 ngày (Type I errors). Ngày 35: Threshold optimization tìm DTI ceiling tối ưu 0.42 thay vì 0.45. Ngày 40: A/B test: 10% traffic dùng v4.0 thresholds. Ngày 55: Statistical significance đạt → rollout 100%. Ngày 60: LoanBot v4.0 live — Type I errors giảm từ 18% xuống 8.2%.

Self-Improvement Trigger Conditions
Trigger khi: Accuracy Drop >3% OR Type I Error >15% OR PSI >0.25 OR Regulation Change
PSI (Population Stability Index) đo drift của borrower population. PSI <0.1 = stable; 0.1–0.25 = monitor; >0.25 = retrain. Accuracy Drop >3% so với baseline trong rolling 14-day window → auto-trigger improvement cycle.

Python (Ground Truth Pipeline — Thu thập Outcome Data) 
from dataclasses import dataclass
from datetime import datetime, timedelta
from typing import List, Optional
import json

@dataclass
class LoanOutcome:
    application_id: str
    agent_decision: str       # APPROVE / REVIEW / REJECT
    agent_confidence: float
    disbursed: bool            # Was loan actually given?
    outcome_30d: Optional[str]  # REPAID / DEFAULTED / PENDING
    credit_score: int
    dti_ratio: float
    employment_months: int
    decision_date: datetime

class GroundTruthPipeline:
    """Thu thập outcomes sau 30 ngày và match với agent decisions."""

    def __init__(self, repayment_db_connection):
        self.db = repayment_db_connection
        self.outcomes: List[LoanOutcome] = []

    def collect_30day_outcomes(self, cutoff_date: datetime) -> int:
        """Collect outcomes for loans approved 30+ days ago."""
        lookback = cutoff_date - timedelta(days=30)

        # Query: decisions made 30-60 days ago that have outcome
        records = self.db.query("""
            SELECT a.application_id, a.agent_decision, a.agent_confidence,
                   a.credit_score, a.dti_ratio, a.employment_months, a.decision_date,
                   p.repayment_status as outcome_30d,
                   (p.repayment_status IS NOT NULL) as disbursed
            FROM agent_decisions a
            LEFT JOIN payment_records p ON a.application_id = p.application_id
            WHERE a.decision_date BETWEEN %s AND %s
              AND a.agent_decision = 'APPROVE'
        """, (lookback - timedelta(days=30), lookback))

        for r in records:
            self.outcomes.append(LoanOutcome(**r))

        return len(records)

    def check_trigger_conditions(self) -> dict:
        """Kiểm tra xem có cần trigger improvement cycle không."""
        if not self.outcomes:
            return {'trigger': False, 'reason': 'No data'}

        disbursed = [o for o in self.outcomes if o.disbursed and o.outcome_30d]
        if not disbursed:
            return {'trigger': False, 'reason': 'No outcome data yet'}

        # Type I error rate: approved loans that defaulted
        defaults = sum(1 for o in disbursed if o.outcome_30d == 'DEFAULTED')
        type_i_rate = defaults / len(disbursed)

        triggers = []
        if type_i_rate > 0.15: triggers.append(f'Type I error {type_i_rate:.1%} > 15% threshold')
        if len(disbursed) >= 1000: triggers.append(f'Sufficient data: {len(disbursed)} outcomes')

        return {
            'trigger': bool(triggers),
            'type_i_rate': type_i_rate,
            'n_outcomes': len(disbursed),
            'reasons': triggers
        }

Bài 2: Error Taxonomy — Phân Loại và Định Giá Lỗi

🎯 Sau bài này, bạn sẽ:

  • Phân biệt Type I (False Positive) vs Type II (False Negative) và chi phí thực tế trong loan context
  • Build confusion matrix và segment errors theo credit score zone, DTI zone, employment
  • Tính Business Cost Matrix: cost của từng loại error tính bằng VNĐ thực tế
  • Identify root cause: error do threshold sai, persona bias, hay data quality vấn đề?
📌 Khái niệm: Type I vs Type II Error trong Credit

Type I Error (False Positive): Agent quyết định APPROVE nhưng khách hàng thực sự không đủ tiêu chuẩn → loan disbursed → customer defaults → ngân hàng mất vốn. Chi phí ≈ loan amount × (1 − recovery rate). Type II Error (False Negative): Agent quyết định REJECT nhưng khách hàng thực sự đáng vay → loan not given → ngân hàng mất interest income. Chi phí ≈ loan amount × interest_rate × loan_term. Trong FinTech Corp: Type I cost ≈ 180M VNĐ/case (loss); Type II cost ≈ 42M VNĐ/case (opportunity). Implication: Type I 4.3× more expensive → chấp nhận tăng một chút Type II để giảm Type I.

⚠️ Root Cause Analysis — Nơi Errors Tập Trung

Error analysis trên 5.200 outcomes cho thấy 72% Type I errors tập trung trong segment: DTI 0.40–0.45 VÀ employment <12 tháng VÀ score 600–650. Đây là "danger zone" mà LoanBot v3.9 consistently overestimates creditworthiness. Root cause: OptimistAgent được prompt để prioritize income stability over employment tenure — khi employment ngắn + DTI cao cùng lúc, OptimistAgent anchor quá nhiều vào income amount mà ignore tenure risk.

Python (Error Segmentation Analysis) 
from typing import Dict, List, Tuple
import statistics

def segment_errors(outcomes: List[LoanOutcome]) -> Dict:
    """Phân tích Type I errors theo các segment để tìm root cause."""

    disbursed = [o for o in outcomes if o.disbursed and o.outcome_30d]

    segments = {
        'dti_danger_zone': [],     # DTI 0.40-0.45
        'short_employment': [],    # emp < 12mo
        'score_borderline': [],    # score 600-650
        'combined_risk': [],       # all 3 risk factors
        'other': []
    }

    for o in disbursed:
        is_dti  = 0.40 <= o.dti_ratio <= 0.45
        is_emp  = o.employment_months < 12
        is_score = 600 <= o.credit_score <= 650

        if is_dti and is_emp and is_score:
            segments['combined_risk'].append(o)
        elif is_dti:
            segments['dti_danger_zone'].append(o)
        elif is_emp:
            segments['short_employment'].append(o)
        elif is_score:
            segments['score_borderline'].append(o)
        else:
            segments['other'].append(o)

    results = {}
    for seg, cases in segments.items():
        if not cases: continue
        defaults = sum(1 for o in cases if o.outcome_30d == 'DEFAULTED')
        results[seg] = {
            'n_cases':    len(cases),
            'n_defaults': defaults,
            'type_i_rate': round(defaults / len(cases), 3),
            'action': 'IMMEDIATE FIX' if defaults / len(cases) > 0.20 else 'MONITOR'
        }

    # Sort by Type I rate descending
    return dict(sorted(results.items(), key=lambda x: x[1]['type_i_rate'], reverse=True))

# combined_risk: 45% Type I → IMMEDIATE FIX
# dti_danger_zone: 28% Type I → IMMEDIATE FIX
# short_employment: 19% Type I → IMMEDIATE FIX
# score_borderline: 12% Type I → MONITOR
# other: 4.2% Type I → OK
🧠 Kiểm tra nhanh

FinTech Corp Type I cost = 180M VNĐ/case, Type II cost = 42M VNĐ/case. Nếu tăng threshold từ score 600 lên 620: giảm 40 Type I errors/tháng nhưng tăng 85 Type II errors/tháng. Thay đổi này có worth making không?

Gợi ý: Tính net impact: Savings từ giảm Type I = 40 × 180M = 7.2B VNĐ. Additional cost từ tăng Type II = 85 × 42M = 3.57B VNĐ. Net savings = 7.2B − 3.57B = 3.63B VNĐ/tháng. → Worth making. Tuy nhiên cần cẩn thận: 85 Type II errors = 85 khách hàng tốt bị từ chối → ảnh hưởng reputation. Cũng cần check: 85 cases này distribute thế nào trong các segments — nếu tập trung ở một demographic group, có risk discrimination/fairness issue cần xem xét.

Bài 3: Threshold Optimization — Tìm Điểm Tối Ưu

🎯 Sau bài này, bạn sẽ:

  • Implement Grid Search và Binary Search để tìm optimal threshold trên validation set
  • Dùng Business Cost Matrix thay vì F1 score để optimize — business impact, không phải accuracy
  • Handle multi-dimensional threshold space: credit score + DTI + employment cùng lúc
  • Biết cách validate thresholds: tránh overfitting trên historical data
💡 Analogy: Thermostat vs Temperature Control AI

Threshold optimization giống điều chỉnh thermostat — không phải chỉ tăng/giảm 1 độ rồi xem kết quả, mà phải sweep qua range và tìm điểm cho kết quả tốt nhất. Điểm phức tạp hơn thermostat: LoanBot có 3 "nhiệt độ" đồng thời (score, DTI, employment) — thay đổi một cái ảnh hưởng đến tất cả. Phải optimize trên không gian 3 chiều, không phải 1 chiều.

Python (Business-Cost Threshold Optimizer) 
from dataclasses import dataclass
from typing import Tuple
import itertools

@dataclass
class ThresholdConfig:
    min_credit_score: int    # Minimum credit score to consider
    max_dti: float          # DTI ceiling
    min_employment_months: int  # Minimum employment duration

@dataclass
class ThresholdResult:
    config: ThresholdConfig
    type_i_errors: int
    type_ii_errors: int
    net_cost_million_vnd: float
    business_metric: float  # Lower = better (total cost)

class ThresholdOptimizer:
    # Business cost constants (VNĐ million)
    TYPE_I_COST  = 180  # Default loss per case
    TYPE_II_COST = 42   # Opportunity cost per case

    def __init__(self, validation_outcomes: List[LoanOutcome]):
        self.val = validation_outcomes

    def evaluate_config(self, cfg: ThresholdConfig) -> ThresholdResult:
        """Evaluate a threshold config on validation data."""
        type_i = type_ii = 0

        for o in self.val:
            # Simulate agent decision with this config
            would_approve = (
                o.credit_score >= cfg.min_credit_score and
                o.dti_ratio <= cfg.max_dti and
                o.employment_months >= cfg.min_employment_months
            )
            actual_good = (o.outcome_30d == 'REPAID')

            if would_approve and not actual_good:
                type_i += 1   # Approve bad borrower
            elif not would_approve and actual_good:
                type_ii += 1  # Reject good borrower

        net_cost = type_i * self.TYPE_I_COST + type_ii * self.TYPE_II_COST
        return ThresholdResult(cfg, type_i, type_ii, net_cost, net_cost)

    def grid_search(self) -> ThresholdResult:
        """Search over discrete threshold candidates."""
        score_candidates = range(580, 680, 10)   # 580, 590, ..., 670
        dti_candidates   = [0.38, 0.40, 0.42, 0.43, 0.45]
        emp_candidates   = [6, 9, 12, 18]

        best: Optional[ThresholdResult] = None
        for score, dti, emp in itertools.product(score_candidates, dti_candidates, emp_candidates):
            result = self.evaluate_config(ThresholdConfig(score, dti, emp))
            if best is None or result.net_cost_million_vnd < best.net_cost_million_vnd:
                best = result

        return best

# FinTech Corp grid search result (simulated):
# Best config: score=620, dti=0.42, emp=9 months
# Type I: 945 → 437 cases (-53.8%)
# Type II: 404 → 692 cases (+71.3%)
# Net cost: 174,375M → 107,730M → SAVINGS: 66,645M VNĐ/month ✅

Threshold optimization thuần túy theo business cost có thể tạo ra disparate impact: nếu nhóm dân số A có average credit score thấp hơn nhóm B do lịch sử (không phải do creditworthiness thực tế), tăng score threshold sẽ disproportionately reject nhóm A.

Fairness constraint: Trong optimization loop, thêm constraint: approval_rate_group_A / approval_rate_group_B >= 0.80 (80% rule theo Equal Credit Opportunity Act). Nếu optimal threshold vi phạm fairness constraint → chọn next-best threshold satisfy constraint. Chi phí fairness = additional cost do không dùng business-optimal threshold.

FinTech Corp implement fairness check theo 3 dimensions: gender, age group (18-30 vs 30-50 vs 50+), region (urban vs rural).

Bài 4: Persona Rebalancing — Bayesian Weight Updates

🎯 Sau bài này, bạn sẽ:

  • Track per-agent accuracy trên ground truth để detect agent bias
  • Implement Bayesian weight update: accuracy improves → weight tăng
  • Phát hiện và correct agent systematic bias: OptimistAgent consistently over-approve borderlines
  • Biết khi nào thay đổi weight không đủ — cần update persona system prompt
📌 Khái niệm: Bayesian Weight Update

Bắt đầu với prior weights (Optimist=1.0, Pessimist=1.2, Regulator=1.5). Sau mỗi cycle, update dựa trên observed accuracy: new_weight = prior_weight × (accuracy / baseline_accuracy). Nếu Pessimist accuracy = 81% vs baseline 70% → multiplier = 81/70 = 1.157 → new weight = 1.2 × 1.157 = 1.39 ≈ 1.4. Nếu OptimistAgent accuracy chỉ 61% → multiplier = 61/70 = 0.871 → new weight = 1.0 × 0.871 = 0.87. Clip weights trong range [0.5, 2.0] để prevent extreme dominance.

Python (Per-Agent Accuracy Tracking + Weight Update) 
from dataclasses import dataclass, field
from typing import Dict, List

@dataclass
class AgentPerformanceRecord:
    agent_name: str
    total_votes: int = 0
    correct_votes: int = 0
    type_i_votes: int = 0   # Voted approve, outcome defaulted
    type_ii_votes: int = 0  # Voted reject, outcome would-have-repaid
    current_weight: float = 1.0

    @property
    def accuracy(self) -> float:
        return self.correct_votes / self.total_votes if self.total_votes > 0 else 0.0

class PersonaRebalancer:
    BASELINE_ACCURACY = 0.70   # Expected if random guessing among valid choices
    WEIGHT_FLOOR = 0.5
    WEIGHT_CEIL  = 2.0
    MIN_SAMPLES  = 200         # Min samples before updating weight

    def __init__(self):
        self.records: Dict[str, AgentPerformanceRecord] = {}

    def record_vote(self, agent_name: str, vote: str, outcome: str):
        """Record whether agent vote matched actual outcome."""
        if agent_name not in self.records:
            self.records[agent_name] = AgentPerformanceRecord(agent_name)

        rec = self.records[agent_name]
        rec.total_votes += 1

        # Correctness: agent said APPROVE, loan repaid (true positive)
        #              agent said REJECT/REVIEW, loan would have defaulted (true negative)
        correct = (vote == 'APPROVE' and outcome == 'REPAID') or \
                  (vote in ['REJECT', 'REVIEW'] and outcome == 'DEFAULTED')
        if correct:
            rec.correct_votes += 1
        if vote == 'APPROVE' and outcome == 'DEFAULTED':
            rec.type_i_votes += 1

    def compute_new_weights(self) -> Dict[str, float]:
        """Compute updated weights based on observed accuracy."""
        new_weights = {}
        for name, rec in self.records.items():
            if rec.total_votes < self.MIN_SAMPLES:
                new_weights[name] = rec.current_weight  # Not enough data — keep current
                continue
            multiplier = rec.accuracy / self.BASELINE_ACCURACY
            raw = rec.current_weight * multiplier
            # Clip to prevent extreme values
            new_weights[name] = round(max(self.WEIGHT_FLOOR, min(self.WEIGHT_CEIL, raw)), 2)
        return new_weights

Bài 5: A/B Testing in Production — Safe Deployment

🎯 Sau bài này, bạn sẽ:

  • Design A/B test cho agent improvements: control (v3.9) vs treatment (v4.0)
  • Tính sample size cần thiết để đạt statistical significance (power analysis)
  • Implement traffic splitting: shadow mode → canary → gradual rollout
  • Define stopping criteria: khi nào rollout thành công, khi nào rollback
🔑 Key Insight: Shadow Mode trước khi Canary

Trong credit decision context, không thể đơn giản dùng 50/50 A/B split ngay — mỗi wrong decision có real financial consequence. Shadow Mode: v4.0 runs in parallel với v3.9 nhưng không ra quyết định thực tế — chỉ log predictions để so sánh offline. Sau 2 tuần shadow: nếu v4.0 predictions align với ground truth tốt hơn → move to Canary (5% real traffic). Sau 2 tuần canary với statistical significance → Gradual Rollout (25% → 50% → 100%). Total time-to-full-deploy: 6 tuần — an toàn hơn overnight flip nhưng còn nhanh hơn traditional 6-month review cycle.

Hình 2: A/B Test Rollout Timeline cho LoanBot v4.0. Shadow Mode (0% real, 2 tuần) → Canary 5% (2 tuần, statistical sig) → 25%→50%→100% (3 tuần). Rollback criteria: Type I >20% hoặc error spike >2σ. Total: 7 tuần từ test đến full deploy.
🧠 Kiểm tra nhanh

Sau 2 tuần Canary 5%: control (v3.9) có 1.850 cases với default rate 18.2%. Treatment (v4.0) có 97 cases với default rate 11.3%. Bạn có nên rollout lên 25% không?

Gợi ý: 97 cases là KHÔNG đủ cho statistical significance. Min 500 cases/arm theo design. Với 97 cases treatment, interval confidence rộng — 11.3% default rate có thể là 6%–17% thực tế. Không rollout thêm ngay. Action: (1) Tăng canary từ 5% lên 10% để thu data nhanh hơn; (2) Đợi thêm 1–2 tuần để đạt 300–500 cases; (3) Chạy chi-square test khi đủ sample. Kết quả hiện tại "promising" nhưng chưa "significant." Rollout sớm có risk: nếu v4.0 thực sự không tốt hơn, bạn không biết cho đến khi đã damage 25% khách hàng.

🔬 Fine-tuning & Model Specialization

LoanBot với Claude base model đạt 91% accuracy. Sau 6 tháng với 300,000 quyết định thực tế, FinTech Corp có đủ data để fine-tune một model chuyên biệt đạt 97.3% accuracy — đồng thời giảm 60% token cost vì model đã "hiểu" domain.

⏱ 3 tuần 📖 5 bài giảng 🔬 Fine-tuning Pipeline

Bài 1: Fine-tuning vs Prompt Engineering — Khi Nào Dùng Gì?

🎯 Sau bài này, bạn sẽ:

  • Biết 5 điều kiện xác định khi nào fine-tuning vượt trội prompt engineering
  • Hiểu trade-off: cost, time, flexibility, maintenance
  • Tính ROI: khi nào fine-tuning cost-effective
  • Áp dụng decision framework cho LoanBot
💡 Analogy: Nhân viên mới vs Chuyên gia tư vấn

Prompt Engineering = Thuê chuyên gia tư vấn giỏi (Claude) và viết brief chi tiết mỗi lần. Nhanh, linh hoạt, không cần đào tạo — nhưng tốn thời gian viết brief và đắt hơn per-use. Fine-tuning = Tuyển nhân viên mới, đào tạo 3 tháng về nghiệp vụ FinTech Corp. Sau đó nhân viên này hiểu quy trình mà không cần giải thích lại — nhanh hơn, rẻ hơn per-use, nhưng cần đầu tư training ban đầu và khi quy trình thay đổi cần đào tạo lại.

Decision Framework: 5 Điều Kiện Nên Fine-tune

✅ Nên fine-tune khi:

  1. Volume cao: >50,000 requests/tháng → cost savings từ shorter prompts và smaller model justify training cost
  2. Domain rất đặc thù: LoanBot cần biết TT39/2016, NĐ13/2023, FinTech Corp internal thresholds — quá nhiều context để nhét vào system prompt mỗi lần
  3. Output format cứng: luôn muốn JSON với schema cụ thể, không bao giờ thay đổi
  4. Consistency critical: quyết định tín dụng phải nhất quán — không muốn LLM "sáng tạo" cách diễn đạt mỗi lần
  5. Data đủ lớn và chất lượng: ≥1,000 high-quality examples với expert-verified labels

✅ Nên dùng prompt engineering khi:

  1. Prototype/MVP: cần ship nhanh, chưa có enough data
  2. Task thay đổi thường xuyên: regulation mới mỗi quý → fine-tuned model outdated nhanh
  3. Volume thấp: <5,000 requests/tháng → training cost không recover được
  4. Cần latest knowledge: Claude frontier model cập nhật liên tục; fine-tuned model bị frozen
  5. Nhiều diverse tasks: một model phải làm nhiều việc khác nhau — fine-tune một task có thể hurt others
ROI Calculator
Break-even = Training Cost ÷ (Monthly Savings × 12)
LoanBot: Training Cost ≈ $2,000; Savings = $0.003/req × 50,000 req/month = $150/month; Break-even = 2,000 ÷ (150×12) = 1.1 năm. Nếu model dùng >1.1 năm → positive ROI.
Hình 1: Decision Matrix — Fine-tuning phù hợp khi volume cao VÀ task ổn định. LoanBot sau 6 tháng production nằm ở quadrant trên-phải: 50K hồ sơ/tháng, quy trình thẩm định ổn định.

Bài 2: LoRA & QLoRA — Fine-tuning Hiệu Quả với Chi Phí Thấp

🎯 Sau bài này, bạn sẽ:

  • Hiểu tại sao full fine-tuning tốn kém và LoRA giải quyết vấn đề đó như thế nào
  • Biết các hyperparameters quan trọng: rank (r), alpha, target_modules
  • Hiểu QLoRA = LoRA + 4-bit quantization: giảm thêm 75% memory
  • Chọn đúng rank cho LoanBot use case
📌 Khái niệm: Tại Sao LoRA?

Full fine-tuning: update toàn bộ billions of parameters → cần GPU cluster, tốn $10,000+, mất nhiều tuần. LoRA (Low-Rank Adaptation): insight quan trọng — khi adapt model cho domain mới, thay đổi thực sự chỉ nằm trong một low-rank subspace. Thay vì update matrix W (7B params), LoRA thêm 2 small matrices A và B (chỉ ~0.1% của W) và chỉ train A và B. W_new = W + B·A — W frozen, chỉ train B·A. Kết quả: 100x ít parameters cần train → 10x ít VRAM → training cost $10-100 thay vì $10,000+.

Hình 2: LoRA Architecture. Thay vì update toàn bộ W (7B params), LoRA thêm 2 small matrices B·A (rank=16 → chỉ 131K params = 0.78%). W frozen hoàn toàn. Training cost giảm từ $10,000+ xuống ~$50.
📌 Khái niệm: QLoRA — Quantization + LoRA

QLoRA đi thêm một bước: quantize W từ float16 (16-bit) xuống int4 (4-bit) → giảm thêm 4x memory. Model 7B với float16 cần 14GB VRAM; với QLoRA chỉ cần 3.5GB — chạy được trên single consumer GPU (RTX 3090). Nhược điểm: small accuracy drop (~0.5%) do quantization noise. Cho LoanBot: acceptable vì fine-tuning sau đó recover accuracy.

Hyperparameters Quan Trọng

rank (r) LoanBot: r=16

Kích thước low-rank subspace. r=8: nhỏ, ít params, domain không quá đặc thù. r=16: balance tốt cho most cases. r=64: complex domain, nhiều data. Lớn hơn không luôn tốt hơn — overfitting risk tăng.

alpha LoanBot: α=32

Scaling factor. Learning rate effective = (alpha/rank) × lr. Alpha=32 với rank=16 → scaling=2.0. Thường set alpha=2×rank. Quá cao → unstable training; quá thấp → học chậm.

target_modules q_proj, v_proj

Layers nào apply LoRA. Mặc định: q_proj và v_proj trong attention. Thêm k_proj, o_proj, mlp → nhiều params hơn, tốt hơn nhưng tốn VRAM hơn. Bắt đầu với q+v, thêm nếu accuracy không đủ.

dropout LoanBot: 0.05

Regularization để prevent overfitting. 0.0 = no dropout (nếu data đủ lớn). 0.05–0.1 = typical range. Cao hơn nếu training data nhỏ (<1,000 examples). LoanBot với 50K examples → 0.05 đủ.

Python (LoRA config cho LoanBot với PEFT library) 
from peft import LoraConfig, get_peft_model, TaskType
from transformers import AutoModelForCausalLM, BitsAndBytesConfig
import torch

# QLoRA: load model in 4-bit
bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",          # NormalFloat4 — best for LLM weights
    bnb_4bit_compute_dtype=torch.bfloat16,
    bnb_4bit_use_double_quant=True,     # double quantize for extra compression
)

model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Llama-3-8b-instruct",
    quantization_config=bnb_config,
    device_map="auto",
)

# LoRA configuration for LoanBot domain adaptation
lora_config = LoraConfig(
    r=16,                      # rank — balance between params and expressiveness
    lora_alpha=32,             # scaling = alpha/r = 2.0
    target_modules=["q_proj", "v_proj", "k_proj", "o_proj"],
    lora_dropout=0.05,
    bias="none",
    task_type=TaskType.CAUSAL_LM,
)

model = get_peft_model(model, lora_config)
model.print_trainable_parameters()
# Output: trainable params: 6,815,744 || all params: 8,037,224,448 || 0.085%
# Only 0.085% of parameters are trained! Rest are frozen.
🧠 Kiểm tra nhanh

Với LoanBot cần học cả domain knowledge (TT39/2016, FinTech Corp thresholds) lẫn output format (JSON schema cụ thể), bạn nên dùng rank r=8 hay r=32? Giải thích lý do.

Gợi ý: r=16 là compromise tốt hơn. r=8: có thể không đủ expressiveness để học cả domain knowledge phức tạp (Vietnamese banking law) lẫn strict JSON format — risk underfitting. r=32: nhiều params, học được nhiều hơn, nhưng risk overfitting nếu dataset không đủ lớn và tốn VRAM gấp đôi r=16. r=16 là "Goldilocks" — đủ capacity, không overfit với 50K examples, VRAM manageable. Bắt đầu với r=16, evaluate, điều chỉnh nếu cần. Nếu accuracy vẫn thấp → tăng rank và thêm target modules. Nếu overfitting → giảm rank hoặc tăng dropout.

Bài 3: Synthetic Data Generation — Tạo Training Data từ Production

🎯 Sau bài này, bạn sẽ:

  • Hiểu tại sao raw production data không đủ để fine-tune trực tiếp
  • Dùng Claude để generate synthetic training examples từ real cases
  • Apply data quality filters: deduplication, length filter, quality scorer
  • Đảm bảo privacy: anonymize PII trước khi đưa vào training pipeline
⚠️ PII trong Training Data — Risk Nghiêm Trọng

FinTech Corp có 300,000 real loan decisions với tên KH, CCCD, địa chỉ, thu nhập. Tuyệt đối không dùng raw production data để fine-tune — model có thể memorize và leak PII sau này. GDPR Art.17 và NĐ13/2023 yêu cầu xóa PII khi không cần thiết. Pipeline bắt buộc: (1) Anonymize: thay tên → "KH_001", thay CCCD → random 12 digits, thay địa chỉ → tỉnh/thành phố chỉ; (2) Synthetic generation từ anonymized data; (3) Verify: không còn PII nào trong training set.

Python (Synthetic Training Data Generator) 
import anthropic, json, random, hashlib
from dataclasses import dataclass
from typing import List

@dataclass
class TrainingExample:
    instruction: str
    input_data: dict
    output: dict         # expert-verified decision
    quality_score: float # 0.0–1.0
    source: str         # "synthetic" | "production_anonymized"

def anonymize_application(app: dict) -> dict:
    """Remove PII before using in training pipeline."""
    pseudo_id = "KH_" + hashlib.md5(app.get("customer_id", "").encode()).hexdigest()[:6].upper()
    return {
        "customer_id": pseudo_id,       # hash, not original
        "credit_score": app["credit_score"],
        "dti_ratio": app["dti_ratio"],
        "employment_months": app["employment_months"],
        "loan_amount": round(app["loan_amount"], -7),  # round to nearest 10M
        "region": app.get("province", "Unknown"),  # province only, not address
        # REMOVED: name, cccd_number, phone, exact_address, income
    }

async def generate_training_example(
    client: anthropic.AsyncAnthropic,
    app: dict,
    expert_decision: dict
) -> TrainingExample:
    """Use Claude to generate high-quality training example from real case."""
    prompt = f"""Generate a training example for LoanBot fine-tuning.

Application (anonymized):
{json.dumps(app, indent=2, ensure_ascii=False)}

Expert decision:
{json.dumps(expert_decision, indent=2, ensure_ascii=False)}

Create a natural Vietnamese instruction + explanation that a loan officer would write.
The explanation must:
1. Reference specific numbers from the application
2. Cite relevant regulation (TT39/2016 if applicable)
3. Be consistent with the expert decision
4. Be 2-4 sentences

Return JSON: {{"instruction": "...", "explanation": "...", "quality_score": 0.0-1.0}}"""

    response = await client.messages.create(
        model="claude-haiku-4-5-20251001", max_tokens=512,
        messages=[{"role": "user", "content": prompt}]
    )
    data = json.loads(response.content[0].text)
    return TrainingExample(
        instruction=data["instruction"],
        input_data=app,
        output=expert_decision,
        quality_score=data.get("quality_score", 0.8),
        source="synthetic"
    )

def filter_training_data(examples: List[TrainingExample]) -> List[TrainingExample]:
    """Apply quality filters before training."""
    seen_hashes = set()
    filtered = []
    for ex in examples:
        # 1. Quality threshold
        if ex.quality_score < 0.7: continue
        # 2. Length filter
        if len(ex.instruction) < 30 or len(ex.instruction) > 500: continue
        # 3. Deduplication (MD5 of instruction)
        h = hashlib.md5(ex.instruction.encode()).hexdigest()
        if h in seen_hashes: continue
        seen_hashes.add(h)
        filtered.append(ex)
    return filtered

Bài 4: Evaluation & Catastrophic Forgetting

🎯 Sau bài này, bạn sẽ:

  • Thiết kế evaluation suite cho fine-tuned LoanBot model
  • Hiểu catastrophic forgetting và cách phát hiện nó
  • Implement early stopping dựa trên validation accuracy
  • So sánh base model vs fine-tuned trên 4 dimensions
📌 Khái niệm: Catastrophic Forgetting

Catastrophic forgetting: khi fine-tune LoanBot trên loan data, model có thể "quên" general capabilities — ví dụ không còn có thể viết email hay translate tiếng Anh nữa. Với LoanBot: acceptable nếu chỉ dùng để thẩm định tín dụng. Nhưng nguy hiểm nếu forget: khả năng follow instructions, common sense reasoning, JSON formatting (cần cho output). Cách detect: test general capability benchmarks trước và sau fine-tuning. Cách mitigate: (1) dùng lower learning rate; (2) đừng fine-tune quá nhiều epochs; (3) xen lẫn general instructions vào training data (~10-20% của dataset).

Python (4-Dimension Evaluation Suite) 
from dataclasses import dataclass, field
from typing import List

@dataclass
class EvalResult:
    dimension: str
    score: float       # 0.0–1.0
    baseline: float    # base model score
    passed: bool

class FineTunedModelEvaluator:
    """Compare base vs fine-tuned model on 4 critical dimensions."""

    async def evaluate_domain_accuracy(self, model_fn, dataset) -> EvalResult:
        """Dimension 1: Domain task accuracy (primary metric)."""
        correct = sum(
            model_fn(c)["decision"] == c.expert_decision
            for c in dataset
        )
        score = correct / len(dataset)
        return EvalResult("domain_accuracy", score, baseline=0.91, passed=score >= 0.95)

    async def evaluate_json_compliance(self, model_fn, test_cases) -> EvalResult:
        """Dimension 2: Output always valid JSON with required schema."""
        valid = 0
        for case in test_cases:
            try:
                result = model_fn(case)
                assert {"decision", "confidence", "explanation"}.issubset(result)
                assert result["decision"] in {"APPROVE", "REJECT", "REVIEW"}
                valid += 1
            except: pass
        score = valid / len(test_cases)
        return EvalResult("json_compliance", score, baseline=0.98, passed=score >= 0.99)

    async def evaluate_general_capability(self, model_fn) -> EvalResult:
        """Dimension 3: Detect catastrophic forgetting of general skills."""
        general_tasks = [
            ("Translate to English: 'Xin chào'", lambda r: "hello" in r.lower()),
            ("What is 234 × 17?",              lambda r: "3978" in r),
            ("Format as bullet list: a, b, c",  lambda r: "•" in r or "-" in r),
        ]
        passed = sum(1 for task, check in general_tasks if check(model_fn(task)))
        score = passed / len(general_tasks)
        return EvalResult("general_capability", score, baseline=1.0, passed=score >= 0.85)

    async def evaluate_latency(self, model_fn, n_samples: int = 50) -> EvalResult:
        """Dimension 4: Latency improvement (fine-tuned smaller model is faster)."""
        import time
        times = []
        for _ in range(n_samples):
            t = time.time()
            model_fn({"credit_score": 650, "dti": 0.38})
            times.append(time.time() - t)
        p50 = sorted(times)[n_samples // 2]
        # Fine-tuned model: p50 < 2s (vs base Claude: ~5s with long system prompt)
        score = min(1.0, 5.0 / max(p50, 0.1))  # normalize against 5s baseline
        return EvalResult("latency", score, baseline=0.50, passed=p50 < 2.0)
🧠 Kiểm tra nhanh

Fine-tuned LoanBot đạt domain accuracy 97.2% (tốt hơn baseline 91%) nhưng general_capability chỉ còn 65% (baseline: 100%). Có nên deploy không? Tại sao?

Gợi ý: Không nên deploy ngay. 65% general capability = severe catastrophic forgetting — model có thể fail trên general instruction following, JSON formatting, math. Ngay cả trong loan domain, các câu hỏi follow-up ("giải thích thêm", "so sánh với trường hợp tương tự") có thể fail. Fix: (1) Giảm learning rate 10x và train lại; (2) Tăng general instruction data trong training set lên 20-30%; (3) Giảm số epochs; (4) Thêm early stopping criterion: nếu general_capability < 90% → stop. Threshold acceptable: general_capability ≥ 85%. 65% là red flag nghiêm trọng.

Bài 5: Model Registry & Deployment — Quản Lý Model Lifecycle

🎯 Sau bài này, bạn sẽ:

  • Thiết kế model registry: versioning, metadata, rollback
  • Implement A/B testing: routing traffic giữa base và fine-tuned model
  • Shadow mode: chạy fine-tuned model song song nhưng không dùng output để giám sát
  • Canary deployment: rollout 5% → 25% → 100% với automated rollback
🏢 Tình huống: LoanBot Model Registry

FinTech Corp quản lý 3 model versions đồng thời:

  • v3.5-base: Claude claude-sonnet-4-6 + long system prompt. Production 100%. Baseline accuracy 91%.
  • v3.6-ft-shadow: LLaMA-3-8B + LoRA fine-tuned. Shadow mode — chạy 100% requests nhưng output không dùng. Đang monitor accuracy, latency.
  • v3.6-ft-canary: Sau 2 tuần shadow OK → promote to canary 5%. Nếu 7 ngày stable → 25% → 100%.
Python (Model Registry & Canary Router) 
from dataclasses import dataclass, field
from typing import Dict, Optional
import random, time

@dataclass
class ModelVersion:
    version_id:   str
    model_path:   str
    status:       str   # shadow | canary | production | retired
    traffic_pct:  float  # 0.0–1.0 — fraction of requests routed here
    accuracy:     Optional[float] = None
    p50_latency:  Optional[float] = None
    promoted_at:  Optional[float] = None
    metrics: Dict = field(default_factory=dict)

class ModelRegistry:
    def __init__(self):
        self.versions: Dict[str, ModelVersion] = {}
        self.routing_log = []

    def register(self, version: ModelVersion):
        self.versions[version.version_id] = version

    def route_request(self, request_id: str) -> ModelVersion:
        """Route request to correct model based on traffic split.
        Deterministic: same request_id always routes to same model (consistent UX).
        """
        active = [v for v in self.versions.values()
                  if v.status in ("canary", "production")]
        # Deterministic hashing for consistent routing
        bucket = int(request_id.replace("-", "", 1)[:8], 16) % 100 / 100
        cumulative = 0.0
        for v in sorted(active, key=lambda x: x.version_id):
            cumulative += v.traffic_pct
            if bucket <= cumulative:
                return v
        return active[-1]  # fallback to last

    def promote_canary(self, version_id: str, new_pct: float):
        """Increase canary traffic share — reduce production share."""
        canary = self.versions[version_id]
        prod = [v for v in self.versions.values() if v.status == "production"][0]
        canary.traffic_pct = new_pct
        prod.traffic_pct = 1.0 - new_pct
        if new_pct >= 1.0:
            canary.status = "production"
            prod.status = "retired"

    def rollback(self, reason: str):
        """Emergency rollback — set canary to 0%, production to 100%."""
        for v in self.versions.values():
            if v.status == "canary":
                v.traffic_pct = 0.0
                v.status = "shadow"   # demote back to shadow
            elif v.status == "production":
                v.traffic_pct = 1.0
        print(f"🚨 ROLLBACK: {reason}")

  • Training data provenance: nguồn gốc data, anonymization procedure, date range, n=X examples
  • Evaluation results: domain accuracy, by-difficulty breakdown, catastrophic forgetting test results
  • Known limitations: edge cases model vẫn struggle (out-of-distribution applications, new regulation không có trong training data)
  • Intended deployment context: chỉ dùng trong FinTech Corp loan processing, không transfer sang domains khác

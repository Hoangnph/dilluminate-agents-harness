1.1 Hardcoded Team vs Agent Department

💡 Analogy: Phòng nhân sự vs giao nhiệm vụ trực tiếp

Hãy tưởng tượng một startup 5 người: CEO biết tên và số điện thoại của từng người, giao việc trực tiếp. Khi startup lớn lên thành 50 người, CEO không thể nhớ từng người — cần phòng nhân sự với hệ thống lưu trữ năng lực, ca làm việc, và phân công tự động.

Multi-agent system cũng vậy: Agent Registry là "phòng nhân sự" của hệ thống — biết mọi agent đang available, năng lực (capabilities), và trạng thái (health). Supervisor không cần biết cụ thể agent nào — chỉ cần yêu cầu "tôi cần agent có khả năng kiểm tra tín dụng".

📌 Ba Pillars của Enterprise Multi-Agent Department
  1. Discovery — "Tìm đúng người"

    Agent Registry & Service Discovery: Supervisor không hardcode tên agent, mà query registry theo capability. Tương tự DNS resolver — biến tên thành địa chỉ.

  2. Persistence — "Không mất việc đang làm"

    State Checkpointing: Mỗi bước xử lý được lưu vào persistent storage. Nếu agent crash giữa chừng, có thể resume từ checkpoint thay vì làm lại từ đầu.

  3. Coordination — "Giao tiếp đúng cách"

    A2A Protocol: Chuẩn giao tiếp agent-to-agent — không custom, không mất thông tin khi handoff. Google ra mắt tháng 4/2025, đã trở thành tiêu chuẩn industry.

🧠 Kiểm tra nhanh

FinTech Corp muốn thêm MarketingAgent vào hệ thống LoanBot để gửi offer sau khi phê duyệt khoản vay. Với v2.0 hardcoded, cần sửa mấy file? Với v3.0 Registry, cần làm gì?

v2.0 Hardcoded: Phải sửa ít nhất 3 file: (1) Supervisor code để import và gọi MarketingAgent, (2) Config để thêm API keys, (3) Test files để test luồng mới. Nguy cơ breaking change cao.

v3.0 Registry: (1) Tạo MarketingAgent với capability = ["send_offer", "marketing"]. (2) Register vào AgentRegistry. (3) Supervisor tự phát hiện khi query "tôi cần agent send_offer sau approval". Không cần sửa Supervisor code.

  • 23% doanh nghiệp scale multi-agent thành công — vì hardcoded architecture không flex
  • 3 pillars: Discovery (Registry), Persistence (Checkpointing), Coordination (A2A Protocol)
  • Chuyển từ "biết tên" sang "biết capability" — fundamental shift trong multi-agent design
  • LoanBot v3.0 = 8 agents (5 cũ + ComplianceAgent + FraudAgent + AuditAgent)

Agent Registry & ANS Pattern — Thư Mục Agents

🎯 Sau bài này, bạn sẽ:

  • Hiểu Agent Name Service (ANS) hoạt động như thế nào — tương tự DNS
  • Thiết kế được Capability Card cho một agent mới
  • Biết vòng đời Registration → Health Check → Deregistration
💡 Analogy: DNS của internet vs ANS của agent

Khi bạn gõ google.com, DNS resolver không tìm kiếm thủ công — nó query hệ thống phân tán lưu mapping: google.com → 142.250.x.x. Bạn không cần biết IP thực sự.

Agent Name Service (ANS) hoạt động tương tự: Supervisor query "capability:credit_check" → ANS trả về CreditAgent@host:port. Supervisor không cần biết tên cụ thể của agent.

2.1 Capability Card — Hồ Sơ Năng Lực Agent

Mỗi agent khi đăng ký vào Registry phải cung cấp Capability Card — tương tự CV khi ứng tuyển vào phòng ban:

Python / JSON Schema 
# Capability Card của CreditAgent khi đăng ký vào LoanBot v3.0 Registry
class CapabilityCard:
    agent_id:      str   # "credit-agent-prod-01"
    agent_version: str   # "2.1.0"
    capabilities:  List[str]  # ["credit_check", "cic_query", "score_interpret"]
    input_schema:  dict  # {"application_id": str, "citizen_id": str}
    output_schema: dict  # {"credit_score": int, "grade": str, "blacklisted": bool}
    sla_ms:        int   # 500 — cam kết trả kết quả trong 500ms
    cost_per_call: float # 0.02 — chi phí ước tính mỗi lần gọi (USD)
    health_endpoint: str  # "http://credit-agent:8080/health"
    tags: List[str]      # ["production", "cic-integration", "pci-compliant"]
    ttl_seconds: int    # 300 — lease expires sau 5 phút nếu không renew

Registry lưu Capability Cards và cho phép query theo nhiều chiều:

Python 
# Supervisor v3.0 query Registry — không hardcode agent name

        """Tìm agents theo capability — trả về list sorted theo SLA."""
        ...

# Supervisor dùng như sau:

    capability="credit_check",

)

2.2 Vòng Đời Agent trong Registry

  1. Registration

    Agent khởi động → gửi CapabilityCard tới Registry với TTL (Time-To-Live). Registry cấp lease và trả về agent_id.

  2. Heartbeat / Renewal

    Mỗi 60 giây, agent gửi heartbeat để renew lease. Registry track last_seen và health metrics.

  3. Health Check

    Registry định kỳ gọi /health endpoint của mỗi agent. Agent trả về trạng thái: HEALTHY / DEGRADED / UNHEALTHY.

  4. Deregistration

    Agent shutdown bình thường → gửi deregister signal. Registry xóa khỏi pool ngay lập tức. Nếu crash: TTL hết hạn → tự động bị xóa.

Agent Availability Rule
Available = HEALTHY ∧ TTL_active ∧ SLA_within_budget
Supervisor chỉ route đến agents thỏa mãn cả 3 điều kiện — không cần manual monitoring

💡 Điểm cốt lõi Bài 2

  • Capability Card = hồ sơ năng lực agent: capabilities, SLA, cost, schema, health endpoint
  • Registry query theo capability (không theo tên) — Supervisor không cần biết agent cụ thể
  • TTL-based lease: agent tự động bị xóa khỏi pool nếu crash (không cần manual cleanup)
  • Health check tự động: Supervisor chỉ route đến agents HEALTHY + within SLA budget

A2A Protocol & State Management — Giao Tiếp Chuẩn & Không Mất Dữ Liệu

🎯 Sau bài này, bạn sẽ:

  • Hiểu A2A Protocol (Google, 4/2025) khác gì với cách giao tiếp agent cũ
  • Biết tại sao State Checkpointing quan trọng và implement như thế nào
  • Thiết kế được Task với checkpoint/resume cho một workflow LoanBot
💡 Analogy: Bưu phẩm có tracking vs giao tay

Giao tay trực tiếp: nhanh nhưng không có bằng chứng, không recover được nếu người nhận vắng mặt. Bưu phẩm có tracking: mỗi bước được ghi nhận — xuất phát, trung chuyển, giao thành công — và có thể "tiếp tục" từ trạm cuối nếu gặp sự cố.

A2A Protocol là "hệ thống tracking" cho agent handoffs: mỗi task có ID, trạng thái được ghi nhận, và có thể resume từ bước cuối mà không cần làm lại từ đầu.

Python — A2A Task Object 
# A2A Task — unit of work giữa hai agents
from dataclasses import dataclass
from enum import Enum
from typing import Dict, Any, Optional, List

class TaskState(Enum):
    SUBMITTED  = "submitted"   # Task mới được tạo
    WORKING    = "working"     # Agent đang xử lý
    INPUT_NEEDED = "input-needed" # Agent cần thêm input
    COMPLETED  = "completed"   # Xử lý thành công
    FAILED     = "failed"      # Lỗi — có thể retry
    CANCELLED  = "cancelled"   # Bị hủy bởi Supervisor

@dataclass
class A2ATask:
    task_id:    str           # Unique ID xuyên suốt workflow
    from_agent: str           # "supervisor-01"
    to_agent:   str           # "credit-agent-prod-01"
    capability: str           # "credit_check" — không phải tên agent
    input:      Dict[str, Any] # Dữ liệu đầu vào
    state:      TaskState = TaskState.SUBMITTED
    artifacts:  List[dict] = field(default_factory=list)  # Kết quả
    checkpoint: Optional[dict] = None  # State để resume
    trace_id:   str = field(default_factory=lambda: str(uuid4()))

3.2 State Checkpointing — "Lưu Game" Cho Agents

💡 Analogy: Điểm lưu trong video game

Khi chơi game boss dài 30 phút, bạn luôn muốn game tự save sau mỗi checkpoint. Nếu mất điện lúc 25 phút, bạn tiếp tục từ checkpoint 20 phút — không phải bắt đầu lại từ đầu.

Agent checkpointing hoạt động y hệt: trước mỗi tool call quan trọng (gọi CIC, kiểm tra BHXH...), Harness lưu toàn bộ state của task. Nếu agent crash sau đó, workflow resume từ checkpoint thay vì xử lý lại hồ sơ từ đầu — tiết kiệm chi phí API và thời gian của khách hàng.

Local State

Biến nội bộ của agent: kết quả tính toán trung gian, context window hiện tại

Shared State

Dữ liệu chia sẻ giữa agents: thông tin hồ sơ, credit score đã fetch

Workspace State

Artifacts tạm: draft báo cáo, intermediate analysis, tool outputs

Execution State

Vị trí trong workflow: đang ở step nào, đã gọi tool nào, còn bao nhiêu bước

Hình 3.1 — Checkpoint/Resume: crash giữa chừng → resume từ điểm cuối, không gọi lại API
🧠 Kiểm tra nhanh

LoanBot đang xử lý hồ sơ FC-2024-100. Đã checkpoint sau bước 1 (credit score = 720). Bước 2 (income check) đang chạy thì server restart. Điều gì xảy ra tiếp theo?

Khi server restart: Supervisor đọc TaskState từ database. Task TK-100 có state=WORKING, checkpoint={step: 2, credit_score: 720, tools_called: ["check_credit"]}.

Resume: Supervisor KHÔNG gọi CreditAgent lại (credit_score đã có). Thay vào đó, tạo A2ATask mới gửi cho IncomeAgent với checkpoint data. IncomeAgent tiếp tục từ step 2.

Kết quả: Khách hàng không biết có incident. Hồ sơ được xử lý đúng hạn. CIC không bị gọi lần 2 (tiết kiệm chi phí và tránh duplicate audit log).

💡 Điểm cốt lõi Bài 3

  • A2A Protocol (Google 4/2025): chuẩn REST cho agent-to-agent handoffs — Task object + Agent Card + Artifacts
  • TaskState machine: SUBMITTED → WORKING → INPUT_NEEDED / COMPLETED / FAILED
  • 4 loại state cần persist: Local, Shared, Workspace, Execution state
  • Checkpoint trước mỗi external API call — đây là điểm thất bại chính (CIC timeout, BHXH chậm)

Framework Deep-Dive SOTA — LangGraph vs AutoGen vs CrewAI

🎯 Sau bài này, bạn sẽ:

  • Biết điểm mạnh của từng framework: LangGraph (state persistence), AutoGen (async actor), CrewAI (time-to-value)
  • Áp dụng ma trận quyết định để chọn framework phù hợp với use case
  • Hiểu tại sao LoanBot v3.0 chọn Anthropic SDK + LangGraph-inspired patterns

Năm 2025 xuất hiện 3 SOTA frameworks được enterprise adopt rộng rãi nhất cho production multi-agent:

🗺️ LangGraph
by LangChain · v1.0 (Oct 2025)
Mô hình: Directed Graph — nodes = agents, edges = transitions
Killer feature: State persistence với PostgreSQL/Redis checkpointing
Production users: Uber, JPMorgan, BlackRock, Klarna, Cisco
✅ Stateful ✅ HITL Gates ✅ Checkpoint/Resume ⚠️ Verbose setup
Chọn khi: Mission-critical workflows cần state persistence, complex HITL, và audit trail
⚡ AutoGen v0.4
by Microsoft · Redesign hoàn toàn 1/2025
Mô hình: Event-Driven Actor Model — agents nhận/gửi messages async
Killer feature: Non-blocking async — agents không đợi nhau, throughput cao
Ecosystem: Magentic-One (generalist team), AutoGen Studio (low-code)
✅ High throughput ✅ Async native ✅ MS ecosystem ⚠️ Debugging complex
Chọn khi: High-concurrency, event-driven workflows, tích hợp Microsoft Azure ecosystem
🚢 CrewAI
Open-source · 47.8K stars · 63% Fortune 500
Mô hình: Crew = Agents + Tasks + Process (sequential/hierarchical)
Killer feature: Fastest time-to-value — RBAC, audit, MCP servers built-in
Scale: 2 tỷ agent tasks trong 12 tháng
✅ Quick start ✅ Built-in governance ✅ MCP native ⚠️ Less flexible
Chọn khi: Prototype → production nhanh, team nhỏ, workflows có cấu trúc rõ ràng

🔑 Key Insight: Đừng chọn framework vì "phổ biến"

23% doanh nghiệp scale thành công multi-agent — và nhiều trong số đó không dùng framework nào cả, hoặc tự build orchestration layer. Framework phù hợp là framework giải quyết được bottleneck cụ thể của bạn:

  • State persistence là vấn đề số 1? → LangGraph
  • Throughput cao, nhiều concurrent workflows? → AutoGen v0.4
  • Team nhỏ, cần ship nhanh? → CrewAI
  • Kiểm soát tối đa, không vendor lock-in? → Anthropic SDK + custom orchestration

💡 Điểm cốt lõi Bài 4

  • LangGraph: graph-based, state persistence native, best for complex stateful workflows
  • AutoGen v0.4: actor model, async native, best for high-concurrency event-driven systems
  • CrewAI: fastest time-to-value, built-in governance, best for structured workflows
  • Không có "best framework" — chọn theo bottleneck cụ thể của use case

🔑 Key Insight: "Department" không phải "team"

Sự khác biệt giữa LoanBot v2.0 (team) và v3.0 (department) không chỉ là thêm agents. Đó là sự thay đổi về quản trị hệ thống:

  • Registry: Agents tự quản lý lifecycle, tự report health — không cần DevOps can thiệp thủ công
  • A2A: Handoffs không bao giờ mất dữ liệu — mỗi task có tracking end-to-end
  • Checkpointing: Failures không restart toàn bộ workflow — tiết kiệm cost và thời gian
  • Compliance built-in: ComplianceAgent + AuditAgent → auditor có thể kiểm tra mọi quyết định

Thuật Ngữ Quan Trọng Module 7

Agent Registry / ANS
Hệ thống lưu trữ và tra cứu agents theo capability — tương tự DNS. Supervisor query capability, không query tên agent.
Capability Card
"CV" của agent khi đăng ký: capabilities, SLA, cost, schema, health endpoint, TTL. Registry dùng để routing.
A2A Protocol
Agent-to-Agent Protocol (Google, 4/2025). Chuẩn REST cho task handoffs giữa agents: Task object, Agent Card, Artifacts.
TaskState Machine
SUBMITTED → WORKING → INPUT_NEEDED / COMPLETED / FAILED. Mỗi task A2A đi qua state machine, hỗ trợ retry.
Checkpointing
Lưu execution state trước mỗi external API call. Nếu crash, resume từ checkpoint — không gọi lại APIs đã thành công.
LangGraph
Framework by LangChain. Graph-based orchestration, state persistence native với PostgreSQL. Best for stateful workflows.
AutoGen v0.4 Actor Model
Microsoft framework. Event-driven actors giao tiếp async — không blocking. Throughput cao hơn synchronous patterns.
CrewAI
Open-source framework. Crew = Agents + Tasks. 63% Fortune 500 dùng. Fastest time-to-value, built-in RBAC & audit.
TTL-based Lease
Time-To-Live lease trong Registry. Agent phải renew heartbeat trước khi TTL hết — nếu crash, tự động bị xóa khỏi pool.
Cascading Failure
Một agent chậm/lỗi làm toàn bộ hệ thống sụp đổ. Giải pháp: bulkheads (isolate pools) + circuit breaker + timeout.
Agentic RAG
Dynamic retrieval-augmented generation: agent chủ động quyết định nguồn nào cần query (vector DB, web, SQL) theo reasoning.
MCP (Model Context Protocol)
Chuẩn Anthropic cho agent-to-tool integration (JSON-RPC). 500+ public MCP servers. Tích hợp: GitHub, Slack, Postgres, Notion.

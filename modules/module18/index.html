Bài 1: LLM Gateway Architecture — Tại Sao Cần Một Lớp Trung Gian?

🎯 Sau bài này, bạn sẽ:

LoanBot v1.0 gọi Claude API trực tiếp từ application code — đơn giản nhưng fragile. Mỗi khi Anthropic có maintenance, mỗi khi API key hết quota, mỗi khi cần đổi model — toàn bộ hệ thống phải sửa code. LLM Gateway là lớp trung gian giải quyết tất cả vấn đề này mà không thay đổi application logic.

💡 Analogy: Gateway như Tổng Đài Điện Thoại

Tưởng tượng LoanBot là một văn phòng với 100 nhân viên đều cần gọi điện ra ngoài. Nếu mỗi người tự gọi trực tiếp: không có kiểm soát chi phí, không biết ai đang gọi gì, một người bị block tất cả bị ảnh hưởng. Gateway là tổng đài: mọi cuộc gọi đi qua, tổng đài kiểm soát quota, route đến đường truyền tốt nhất, log mọi cuộc gọi, và switch sang nhà mạng backup khi cần.

Hình 1: LLM Gateway Architecture — Direct API (fragile) vs Gateway pattern (resilient). 6 features của production gateway cho LoanBot v3.4
Công thức cốt lõi
Reliability = Gateway + Redundancy + Observability
Không có Gateway → single point of failure. Gateway + Redundancy → 99.9% uptime. + Observability → biết ngay khi xảy ra vấn đề.
Python (LLM Gateway — core class) 
from dataclasses import dataclass, field
from typing import List, Optional
import time, hashlib, json, anthropic

@dataclass
class GatewayConfig:
    primary_keys:   List[str]        # Pool of API keys — round-robin
    backup_keys:    List[str]        # Failover keys from secondary account
    cache_ttl_s:    int = 3600       # 1h cache for identical requests
    max_retries:    int = 3
    timeout_s:      float = 30.0
    rate_limit_rpm: int = 60          # requests per minute per key

class LLMGateway:
    def __init__(self, config: GatewayConfig):
        self.config      = config
        self._key_idx    = 0           # round-robin pointer
        self._cache      = {}           # request hash → response
        self._metrics    = {"requests": 0, "cache_hits": 0, "failures": 0}

    def _next_key(self, pool: List[str]) -> str:
        # Round-robin across key pool
        key = pool[self._key_idx % len(pool)]
        self._key_idx += 1
        return key

    def _cache_key(self, model: str, messages: list, system: str) -> str:
        payload = json.dumps({"model": model, "messages": messages, "system": system}, sort_keys=True)
        return hashlib.md5(payload.encode()).hexdigest()

    def complete(self, model: str, messages: list, system: str = "", **kwargs) -> dict:
        """Main entry point — handles cache, load balance, retry, failover."""
        self._metrics["requests"] += 1

        # 1. Check cache first
        ck = self._cache_key(model, messages, system)
        if ck in self._cache:
            entry = self._cache[ck]
            if time.time() - entry["ts"] < self.config.cache_ttl_s:
                self._metrics["cache_hits"] += 1
                return entry["response"]

        # 2. Try primary keys first, then backup
        all_pools = [(self.config.primary_keys, "primary"),
                     (self.config.backup_keys,  "backup")]
        for pool, pool_name in all_pools:
            if not pool: continue
            for attempt in range(self.config.max_retries):
                try:
                    api_key  = self._next_key(pool)
                    client   = anthropic.Anthropic(api_key=api_key)
                    response = client.messages.create(
                        model=model, messages=messages, system=system,
                        max_tokens=kwargs.get("max_tokens", 2048)
                    )
                    result = response.content[0].text
                    self._cache[ck] = {"ts": time.time(), "response": result}
                    return result
                except Exception as e:
                    if attempt == self.config.max_retries - 1:
                        continue  # try next pool
                    time.sleep(2 ** attempt)  # exponential backoff

        self._metrics["failures"] += 1
        raise RuntimeError("All API keys and pools exhausted")
🧠 Kiểm tra nhanh

LoanBot gọi API với cùng một system prompt cho mọi hồ sơ. Cache TTL = 1 giờ. Nếu 2 hồ sơ khác nhau gọi với cùng messages + system prompt (test scenario), kết quả thứ 2 có được cache không?

Gợi ý: Có — cache key = MD5(model + messages + system). Nếu 2 requests có cùng 3 parameters này, cache key giống nhau → hit cache. Trong thực tế LoanBot, mỗi hồ sơ có customer data khác nhau trong messages → cache key khác nhau → không hit. Cache chỉ hữu ích cho: (1) retry cùng một request sau lỗi, (2) duplicate submission phát hiện, (3) status checks gọi lại nhiều lần.

Bài 2: Load Balancing & Quota Management

🎯 Sau bài này, bạn sẽ:

  • Hiểu Anthropic rate limits và cách calculate capacity cho 50K hồ sơ/tháng
  • Build priority queue: VIP customers được xử lý trước trong peak hours

FinTech Corp có 50,000 hồ sơ/tháng ≈ 1,667/ngày ≈ 70/giờ bình quân. Nhưng peak load (8–10 sáng, cuối tuần) có thể đạt 400–500/giờ. Anthropic rate limit mặc định: 4,000 RPM (requests per minute) = 240,000/giờ — đủ cho một key. Nhưng nếu một key bị block vì lỗi payment? Cần key pool và quota management.

⚠️ Cảnh báo: Rate Limit ≠ Không Cần Quản Lý

Anthropic rate limit là ceiling của provider, không phải ceiling của business. FinTech Corp có thể cần giới hạn: (1) Department A không được dùng quá 20% total quota; (2) Khách hàng priority 3 không được queue cùng priority 1; (3) Test/dev environment không ăn production quota. Quota management là business logic, cần implement riêng trong gateway.

Hình 3: Round-Robin Load Balancing + Sliding Window Quota — mỗi request qua Quota Manager (sliding window 3600s per department) rồi Round-Robin distributor. Key 3 "degraded" bị skip tự động. Health check chạy mỗi 60 giây.
Python (Quota Manager) 
from collections import defaultdict, deque
import time

class QuotaManager:
    """Sliding window quota per team/customer."""
    def __init__(self, limits: dict):
        # limits = {"loan_dept": 1000, "risk_dept": 500} req/hour
        self.limits   = limits
        self.windows  = defaultdict(lambda: deque())  # key → timestamps
        self.window_s = 3600   # 1-hour sliding window

    def can_request(self, team_id: str) -> bool:
        # Clean expired timestamps outside sliding window
        now    = time.time()
        window = self.windows[team_id]
        while window and now - window[0] > self.window_s:
            window.popleft()
        limit = self.limits.get(team_id, 100)  # default 100/hour
        return len(window) < limit

    def record_request(self, team_id: str):
        self.windows[team_id].append(time.time())

    def usage_pct(self, team_id: str) -> float:
        window = self.windows[team_id]
        limit  = self.limits.get(team_id, 100)
        return len(window) / limit * 100

📌 Khái niệm: Graceful Degradation

Khi tất cả API keys fail, LoanBot không crash — nó chuyển sang degraded mode: (1) Queue requests vào Redis với TTL 2 giờ; (2) Gửi alert ngay cho ops team; (3) Trả về response "đang xử lý, kết quả trong 2 giờ" cho applicant; (4) Khi API phục hồi, drain queue theo priority. SLA vẫn được đảm bảo (kết quả trong 2 giờ thay vì 90 giây) — khách hàng được thông báo proactively.

🏢 Tình huống: FC-2024-001 vs FC-2024-002 trong Peak Hours

9:00 sáng thứ Hai: 450 hồ sơ đang chờ xử lý. Gateway queue có 3 priority levels:

  • P1 (VIP + Urgent): FC-2024-001 — loan 800M VNĐ, khách hàng platinum → queue P1, wait <30s
  • P2 (Standard): FC-2024-002 — loan 300M, standard customer → wait ~5 phút
  • P3 (Batch/Reprocessing): 200 hồ sơ bị lỗi cần reprocess → wait lên đến 30 phút

Kết quả: VIP không bao giờ bị block bởi batch workload. P3 chạy "ngầm" khi bandwidth còn trống.

credit_score ≥ 700 → claude-haiku-4-5 (fast, cheap, accuracy đủ cho clear approval)
risk_score 0.35–0.65 → claude-sonnet-4-6 (borderline cần reasoning sâu hơn)
loan_amount ≥ 2B VNĐ → claude-sonnet-4-6 + extended thinking (high stakes)
blacklisted = True → claude-haiku-4-5 (instant reject — không cần reasoning)

🧠 Kiểm tra nhanh

Tại sao dùng haiku cho blacklisted cases thay vì reject ngay mà không gọi API?

Gợi ý: Trong nhiều jurisdictions (bao gồm EU AI Act), automated decisions phải có explainability — kể cả khi reject là hiển nhiên. Gọi haiku để generate explanation + regulation_refs (TT39, EU AI Act Art.22) giúp đảm bảo compliance và audit trail. Chi phí: $0.0003/call với haiku — negligible. Giải pháp thay thế là template-based rejection explanation, nhưng kém linh hoạt hơn với edge cases.

🔑 Key Insight: Gateway là Billing System

FinTech Corp có 5 departments dùng LoanBot. IT Central trả toàn bộ API bill cho Anthropic. Nhưng ai chịu cost-allocation? Không có gateway logging → IT absorbs toàn bộ cost và không có incentive để optimize.

Python (SLA Metrics + Cost Attribution) 
from dataclasses import dataclass, field
from collections import defaultdict
import time

# Cost per 1M tokens (2025 pricing)
MODEL_COSTS = {
    "claude-haiku-4-5":  {"input": 0.25, "output": 1.25},   # $/1M tokens
    "claude-sonnet-4-6": {"input": 3.00, "output": 15.00},
}

@dataclass
class RequestLog:
    request_id:  str
    team_id:     str
    model:       str
    priority:    int
    latency_ms:  int
    input_tokens:  int
    output_tokens: int
    success:     bool
    timestamp:   float = field(default_factory=time.time)

    @property
    def cost_usd(self) -> float:
        c = MODEL_COSTS.get(self.model, {"input": 0, "output": 0})
        return (self.input_tokens * c["input"] + self.output_tokens * c["output"]) / 1_000_000

class SLAMonitor:
    SLA_P99_MS = 5_000    # P99 latency target: 5 seconds
    SLA_ERROR  = 0.001    # <0.1% error rate

    def __init__(self):
        self.logs: list = []
        self.cost_by_team = defaultdict(float)

    def record(self, log: RequestLog):
        self.logs.append(log)
        self.cost_by_team[log.team_id] += log.cost_usd

    def report(self) -> dict:
        if not self.logs: return {}
        latencies = sorted([l.latency_ms for l in self.logs])
        p99_idx   = int(len(latencies) * 0.99)
        errors    = sum(1 for l in self.logs if not l.success)
        return {
            "total_requests":    len(self.logs),
            "p99_latency_ms":    latencies[p99_idx],
            "p99_sla_ok":        latencies[p99_idx] <= self.SLA_P99_MS,
            "error_rate":        errors / len(self.logs),
            "error_sla_ok":      errors / len(self.logs) <= self.SLA_ERROR,
            "total_cost_usd":    sum(l.cost_usd for l in self.logs),
            "cost_by_team":      dict(self.cost_by_team),
        }

Khi P99 latency vượt 80% SLA threshold (4,000ms) — alert trước khi breach thực sự xảy ra. Gateway check mỗi 30 giây:

Python (Proactive Alert) 
def check_sla_forecast(monitor: SLAMonitor, alert_fn):
    report = monitor.report()
    if report.get("p99_latency_ms", 0) > 4000:   # 80% of 5s SLA
        alert_fn(f"⚠️ SLA WARNING: P99={report['p99_latency_ms']}ms approaching 5s limit")
    if report.get("error_rate", 0) > 0.0005:  # 50% of 0.1% target
        alert_fn(f"🚨 ERROR RATE: {report['error_rate']:.4%} — investigate immediately")

📚 Glossary Module 18

Thuật ngữ Định nghĩa
LLM GatewayLớp trung gian giữa application và LLM providers — xử lý auth, routing, retry, caching, logging.
Round-RobinLoad balancing: phân phối requests đều nhau theo vòng qua các API keys trong pool.
Key PoolTập hợp nhiều API keys của cùng hoặc khác nhau tài khoản — tăng throughput và resilience.
FailoverTự động chuyển sang backend dự phòng khi primary fail. L1→L2→L3 (primary→backup→degraded).
Graceful DegradationHệ thống tiếp tục hoạt động (dù kém hơn) thay vì crash khi có partial failure.
Quota ManagementKiểm soát lượng API calls cho từng team/department — ngăn abuse và đảm bảo fairness.
